# LSK 清洗器「新会话响应」与「ids」解析升级影响分析

## 1. 需求概述

| 需求 | 说明 |
|------|------|
| 1 | **新会话响应**：将字段中的 `content=` 替换为空字符串 |
| 2 | **ids**：从 ids 字段提取 `messageId`、`patientid`、`doctorid` 三个值，存入 items 的 metadata |

### 1.1 数据示例

**新会话响应** 当前可能格式：
```
content=发泡实验阳性3级提示心脏可能存在卵圆孔未闭（PFO）...
```
期望输出：`发泡实验阳性3级提示心脏可能存在卵圆孔未闭（PFO）...`（去掉 `content=` 前缀）

**ids** 当前格式：
```
messageId: f98e3c10-2692-4660-8040-1e040a70d87d
patientid: 1993948901082382359
doctorid: 1922824398239559724
```
期望：三个值均提取并写入 Langfuse DataSet Item 的 metadata。

---

## 2. 影响点分析

### 2.1 数据流链路

```
Excel 行
  → LskCleaner.clean()
    → CanonicalItem (current_msg, response_message, message_id, patient_id, ...)
  → canonical_to_dataset_item()
    → DataSetItemData (input, expected_output, metadata)
  → DatasetItemWriter.write_item()
    → Langfuse create_dataset_item(metadata=...)
```

### 2.2 需求 1：新会话响应 `content=` 替换

| 影响层级 | 文件 | 影响说明 |
|----------|------|----------|
| 清洗层 | `parsers/cleaners/lsk.py` | 在 `clean()` 中，对 `response`（新会话响应）做清洗：若以 `content=` 或 `content：` 开头，则去掉该前缀 |
| 转换层 | `parsers/canonical.py` | 无影响，`response_message` 已是清洗后字符串 |
| 写入层 | `dataset/item_writer.py` | 无影响 |

**实现方式**：在 LskCleaner.clean() 中，对 `response` 调用新增工具函数，如 `strip_content_prefix(text)`，去掉行首或整段的 `content=` / `content：` 前缀。

**边界**：
- 若 `content=` 出现在内容中间（如历史会话响应中的多块），仅处理「新会话响应」单字段，不改变 `parse_lsk_history_response` 逻辑（历史会话响应已按块提取 content 值）
- 支持 `content=`、`content：` 两种形式

---

### 2.3 需求 2：ids 多字段提取并写入 metadata

| 影响层级 | 文件 | 影响说明 |
|----------|------|----------|
| 解析工具 | `parsers/cleaners/lsk.py` | 新增 `extract_lsk_ids(value) -> Dict[str, Optional[str]]`，从 ids 文本提取 message_id、patient_id、doctor_id |
| 规范格式 | `parsers/canonical.py` | CanonicalItem 需新增 `doctor_id: Optional[str] = None`；`canonical_to_dataset_item` 需将 doctor_id 写入 metadata |
| 清洗层 | `parsers/cleaners/lsk.py` | 用 `extract_lsk_ids` 替代 `extract_message_id`，将三个值传入 CanonicalItem |
| 写入层 | `dataset/item_writer.py` | 无改动，metadata 已由 DataSetItemData 传入 |

**extract_lsk_ids 设计**：

```python
def extract_lsk_ids(value: Any) -> Dict[str, Optional[str]]:
    """
    从 ids 列提取 messageId、patientid、doctorid。
    支持 messageId: xxx、patientid: xxx、doctorid: xxx 格式（换行分隔）。
    Returns:
        {"message_id": str|None, "patient_id": str|None, "doctor_id": str|None}
    """
```

**CanonicalItem 变更**：
- 新增字段：`doctor_id: Optional[str] = None`
- 保持 `message_id`、`patient_id` 语义不变

**canonical_to_dataset_item 变更**：
- 在 `item_metadata` 中增加：`if item.doctor_id: item_metadata["doctor_id"] = item.doctor_id`

**LskCleaner.clean() 变更**：
- 原：`msg_id = extract_message_id(row.get(self._MSG_ID_COL))`
- 新：`ids_data = extract_lsk_ids(row.get(self._MSG_ID_COL))`，然后 `message_id=ids_data["message_id"]`, `patient_id=ids_data["patient_id"]`（或沿用列 patient_id 若存在），`doctor_id=ids_data["doctor_id"]`

**注意**：当前 CanonicalItem 已有 `patient_id`，来源为「patient_id」列。ids 中的 patientid 与 patient_id 列可能重复或互补。建议：
- 若存在「patient_id」列，优先用列值；
- 否则用 ids 中的 patientid；
- 或统一以 ids 为准（需与业务确认）。

## 3. 影响范围汇总

---

## 4. 与 Sh1128 清洗器的关系

- **Sh1128 系列**：使用 `extract_message_id`、`patient_id` 列，不依赖 ids 多字段解析，**不受影响**
- **CanonicalItem.doctor_id**：新增可选字段，默认 None，Sh1128 不填即可

## 6. patient_id 来源策略（待确认）

ids 中的 `patientid` 与 Excel 列「patient_id」可能同时存在，建议明确优先级：

- **方案 A**：ids 中的 patientid 优先，列值作为兜底
- **方案 B**：列值优先，ids 作为兜底
- **方案 C**：仅用 ids（LSK 无 patient_id 列时适用）

当前 LSK 的 `_CONTEXT_COL_MAP` 无 patient_id 列，`_extract_context` 不提取 patient_id。若 LSK 的 ids 是 patientid 的唯一来源，则直接使用 ids 中的 patientid 即可。
