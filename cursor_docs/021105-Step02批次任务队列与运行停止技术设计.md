# Step02 批次任务队列与运行/停止技术设计

## 1. 概述

### 1.1 背景与目标

当前执行数据清洗的流程（见 `021104-执行数据清洗接口整体运行逻辑与表设计.md`）中，**rewritten_worker_loop** 通过轮询数据库拉取 `status=init` 记录并自动执行，属于 demo 级实现，存在以下问题：

- 无法由用户主动控制「何时开始跑任务、何时停止」；
- 任务一旦创建即可能被后台消费，不利于排查与可控发布。

本设计将 **「自动轮询 DB → 触发跑任务」** 改为 **「Step02 清洗批次管理界面手动触发运行 / 停止」** 的两阶段模式：

- **第一阶段（已完成）**：点击「进行数据清洗」→ 创建批次 + 在 `pipeline_data_items_rewritten` 中创建若干条 `status=init` 记录，接口立即返回。
- **第二阶段（本文档）**：用户在「Step02 清洗批次管理」列表中，对某批次点击「运行」后，才将待执行任务加入**内存任务队列**，由**多消费者**消费并执行清洗流程；停止/清空能力拆分为 **三种模式**（见 §2.2）。

### 1.2 替换的既有逻辑

以下逻辑将被本方案替代或改造：

- **删除或停用**：`rewritten_worker_loop` 的「轮询 `get_init_records` → 更新 processing → 并行执行」；
- **保留并复用**：`run_one_rewritten(rec)` 及内部 `DataSetsItemsRepository.get_by_dataset_and_item_id` + `_run_one`（即 021104 文档 247–256 行中的「run_one_rewritten → … → update_rewritten_data_func」），作为**单任务执行核心**，由消费者调用。

## 2. 整体架构

### 2.1 两阶段与数据流

```
[用户] 点击「进行数据清洗」 → 创建批次 + 创建 init 记录（已有）
[用户] 在批次列表中点击「运行」 → 后端：查询该批次 init+processing 记录 → 入队（按 id 去重）
                                    ↓
                        ┌─────────────────────────────┐
                        │  内存任务队列（线程/协程安全）  │
                        │  元素：rewritten_record_id   │
                        └─────────────────────────────┘
                                    ↓
                        ┌─────────────────────────────┐
                        │  多消费者（固定数量协程/线程）   │
                        │  每消费者：取 id → 查 DB 得 rec │
                        │  → run_one_rewritten(rec)     │
                        └─────────────────────────────┘
```

- **队列**：进程内内存队列，支持多消费者并发取任务，**不**再依赖「轮询 DB」驱动。
- **入队时机**：仅当用户在批次管理界面点击「运行」时，将该批次下「待处理 + 执行中」的记录 id 入队（id 去重，见下）。

### 2.2 运行与停止的三种模式

| 模式 | 入口 | 语义 |
|------|------|------|
| **运行** | 批次管理 · 每行「运行」按钮 | 将该批次下 `status in (init, processing)` 的记录 id 加入队列（已在 `in_flight_ids` 的 id 跳过，避免重复）。 |
| **模式 1：清空全部队列** | 界面**总按钮**（如「清空队列」） | 删除队列中**所有**任务：清空内存队列，并将这些 id 从 `in_flight_ids` 移除，使未被执行的任务不再被消费；**正在执行中的任务不中断**，执行完即结束。相当于「丢弃所有未完成（未开始执行）的排队任务」。 |
| **模式 2：移除当前批次** | 批次管理 · 每行「移除队列」按钮 | 仅从队列中移除**当前批次**的任务：将该批次在队列中尚未被消费的 id 从队列中剔除（并从 `in_flight_ids` 移除）；该批次正在执行中的任务不中断。 |
| **模式 3：单条再次运行** | **Step02 数据清洗管理** · 每行「再次运行」按钮 | 对任意一条改写记录：先将该任务状态改为「进行中」（`status=processing`），再将该 `record_id` 追加到队列中；若已在队列或执行中则按去重跳过。 |

## 3. 队列与去重设计

### 3.1 队列选型（内存、多消费者安全）

- **推荐（单进程）**：`asyncio.Queue`  
  - 与 FastAPI/uvicorn 单进程多协程模型一致；多个消费者为多个 `asyncio` 协程，`queue.get()` 天然互斥，无需额外锁。
- **可选（后续多进程）**：若未来需要多进程共享队列，可替换为 Redis List + BLPOP，或 `multiprocessing.Manager().Queue()`；本设计先实现进程内 `asyncio.Queue`，接口抽象为「入队 / 出队」，便于后续替换。

**队列元素**：存储 `(rewritten_record_id: str, batch_code: str)` 元组。消费者侧根据 id 再查库得到 `DataItemsRewrittenRecord`。携带 `batch_code` 是为了**模式 2**（按批次移除队列）时能够从队列中识别并剔除该批次的未消费任务。

### 3.2 去重与「已在运行」集合

目标：同一 `rewritten_record_id` 在同一时刻只在队列或执行中出现一次。

- **方案 A（推荐）**：进程内维护一个 **「正在参与任务中的 id 集合」**（`in_flight_ids: set[str]`），与队列配合：
  - **入队前**：若 `record_id in in_flight_ids`，则跳过不入队；
  - **入队时**：`in_flight_ids.add(record_id)`，再 `queue.put_nowait((record_id, batch_code or ""))`；
  - **消费者取到任务后**：得到 `(record_id, batch_code)`，直接执行；
  - **消费者执行完毕（成功或失败）**：`in_flight_ids.discard(record_id)`。
  - 这样「运行」时即使某条已是 `processing`（上次中断），也只会入队一次，且不会重复执行。

- **方案 B**：仅用「已在队列中的 id 集合」做入队去重，不区分「在队列」与「正在执行」。实现更简单，但无法区分「正在执行」与「排队中」的统计；若需在 API 中返回「排队数 / 执行中数」，可用方案 A 并额外维护「队列长度」或「执行中数量」。

- **方案 C（仅队列内去重）**：入队前遍历当前队列内容判断是否已存在。`asyncio.Queue` 不易无损遍历，且无法覆盖「正在被某消费者执行」的那条，故不推荐。

**推荐实现**：采用方案 A，使用 `in_flight_ids: set` + 与 `asyncio.Queue` 配合；入队接口在加 id 前检查 `in_flight_ids`，避免重复。

### 3.3 三种停止/清空模式的实现要点

- **运行**：按 **batch_code** 查询该批次下 `status in (init, processing)` 的记录，将每条 `(rec.id, rec.batch_code or "")` 入队，并将 `rec.id` 加入 `in_flight_ids`。
- **模式 1（清空全部队列）**：循环 `queue.get_nowait()` 直到队列空（或 `get` 超时/空），每取出一个 `(record_id, _)` 即 `in_flight_ids.discard(record_id)`。不操作正在执行的任务。
- **模式 2（移除当前批次）**：因队列元素为 `(record_id, batch_code)`，实现方式：**排空队列并回填**——用临时列表，循环从队列 `get`，若 `batch_code == 当前批次` 则仅做 `in_flight_ids.discard(record_id)`，不放入临时列表；否则将该项放入临时列表；最后将临时列表中的项依次 `put_nowait` 回队列。这样该批次在队列中的任务被移除且不再被执行，正在执行的不中断。
- **模式 3（单条再次运行）**：不涉及「停止」，仅「入队单条」：先将该条 DB 状态更新为 `processing`，再若 `record_id not in in_flight_ids` 则 `in_flight_ids.add(record_id)` 且 `queue.put_nowait((record_id, batch_code))`。需从 DB 或入参拿到该记录的 `batch_code`。

## 4. 后端详细设计

### 4.1 模块与职责

| 模块 | 职责 |
|------|------|
| **任务队列服务**（新建） | 内存队列（元素 `(record_id, batch_code)`）+ `in_flight_ids`；提供 `enqueue_batch(batch_code, session)`、`enqueue_one(record_id, batch_code, session)`、`clear_all_queue()`、`remove_batch_from_queue(batch_code)`、`get_queue_stats()`；启动时启动 N 个消费者协程。 |
| **消费者协程** | 循环：`queue.get()` 得到 `(record_id, batch_code)` → 根据 record_id 查 `DataItemsRewrittenRecord` → `run_one_rewritten(rec)` → 从 `in_flight_ids` 移除 record_id；异常时 `_mark_failed` 并移除 id。 |

| **模式 3 API** | 在 Step02 数据清洗管理侧：接收「再次运行」请求（单条 record_id），先将该条状态改为 processing，再调用 `enqueue_one(record_id, batch_code, session)`。 |

### 4.2 仓储层：按批次获取「待执行」记录

需要能按批次查询 **status in (init, processing)** 的记录，供「运行」时入队使用。

- **新增方法**（建议放在 `DataItemsRewrittenRepository`）：
  - `get_pending_by_batch_code(batch_code: str) -> List[DataItemsRewrittenRecord]`
  - 条件：`batch_code == batch_code`，`status.in_([STATUS_INIT, STATUS_PROCESSING])`，且 `source_dataset_id`、`source_item_id` 非空（与现有 `get_init_records` 类似）；
  - 按 `created_at` 升序，不限制条数（或设较大 limit），返回完整 ORM 列表；调用方只取 `rec.id` 入队。

**关键代码片段（Repository）**：

```python
# backend/infrastructure/database/repository/data_items_rewritten_repository.py

async def get_pending_by_batch_code(
    self,
    batch_code: str,
) -> List[DataItemsRewrittenRecord]:
    """
    按批次获取待执行任务：status in (init, processing)。
    用于「运行」时入队，包含上次中断的 processing 记录。
    """
    if not batch_code or not batch_code.strip():
        return []
    conditions = [
        DataItemsRewrittenRecord.batch_code == batch_code.strip(),
        DataItemsRewrittenRecord.status.in_([STATUS_INIT, STATUS_PROCESSING]),
        DataItemsRewrittenRecord.source_dataset_id.isnot(None),
        DataItemsRewrittenRecord.source_dataset_id != "",
        DataItemsRewrittenRecord.source_item_id.isnot(None),
        DataItemsRewrittenRecord.source_item_id != "",
    ]
    stmt = (
        select(DataItemsRewrittenRecord)
        .where(and_(*conditions))
        .order_by(DataItemsRewrittenRecord.created_at.asc())
    )
    result = await self.session.execute(stmt)
    return list(result.scalars().all())
```

### 4.3 任务队列服务（关键逻辑）

- **单例**：应用内一个队列 + 一个 `in_flight_ids` 集合，供 API 与消费者共享。
- **队列**：`asyncio.Queue`，元素为 `Tuple[str, str]`，即 `(rewritten_record_id, batch_code)`。
- **入队批次**：`enqueue_batch(batch_code: str, session: AsyncSession) -> int`  
  - 调用 `DataItemsRewrittenRepository(session).get_pending_by_batch_code(batch_code)`；  
  - 对每个 `rec`，若 `rec.id not in in_flight_ids`，则 `in_flight_ids.add(rec.id)` 且 `queue.put_nowait((rec.id, rec.batch_code or ""))`；  
  - 返回本次入队数量。
- **入队单条（模式 3）**：`enqueue_one(record_id: str, batch_code: str, session: AsyncSession) -> bool`  
  - 若 `record_id in in_flight_ids` 则返回 False（已排队或执行中）；  
  - 否则 `in_flight_ids.add(record_id)`，`queue.put_nowait((record_id, batch_code or ""))`，返回 True。调用方负责先将该条 DB 状态改为 `processing`。
- **模式 1：清空全部**：`clear_all_queue() -> int`  
  - 计数 n=0，循环：`get_nowait()` 取出 `(record_id, _)`，`in_flight_ids.discard(record_id)`，n+=1，直到 `QueueEmpty`；返回 n。
- **模式 2：移除当前批次**：`remove_batch_from_queue(batch_code: str) -> int`  
  - 临时列表 `to_put_back`；计数 n=0；循环 `get_nowait()` 取出 `(record_id, bc)`，若 `bc == batch_code` 则 `in_flight_ids.discard(record_id)` 且 n+=1，否则 `to_put_back.append((record_id, bc))`，直到 `QueueEmpty`；再将 `to_put_back` 中每项 `put_nowait` 回队列；返回 n。
- **消费者**：固定 N 个（如 4）协程，每个循环：
  - `(record_id, batch_code) = await queue.get()`；
  - 使用 `get_session_factory()()` 新建 session，`DataItemsRewrittenRepository(session).get_by_id(record_id)` 得到 `rec`（若 None 则 `_mark_failed(record_id, "记录不存在")` 并 `in_flight_ids.discard(record_id)`，继续下一轮）；
  - 调用 `await run_one_rewritten(rec)`（现有 `rewritten_service.run_one_rewritten`）；
  - 无论成功失败，最后 `in_flight_ids.discard(record_id)`。
  - 若 `run_one_rewritten` 抛异常，在消费者内 `_mark_failed(record_id, str(e))` 再 discard。

**关键代码片段（队列服务）**：

```python
# backend/pipeline/rewritten_queue_service.py（新建，示意）
# 队列元素为 (record_id: str, batch_code: str)

import asyncio
from typing import Set, Tuple
from sqlalchemy.ext.asyncio import AsyncSession

from backend.infrastructure.database.connection import get_session_factory
from backend.infrastructure.database.repository.data_items_rewritten_repository import (
    DataItemsRewrittenRepository,
)
from backend.pipeline.rewritten_service import run_one_rewritten, _mark_failed

_queue: asyncio.Queue = None  # 元素 (record_id, batch_code)
_in_flight_ids: Set[str] = None
_CONSUMER_COUNT = 4

def _ensure_queue():
    global _queue, _in_flight_ids
    if _queue is None:
        _queue = asyncio.Queue()
        _in_flight_ids = set()

async def enqueue_batch(batch_code: str, session: AsyncSession) -> int:
    _ensure_queue()
    repo = DataItemsRewrittenRepository(session)
    records = await repo.get_pending_by_batch_code(batch_code)
    added = 0
    for rec in records:
        rid = rec.id
        if rid not in _in_flight_ids:
            _in_flight_ids.add(rid)
            _queue.put_nowait((rid, rec.batch_code or ""))
            added += 1
    return added

def enqueue_one(record_id: str, batch_code: str, session: AsyncSession) -> bool:
    _ensure_queue()
    if record_id in _in_flight_ids:
        return False
    _in_flight_ids.add(record_id)
    _queue.put_nowait((record_id, batch_code or ""))
    return True

def clear_all_queue() -> int:
    """模式 1：清空队列中所有任务，返回移除数量。"""
    _ensure_queue()
    n = 0
    while True:
        try:
            record_id, _ = _queue.get_nowait()
            _in_flight_ids.discard(record_id)
            n += 1
        except asyncio.QueueEmpty:
            break
    return n

def remove_batch_from_queue(batch_code: str) -> int:
    """模式 2：从队列中移除当前批次的任务，返回移除数量。"""
    _ensure_queue()
    to_put_back: list[Tuple[str, str]] = []
    n = 0
    while True:
        try:
            record_id, bc = _queue.get_nowait()
            if bc == batch_code:
                _in_flight_ids.discard(record_id)
                n += 1
            else:
                to_put_back.append((record_id, bc))
        except asyncio.QueueEmpty:
            break
    for item in to_put_back:
        _queue.put_nowait(item)
    return n

async def _consumer_loop():
    _ensure_queue()
    session_factory = get_session_factory()
    while True:
        record_id, _batch_code = await _queue.get()
        try:
            async with session_factory() as session:
                repo = DataItemsRewrittenRepository(session)
                rec = await repo.get_by_id(record_id)
            if not rec:
                await _mark_failed(record_id, "记录不存在")
            else:
                await run_one_rewritten(rec)
        except Exception as e:
            await _mark_failed(record_id, str(e))
        finally:
            _in_flight_ids.discard(record_id)

def start_consumers():
    _ensure_queue()
    for _ in range(_CONSUMER_COUNT):
        asyncio.create_task(_consumer_loop())
```

（实际实现中需注意：`start_consumers` 应在应用 lifespan 中只调用一次；队列与 set 的初始化建议在 lifespan 内完成；`clear_all_queue` / `remove_batch_from_queue` 在异步上下文中调用时若需与事件循环兼容，可用 `get_nowait` 的循环即可。）

### 4.4 API 设计

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/data-cleaning/rewritten-batches/run` | 按批次运行。Body: `{ "batch_code": "xxx" }`。返回 `{ "enqueued": 12 }`。 |
| POST | `/data-cleaning/rewritten-batches/clear-queue` | **模式 1**：清空全部队列。无 Body。返回 `{ "removed": 5 }` 表示从队列移除的任务数。 |
| POST | `/data-cleaning/rewritten-batches/remove-batch` | **模式 2**：移除当前批次在队列中的任务。Body: `{ "batch_code": "xxx" }`。返回 `{ "removed": 3 }`。 |
| POST | `/data-cleaning/data-items-rewritten/{item_id}/rerun` | **模式 3**：单条再次运行。无 Body（item_id 在路径）。先将该条状态改为 processing，再入队。返回 `{ "enqueued": true }` 或 `{ "enqueued": false, "message": "已在队列或执行中" }`。 |
| GET | `/data-cleaning/rewritten-batches/queue-stats`（可选） | 返回 `{ "queue_size": 5, "in_flight_count": 4 }`，便于前端展示。 |

**运行接口要点**：

- 校验 `batch_code` 在 `pipeline_rewritten_batches` 中存在（可选）；
- 使用当前请求的 `AsyncSession` 调用 `get_pending_by_batch_code`，再调用队列服务的 `enqueue_batch(batch_code, session)`；
- 返回本次入队数量；若为 0 可提示「无待处理或执行中的任务」或「任务已在队列或执行中」。

**模式 3 接口要点**：

- 根据 `item_id` 查询 `DataItemsRewrittenRecord`，若不存在则 404；
- 调用 `DataItemsRewrittenRepository.update_status(item_id, STATUS_PROCESSING)`（或 `update` 仅更新 status 字段）；
- 调用队列服务 `enqueue_one(item_id, record.batch_code or "", session)`；
- 若 `enqueue_one` 返回 False，返回 200 但 `enqueued: false` 及提示已在队列或执行中。

### 4.5 生命周期：移除旧 Worker，启动消费者

- **main.py lifespan**：  
  - 不再启动 `rewritten_worker_loop`（删除或保持注释）；  
  - 在应用启动时调用「任务队列服务」的 `start_consumers()`，创建 N 个消费者协程；  
  - 关闭时无需对 asyncio.Queue 做特殊清理（进程退出即清空）。

## 5. 前端设计（概要）

### 5.1 Step02 清洗批次管理页

- **位置**：Step02 清洗批次管理列表页，操作列（当前为「后续迭代中提供操作」）。
- **总按钮（模式 1）**：在列表上方或表格外增加一个「清空队列」按钮，点击后调用 `POST /data-cleaning/rewritten-batches/clear-queue`，成功后提示「已清空队列，共移除 N 条」并刷新列表。
- **操作列（每行）**：
  - **运行**：对当前行批次发起 `POST /data-cleaning/rewritten-batches/run`，body `{ batch_code: scope.row.batch_code }`；成功后提示「已加入队列，共 N 条」并刷新列表。
  - **移除队列（模式 2）**：对当前行批次发起 `POST /data-cleaning/rewritten-batches/remove-batch`，body `{ batch_code: scope.row.batch_code }`；成功后提示「已从队列移除该批次 N 条」并刷新列表。

### 5.2 Step02 数据清洗管理页（模式 3）

- **位置**：Step02 数据清洗管理列表（改写后数据项列表），每行操作列增加「再次运行」按钮。

## 6. 与 021104 的流程对照（替换后）

原 021104 §11 依赖关系中的：

```
rewritten_worker_loop
  → DataItemsRewrittenRepository (get_init_records, update_status)
  → run_one_rewritten
       → ...
```

替换为：

```
用户点击「运行」→ API enqueue_batch
  → DataItemsRewrittenRepository (get_pending_by_batch_code)
  → 队列服务 (in_flight_ids 去重 + queue.put_nowait)

消费者协程（N 个）
  → queue.get() → DataItemsRewrittenRepository (get_by_id)
  → run_one_rewritten(rec)
       → DataSetsItemsRepository (get_by_dataset_and_item_id)
       → _run_one
            → build_state_from_record → graph.ainvoke
                 → update_rewritten_data_func
  → in_flight_ids.discard(record_id)
```

`run_one_rewritten` 及内部对 `update_status(rec.id, STATUS_PROCESSING)` 的时机可由其内部在真正执行前更新（若需与现有逻辑一致，可在消费者取到任务后、执行前先更新为 processing，再执行 `run_one_rewritten`；或保留在 flow 内由 update_rewritten_data_func 写 success/failed）。具体是否在消费者内先置 processing 可与现有 `run_one_rewritten` 行为保持一致（当前 `run_one_rewritten` 未在入口置 processing，由原 worker 在拉取后统一置；改为队列后可在消费者取到任务后、调用 `run_one_rewritten` 前先 `update_status(record_id, STATUS_PROCESSING)`，与旧语义一致）。

## 8. 风险与扩展

- **进程重启**：队列在内存，重启后队列与 `in_flight_ids` 清空；未消费的 id 仍为 init/processing，用户可再次点击「运行」重新入队。
- **多实例**：若未来部署多个 uvicorn 进程，每个进程有独立队列，同一批次可能在多实例各入队一次，导致重复执行。届时需引入分布式队列（如 Redis）和分布式去重（如 Redis Set 记录 in_flight_ids）。
- **模式 1/2 与执行中任务**：清空队列、移除批次均只操作「尚未被消费者取走的任务」；已被消费者取走正在执行的任务不会被打断，执行完后正常结束并从 `in_flight_ids` 移除。
