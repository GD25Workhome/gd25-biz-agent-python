# Step02 改写任务：入队、消费与状态变化梳理

本文档梳理从任务被加入队列（所有入队方式）、消费者消费、到任务完成或失败的全流程及 `DataItemsRewrittenRecord.status` 的变化情况。相关实现见 `backend/pipeline/rewritten_queue_service.py`、`rewritten_service.py`、`data_items_rewritten_repository.py`，设计文档见 `021105-Step02批次任务队列与运行停止技术设计.md`。

---

## 1. 状态常量

| 状态 | 常量值 | 含义 |
|------|--------|------|
| init | `init` | 待执行，已创建未入队或未开始执行 |
| processing | `processing` | 执行中（已入队且被消费者取到，或单条再次运行前被接口显式置为 processing） |
| success | `success` | 执行成功，流程内已写入改写结果 |
| failed | `failed` | 执行失败，原因见 `execution_metadata` |

定义位置：`backend/infrastructure/database/repository/data_items_rewritten_repository.py`（`STATUS_*`）。

---

## 2. 任务入队方式汇总

所有入队均通过内存队列，队列元素为 `(record_id: str, batch_code: str)`，并用 `_in_flight_ids: Set[str]` 做按 `record_id` 去重，避免同一任务在队列中重复出现。

| 入队方式 | 入口 | 条件与行为 |
|----------|------|------------|
| **按批次入队** | `POST /rewritten-batches/run` → `enqueue_batch(batch_code, session)` | 1）按 `batch_code` 校验批次存在；2）`get_pending_by_batch_code(batch_code)` 只拉取 **status ∈ {init, processing}** 且 `source_*` 非空的记录；3）对每条记录若 `rid not in _in_flight_ids` 则加入 `_in_flight_ids` 并 `put_nowait((rid, batch_code))`。**已 success/failed 的任务不会被查出来，因此不会入队。** |
| **单条再次运行** | `POST /data-items-rewritten/{item_id}/rerun` → `enqueue_one(item_id, batch_code)` | 1）先将该条 DB 状态更新为 **processing**；2）若 `record_id not in _in_flight_ids` 则入队并加入 `_in_flight_ids`，否则返回“该任务已在队列或执行中”。**调用方负责先改状态再入队。** |

说明：

- **清空队列**（`clear_all_queue`）、**按批次移除**（`remove_batch_from_queue`）只从队列中移除元素并从 `_in_flight_ids` 中移除对应 id，**不会**修改 DB 中记录的 status。
- 入队时 **不会** 把 DB 状态从 init 改为 processing；按批次入队时 DB 仍为 init/processing，单条再次运行时由接口在入队前改为 processing。

---

## 3. 入队时的状态与去重

- **按批次入队**  
  - 只选择 **init / processing** 的记录，success/failed 不会再次入队。  
  - 同一 `record_id` 若已在 `_in_flight_ids`（已在队列或正被消费），不会再次入队。

- **单条再次运行**  
  - 任意当前状态都可触发（包括 success/failed）；接口先把该条改为 **processing** 再入队。  
  - 若该 id 已在 `_in_flight_ids`，则不入队，返回“该任务已在队列或执行中”。

---

## 4. 消费者流程与状态变化

消费者由 `start_consumers()` 启动多个 `_consumer_loop()` 协程，每个循环：从队列取 `(record_id, _batch_code)` → 查库 → 执行或标记失败 → 最后从 `_in_flight_ids` 移除该 id。

### 4.1 消费者主流程（伪代码）

```
await _queue.get() → (record_id, _batch_code)
session 查库: rec = repo.get_by_id(record_id)
if not rec:
    _mark_failed(record_id, "记录不存在")   → DB: status = failed
else:
    run_one_rewritten(rec)                 → 见下文
finally:
    _in_flight_ids.discard(record_id)
```

### 4.2 run_one_rewritten 内状态变化

| 情况 | 行为 | 结果 status |
|------|------|-------------|
| `source_dataset_id` 或 `source_item_id` 为空 | 本方法内 `update_status(rec.id, STATUS_FAILED, ...)` | **failed** |
| 根据 source 查不到 `DataSetsItemsRecord` | 本方法内 `update_status(rec.id, STATUS_FAILED, ...)` | **failed** |
| 正常进入流程 | 调用 `_run_one(...)`，内部执行 `graph.ainvoke`；流程最后一个节点 `update_rewritten_data_func` 根据结果写库 | **success** 或 **failed** |
| `_run_one` 内部异常（如 graph.ainvoke 抛错） | 异常抛到消费者，消费者 `except` 里调用 `_mark_failed(record_id, str(e))` | **failed** |

流程内成功/失败由 **update_rewritten_data_func** 决定：若 `edges_var` 有有效改写数据则 `update_status(..., STATUS_SUCCESS)`，否则 `STATUS_FAILED`，并写 `execution_metadata`（见 `backend/domain/flows/implementations/update_rewritten_data_func.py`）。

### 4.3 消费者侧标记失败

以下情况由消费者直接调用 `_mark_failed(record_id, reason)`，将记录置为 **failed**：

- `get_by_id(record_id)` 得到 `None`（记录不存在）；
- 执行 `run_one_rewritten(rec)` 过程中抛异常。

## 5. 端到端状态流转小结

```
[创建]  create_init_batch / 创建接口
    → status = init

[入队]  仅以下两种方式会向队列追加任务：
    • 按批次运行：get_pending_by_batch_code(batch_code) → status ∈ {init, processing} 的才入队
    • 单条再次运行：接口先把该条改为 processing，再 enqueue_one

[消费]  消费者取 (record_id, batch_code)
    → get_by_id(record_id)
    → 无记录：_mark_failed → failed
    → 有记录：run_one_rewritten(rec)
        → 前置校验失败（source 为空、来源记录不存在）→ update_status failed
        → 流程正常结束 → update_rewritten_data_func 写 success 或 failed
        → 流程或 run_one_rewritten 抛异常 → 消费者 _mark_failed → failed
    → finally: _in_flight_ids.discard(record_id)
```

**重要**：消费者和 `run_one_rewritten` 当前**不会**在“记录已是 success”时跳过执行。若同一 `record_id` 因重复入队等原因再次被消费，会再执行一次流程（幂等性由业务/流程保证，队列层不保证）。
