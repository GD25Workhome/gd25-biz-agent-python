# Step02 清洗批次管理模块技术设计

## 1. 背景与目标
- Step02 阶段已支持在 `pipeline_data_items_rewritten` 表内按批次记录改写结果，但缺少面向运营/开发的批次维度看板，无法查看每个批次的整体进展。
- 现有 `frontend/data-cleaning.html` 侧边栏中只有「Step02 数据清洗管理」用于数据项明细，没有批次汇总页面。
- 本设计旨在新增「Step02清洗批次管理」菜单及列表页，展示 `RewrittenBatchRecord` 的元数据，并补充与 `DataItemsRewrittenRecord` 之间的统计信息，方便排查和对账。

## 2. 需求拆解
| # | 需求 | 说明 |
|---|---|---|
| 1 | 新增菜单项 | 侧边栏在「Step02 数据清洗管理」之前插入「Step02清洗批次管理」，点击后打开新 Tab。 |
| 2 | 列表数据源 | 读取 `backend/infrastructure/database/models/rewritten_batch.py` 中的全部字段。 |
| 3 | 统计字段 | 追加实际关联的 `DataItemsRewrittenRecord` 数量及按 status（init/processing/success/failed）分组的计数。 |
| 4 | 查询与分页 | 查询条件仅支持 `batch_code`（模糊匹配），分页行为与其他页面一致（`limit`+`offset`，前端显示页码）。 |
| 5 | 操作列 | 先保留空白位置，后续迭代再填充。 |

## 3. 数据模型现状梳理
- `RewrittenBatchRecord`（`backend/infrastructure/database/models/rewritten_batch.py`）
  - 核心字段：`batch_code`、`total_count`、`create_params`、`status`、`created_at`、`updated_at`。
  - `total_count` 表示批次创建时预计的任务数量，为静态值。
- `DataItemsRewrittenRecord`（`backend/infrastructure/database/models/data_items_rewritten.py`）
  - 通过 `batch_code` 与批次建立逻辑关联。
  - 状态枚举在 `backend/infrastructure/database/repository/data_items_rewritten_repository.py` 中定义（`STATUS_INIT` 等），需要复用避免魔法字符串。

## 4. 后端设计
### 4.1 API 设计
- 新增 FastAPI 路由：`GET /rewritten-batches`
  - Query 参数：`batch_code`（Optional[str]，模糊匹配）、`limit`（int，默认 20）、`offset`（int，默认 0）。
  - 响应字段：
    - `items`: `List[RewrittenBatchWithStats]`
      - `id`, `batch_code`, `total_count`, `status`, `create_params`, `created_at`, `updated_at`
      - `data_items_total`: 实际存在于 `DataItemsRewrittenRecord` 的数量
      - `status_init_count` / `status_processing_count` / `status_success_count` / `status_failed_count`
    - `total`: 满足查询条件的批次数。
- API Schema：
  - 在 `backend/app/api/schemas/data_cleaning.py` 新增 `RewrittenBatchStats`（响应模型，含 typed 字段与默认值 0）。

### 4.2 Repository 改造
- 文件：`backend/infrastructure/database/repository/rewritten_batch_repository.py`
- 新增方法 `async def get_batches_with_stats(...):`
  - 入参：`batch_code: Optional[str]`, `limit: int`, `offset: int`.
  - 查询方案：
    1. 基于 `RewrittenBatchRecord` 构建基础查询与 `count` 查询。
    2. 通过 `select()` + `outerjoin` 到 `DataItemsRewrittenRecord`，或者使用子查询进行聚合。
    3. 推荐使用子查询统计，避免多倍行数：
       ```sql
       SELECT
         t.id,
         t.batch_code,
         t.total_count,
         ...
         COALESCE(stats.data_items_total, 0) AS data_items_total,
         COALESCE(stats.status_init_count, 0) AS status_init_count,
         ...
       FROM rewritten_batches AS t
       LEFT JOIN (
         SELECT
           batch_code,
           COUNT(*) AS data_items_total,
           COUNT(*) FILTER (WHERE status = 'init') AS status_init_count,
           ...
         FROM data_items_rewritten
         GROUP BY batch_code
       ) AS stats ON stats.batch_code = t.batch_code;
       ```
    4. SQLAlchemy 实现可通过 `func.count().filter(...)` 或 `func.sum(case(...))` 完成。
  - 返回 `(List[RewrittenBatchWithStatsDTO], total)`，DTO 可为 `Dict[str, Any]` 或自定义 dataclass。

### 4.3 路由实现
- 文件：`backend/app/api/routes/data_cleaning.py`
  - 引入 `RewrittenBatchRepository` 并在依赖的 `AsyncSession` 上下文中调用 `get_batches_with_stats`。
  - 将返回值映射到 Pydantic 模型列表。
  - 错误处理：捕获 DB 异常统一转为 `HTTPException(status_code=500, detail="查询批次失败")`。

### 4.4 性能与一致性
- 批次数量相对可控，统计查询引入聚合但可依赖索引：
  - 确保 `pipeline_data_items_rewritten.batch_code` 已有索引（若无需在本迭代补建）。
- 若后续批次数增长，可考虑缓存（e.g. Redis）或按 `status` 增量维护，但本次先以内联聚合为准。

### 4.5 测试策略
- `cursor_test/test_rewritten_batch_api.py`
  - 场景 1：仅 `RewrittenBatchRecord`，无数据项 → 统计字段全部为 0。
  - 场景 2：插入不同 `status` 的数据项，验证统计值正确。
  - 场景 3：`batch_code` 查询模糊过滤、`limit/offset` 分页行为。

## 5. 前端设计
### 5.1 菜单与 Tab
- `frontend/data-cleaning.html`
  - 在 Step02 现有菜单上方新增：

- `frontend/js/pipeline_cleaning.js`
  - `tabConfigs` 增加 `rewritten-batches` 配置，标题与组件保持菜单一致。

### 5.2 新组件 `PipelineRewrittenBatchesComponent`
- 位置：`frontend/js/pipeline_rewritten_batches.js`
- 结构与已有列表组件保持一致：
  - 查询区域：仅一个 `el-input` (`v-model="queryBatchCode"`) + 查询/重置按钮。
  - 表格字段：
    - `batch_code`
    - `total_count`
    - `data_items_total`
    - 四个状态计数字段（可直显示 + 彩色标签）
    - `status`（批次级状态，如无可展示 `-`）
    - `created_at` / `updated_at`
    - 操作列保留空区域（例如放置 `el-space` 或注释提示）。
  - 支持分页组件，与其他页面同样的 `limit`、`offset`、`PAGE_SIZE_OPTIONS`。
- 数据请求：
  ```js
  const params = {
    batch_code: queryBatchCode.value?.trim() || undefined,
    limit: limit.value,
    offset: offset.value
  };
  const res = await axios.get(`${API_PREFIX}/rewritten-batches`, { params });
  ```
- 表格样式复用 `el-table` `stripe border size="small"`，保持 UI 一致。

### 5.3 交互细节
- 初次加载自动触发 `loadBatches()`。
- 查询时重置 `offset`。
- 操作列提供 `div` 占位文本（如 “待迭代”），便于后续加按钮。

## 7. 开发顺序建议
1. **后端**：仓储聚合 → API schema → FastAPI 路由 → 单测。
2. **前端**：新组件模板 → API 接口封装 → 菜单 + Tab 注入 → UI 联调。
3. **联调**：通过现有 `backend/main.py` 启动 + `frontend/data-cleaning.html` 静态页面验证。

## 8. 风险与后续迭代
- 统计口径依赖 `batch_code` 完整性，如存在 `NULL` 或空字符串需在 SQL 中提前过滤。
- 批次状态 `status` 字段当前未严格维护，可能为空，前端需容错显示 `-`。
- 未来可在操作列补充：
  - 一键跳转至「Step02 数据清洗管理」并同步 `batch_code` 过滤。
  - 批次重试/终止等能力。
