# 回复中引用文章链接改造设计

**文档编号**：012702  
**日期**：2026-01-27  
**范围**：medical_agent_v6_1 流程改造，使回复内容中能输出「可点击的引用文章链接」段落。

---

## 一、改造目标与范围

- **目标**：在核心 Agent 的回复中，增加「引用文章」独立段落，以 Markdown 可点击链接形式呈现。
- **流程**：`config/flows/medical_agent_v6_1/flow.yaml`
- **涉及节点**：`optimization_agent` → `rag_node` → `core_agent`

---

## 二、改造点 1：optimization_agent 疾病信息单独输出与提示词组织

### 2.1 需求

- 将「疾病信息」单独作为下游可用字段输出，便于 rag 节点用其查科普文章。
- 提取逻辑沿用 `10-optimization-agent.md` 中与 **疾病类型** 相关的约定（如第 107–108 行：`input_tags` 中的「疾病类型：如 "高血压"、"糖尿病"」）。

### 2.2 方案要点

1. **新增输出字段**  
   在 optimization_agent 的 JSON 输出中增加 **`disease_for_article`**（或 `disease_tags`）：
   - 类型：字符串数组。
   - 含义：仅包含从当前上下文中识别出的「疾病类型」标签，供下游 rag 查文章使用。
   - 示例：`["高血压"]`、`["高血压","糖尿病"]`，无疾病则为 `[]`。

2. **与现有 `input_tags` 的关系**  
   - 语义上等价于从 `input_tags` 里筛出「疾病类型」那一类。
   - 单独字段的好处：下游不再解析 `input_tags` 的语义，契约清晰；若希望零改动兼容，也可约定 rag 从 `input_tags` 中按约定取「疾病类型」标签。

3. **提示词组织建议（避免臃肿）**  
   - **主说明保持简洁**：在「输出任务说明」里只写「新增字段名 + 含义 + 示例」，例如：
     - `disease_for_article`：从本轮的疾病类型中提取的标签数组，仅用于下游文章检索；无则 `[]`。示例：`["高血压"]`。
   - **详细规则复用**：在正文中用「参见 §xxx 疾病类型」引用已有条款（如 107–108 行附近的 `input_tags` 中疾病类型说明），避免把规则全文再抄一遍。
   - **可选**：若希望进一步减负，可把「血压/心率/症状等规则」抽到独立片段文件（如 `10-optimization-agent-rules.md`），在本提示词中通过「引用片段」引入，本文件只保留结构、输出格式和少量索引式说明。

### 2.3 输出格式变更示例

在现有 JSON 上增加一格即可，例如：

```json
{
  "scene_summary": "string",
  "optimization_question": "string",
  "input_tags": ["tag1", "tag2", ...],
  "response_tags": ["tag1", "tag2", ...],
  "disease_for_article": ["高血压"]
}
```

## 三、改造点 2：rag_node 并行查文章向量库与结果下传

### 3.1 需求

- 在现有「案例检索」（embedding_records）基础上，**并行**检索「科普文章」向量库（`PopularScienceArticle`）。
- 将文章检索结果传入下一节点，供 core_agent 生成「引用文章链接」段落。

### 3.2 现状与缺口

- **flow 与节点类型**：当前 `rag_node` 为 `type: rag_agent`，对应实现为 `RagAgentNodeCreator`。
- **数据来源**：仅从 `edges_var` 取 `scene_summary`、`optimization_question`，只查 `embedding_records`，结果写入 `edges_prompt_vars[output_field]`（默认 `retrieved_examples`）。
- **已有基础**：
  - `PopularScienceArticle` 模型与表已存在（`rag_models.py` 约 244–245 行）。
  - `search_popular_science_articles(disease, top_k, similarity_threshold, min_results)` 已在 `backend/infrastructure/rag/retrieval.py` 中实现，入参为疾病名，返回 `article_material_id`、`article_title`、`article_content`、`similarity` 等。
- **缺口**：
  - rag_agent 未读取「疾病」类信息，未调用 `search_popular_science_articles`。
  - 未把「文章列表」写入 `edges_prompt_vars`，下游 core_agent 拿不到。

### 3.3 方案要点

1. **输入**  
   - 继续使用 `scene_summary`、`optimization_question` 用于案例检索（现有逻辑不变）。  
   - 新增从 `edges_var` 读取 **`disease_for_article`**（优先）或从 `input_tags` 中按约定解析出的疾病列表。  
   - 若为多疾病，可选用「主疾病」一条查文章（如列表第一个），或对每个疾病各查 top_k 再合并去重；设计文档阶段建议先约定「取第一个非空疾病」或「多疾病合并查询」的规则。

2. **并行检索**  
   - 案例检索：沿用当前 `_search_similar_cases`（embedding_records）。  
   - 文章检索：在得到疾病名后，调用 `search_popular_science_articles(disease=..., top_k=..., ...)`。  
   - 两路可并行执行（asyncio 或线程池），以降低耗时。

3. **输出与下传**  
   - `edges_prompt_vars` 中除现有 `retrieved_examples` 外，新增 **`retrieved_articles`**（或 `article_links`）：  
     - 建议存「已渲染好的 Markdown 链接段落」字符串，便于 core 提示词中直接占位替换；  
     - 若偏好结构化，也可传文章对象列表，在「提示词构建 / 格式化」层再转成 Markdown。
   - 文章链接格式需可点击：`[标题](url)`。  
     - URL 若业务已有规则（如 `/article/{article_material_id}`），则在 rag 或共用格式化函数里用 `article_material_id` 拼接；  
     - 若暂无规则，需在表结构或配置中预留「文章详情 URL」来源（如 `article_url` 字段或映射表），并在本改造中一并设计。

4. **配置扩展（可选）**  
   - 在 `rag_agent` 的 flow 配置中增加例如：  
     - `article_top_k`、`article_similarity_threshold`；  
     - `articles_output_field: "retrieved_articles"`，以便与现有 `output_field` 并列、互不覆盖。

### 3.4 基础代码是否需补充

- **向量表与检索**：`PopularScienceArticle` 与 `search_popular_science_articles` 已具备，可直接用。  
- **需补齐的**：  
  - 在 **RagAgentNodeCreator**（或封装出的「文章检索 + 格式化」层）中：读取疾病、调用 `search_popular_science_articles`、将结果格式化为 Markdown 链接段落并写入 `edges_prompt_vars`。  
  - 若采用「可点击链接」：明确 **文章 URL 的来源**（现有 `article_material_id` 如何映射到前端/详情页 URL），必要时在表或配置中增加对应字段或规则。

## 四、改造点 3：core_agent 提示词接受文章并追加输出

### 4.1 需求

- 提示词中能**接受「文章」占位符**，且占位符命名与上一步输出一致（如 `retrieved_articles` 或 `article_links`）。
- 回复内容中**必须有一段是「引用文章」**，且该段为 **独立的、Markdown 格式的可点击链接**。

### 4.2 占位符

- 在 `50-core_agent.md` 中增加占位符，与 `edges_prompt_vars` 中的键一致，例如：  
  **`{retrieved_articles}`** 或 **`{article_links}`**。  
- 该处填入的应是「已成型的一段 Markdown」，例如：

```markdown
## 参考文章
- [文章标题1](url1)
- [文章标题2](url2)
```

或「无文章时」的占位内容，如「（暂无推荐文章）」，由 rag 侧统一生成，避免 core 自由发挥格式。

### 4.3 回复结构与约束

- 在「输出格式要求」或「行为规则」中明确：  
  - 在正常回复正文之后，**必须**增加单独一段：「引用/参考文章」；  
  - 该段仅允许使用给定的 `{retrieved_articles}` 内容，或在此基础上做最小编辑（如加一句引导语），**不得**自己编造链接或标题。  
- 若希望前端渲染可点击，需保证该段是纯 Markdown 链接语法，且 URL 可用（由上文「文章 URL 规则」保证）。

### 4.4 与现有占位符的关系

- 现有已有 `{retrieved_examples}`、`{llm_context_part}`、`{llm_rule_part}`、`{end_llm_resopnse}` 等；  
- 新增 `{retrieved_articles}` 与它们并列，在 `build_system_message` 时统一从 `edges_prompt_vars` 取值即可（`sys_prompt_builder` 已支持按名替换）。

## 五、其它需考虑的补充点

1. **文章 URL 与前端约定**  
   - **已约定**：文章详情链接地址为 `http://localhost:8000/api/v1/articles/{id}`，其中 `{id}` 为文章 id（实现中优先使用 `article_material_id`，若无则使用 `id`）。  
   - 若后续前端路由变更，可在 RagAgentNodeCreator 中调整常量 `ARTICLE_BASE_URL` 或通过配置注入。

2. **多疾病时的文章数量与去重**  
   - 多疾病时是否对每种疾病都查一次、合并后按相似度或标题去重；或只取「主疾病」一次查够 top_k。  
   - 需在 rag 设计里写死一种策略，避免同一轮对话中文章过多或重复。

3. **无疾病时的行为**  
   - 当 `disease_for_article` 为空或未识别到疾病时，rag 不调文章检索，`retrieved_articles` 置为「（暂无推荐文章）」之类占位文案，并在 core 提示词中说明「仅当本段有内容时输出该段」。

4. **降级与性能**  
   - 文章检索超时或异常时，应对策略：仅落空为「暂无推荐文章」而不影响案例检索与主回复；若采用并行，需避免文章检索拖慢整体耗时（如设独立超时或限流）。

5. **埋点与效果评估**  
   - 若需统计「点击了某篇推荐文章」等行为，需在「文章链接」上预留统计参数或标识（如来源=medical_agent_v6_1、会话 id等），可在实施阶段再细化。

6. **flow 编排**  
   - 当前边为：builder_prompt_context_node → optimization_agent → rag_node → core_agent。  
   - 本改造不改变节点与边，只扩展示例/文章两条数据线；若未来要把「文章」拆成独立节点再合并，可再做一期编排改造，本次不改。

## 六、数据流与依赖关系简图

```
builder_prompt_context_node
    │
    ▼
optimization_agent
    │ 输出：scene_summary, optimization_question, input_tags, response_tags, disease_for_article
    ▼
rag_node (rag_agent)
    │ 输入：scene_summary, optimization_question, disease_for_article
    │ 检索：embedding_records（案例） || PopularScienceArticle（文章）
    │ 输出到 edges_prompt_vars：retrieved_examples, retrieved_articles
    ▼
core_agent
    │ 占位符：{retrieved_examples}, {retrieved_articles}, ...
    │ 行为：正文 + 必选「引用文章」段（Markdown 可点击链接）
    ▼
用户可见回复
```

## 七、实施顺序建议

1. **契约与配置**：定稿「疾病」输出字段名（`disease_for_article`）与「文章」占位符名（`retrieved_articles`）、文章 URL 规则。  
2. **optimization_agent**：改输出 schema 与提示词（含对疾病类型的引用说明），保证 `disease_for_article` 稳定输出。  
3. **rag_node**：在 RagAgentNodeCreator 中接好 `disease_for_article`、调用 `search_popular_science_articles`、格式化 `retrieved_articles` 并写入 `edges_prompt_vars`。  
4. **core_agent**：在 50-core_agent 中增加 `{retrieved_articles}` 占位符与「引用文章」段的输出规范。  
5. **联调与降级**：验证无疾病/无文章/检索失败时的占位文案与前端链接可点性；必要时补 URL 来源与埋点。
