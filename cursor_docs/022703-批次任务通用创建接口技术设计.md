## 1. 背景与目标

### 1.1 背景

- 在 `022701-批次任务创建模版设计方案.md` 中，已经定义了通用的批次任务创建模版类 `BatchCreateTemplate`，以及基于 `job_type` 的任务类型抽象。
- 在 `022702-Step03批量Embedding前后端改造清单.md` 中，针对「Step03 批量创建 Embedding」设计了一个 **Embedding 专用** 的批次创建接口方案（`/data-cleaning/embedding-batches/execute`）。
- 随着后续业务扩展（例如新增其它批次任务类型），如果继续为每种业务单独增加 API，将导致接口数量膨胀、复用度低。

### 1.2 目标

- 在 **API 层 / Service 层 / Domain 层** 之间，建立一条 **与具体业务解耦的“通用批次创建链路”**：
  - API 只关心「创建一个 `job_type` 对应的批次任务」，而不关心具体业务字段。
  - Service 负责将 `job_type` 分发到对应的 `BatchCreateTemplate` 实现（创建 Handler 注册表）。
  - Domain 层的各个 Handler 负责解释自己的 `query_params`、校验业务字段并组装 `TaskPreCreateItem`。
- 通过这条链路，使 Embedding 只是众多 `job_type` 中的一种，例如：`"pipeline_embedding"`。

> 说明：本设计文档中的接口是 **通用接口**，后续 Embedding 专用方案（`022702` 中的 `/embedding-batches/execute`）将标记为废弃，并通过文档指向本方案。

## 2. 总体设计概览

### 2.1 关键组件

- **API 层（新的独立路由与 Schema 模块）**
  - 新增路由模块：`backend/app/api/routes/batch_jobs.py`。
  - 新增 Schema 模块：`backend/app/api/schemas/batch_jobs.py`（定义 `BatchJobCreateRequest`、`BatchJobCreateResponse`；路由仅从此模块导入，与 data_cleaning 解耦）。
  - 暴露统一的批次创建接口：`POST /batch-jobs/create`。
  - 请求体只包含：
    - `job_type: str`：批次任务类型（如 `"pipeline_embedding"`）。
    - `query_params: dict`：与具体 `job_type` 绑定的参数对象（原样透传给对应 Handler）。

- **Service 层（批次创建服务）**
  - 新增服务对象：`BatchJobCreateService`（建议路径：`backend/domain/batch/services.py`）。
  - 负责：
    - 维护 `job_type -> BatchCreateTemplate` 映射（Handler 注册表）。
    - 提供统一入口：`async create_batch(session, job_type, query_params) -> BatchJobRecord`。
    - 封装 Handler 创建与依赖注入（`BatchJobRepository`、`BatchTaskRepository`）。

- **Domain 层（批次创建模板及各业务 Handler）**
  - 已有抽象：
    - `BatchCreateTemplate`（`backend/domain/batch/create_template.py`）。
    - `TaskPreCreateItem`（`backend/domain/batch/dto.py`）。
  - 新增 / 落地 Embedding 业务 Handler：
    - `PipelineEmbeddingBatchHandler`：继承 `BatchCreateTemplate`，`job_type="pipeline_embedding"`。
    - 负责解析 `query_params`、通过 `DataItemsRewrittenRepository` 查询 Step02 结果，并转成 `TaskPreCreateItem` 列表。

### 2.2 调用链

```text
前端（Step03 批量创建 Embedding）
    ↓ POST /batch-jobs/create  body: { job_type: "pipeline_embedding", query_params: {...} }
API（batch_jobs.py）
    ↓ 调用 BatchJobCreateService.create_batch(session, job_type, query_params)
Service（BatchJobCreateService）
    ↓ 根据 job_type 从注册表获取对应 Handler 工厂
    ↓ 实例化 PipelineEmbeddingBatchHandler（或其它 Handler）
    ↓ 调用 handler.create_batch(session, query_params)
Domain（BatchCreateTemplate & Handler）
    ↓ query_tasks_for_pre_create 查询待创建任务
    ↓ generate_batch_code 生成批次 code
    ↓ _build_and_persist_batch 创建 batch_job + batch_task
Infra（BatchJobRepository / BatchTaskRepository）
    ↓ 持久化到 batch_job / batch_task 表
返回 BatchJobRecord → API 封装成响应 JSON 返回前端
```

## 3. API 设计（batch_jobs.py）

### 3.1 路由模块

- 新增文件：`backend/app/api/routes/batch_jobs.py`。
- 内容要点：
  - 使用独立的 `APIRouter`，避免与 `data_cleaning.py` 混在一起。
  - 路由前缀建议为 `/batch-jobs`，标签为 `"批次任务"`。

示意结构（伪代码）：

```python
router = APIRouter(prefix="/batch-jobs", tags=["批次任务"])

@router.post("", response_model=BatchJobCreateResponse)
async def create_batch_job_endpoint(
    data: BatchJobCreateRequest,
    session: AsyncSession = Depends(get_async_session),
):
    ...
```

> 说明：本路由模块不再出现「embedding」「rewritten」等具体业务字段命名，所有业务信息都通过 `job_type` + `query_params` 间接表达。

### 3.2 请求 / 响应 Schema

- **文件位置（独立拆分）**  
  - 新增独立 Schema 模块：`backend/app/api/schemas/batch_jobs.py`。  
  - 本接口的请求/响应模型均定义在该文件中，**不复用** `data_cleaning.py` 中的 schema。  
  - 路由模块 `backend/app/api/routes/batch_jobs.py` **仅从** `backend.app.api.schemas.batch_jobs` 导入上述模型，与 data_cleaning 的 API 入口及 Schema 完全解耦，满足「批次任务接口独立」的前提。

- **请求模型：`BatchJobCreateRequest`**

  - 字段定义：
    - `job_type: str`  
      - 描述：批次任务类型，用于从注册表中查找具体 Handler。
      - 示例值：`"pipeline_embedding"`、`"data_clean"` 等。
    - `query_params: Optional[dict]`  
      - 描述：与 `job_type` 配套的筛选 / 创建参数对象，格式由各 `job_type` 自行约定。

- **响应模型：`BatchJobCreateResponse`**

  - 字段定义：
    - `success: bool`：是否成功创建批次。
    - `message: str`：提示信息。
    - `job_type: str`：回显本批次的类型。
    - `batch_code: str`：批次编码（`batch_job.code`）。
    - `total: int`：创建的任务总数（`batch_job.total_count`）。

> 对于 Embedding 来说，前端只需保证：`job_type="pipeline_embedding"` 且按 Embedding 约定组织 `query_params`，不需要使用专用的 Embedding Request/Response 类型。

### 3.3 接口行为

1. **基础校验**
   - `job_type` 不可为空。
   - 当 `query_params` 不是对象（例如传了数组 / 标量）时，返回 400。
2. **调用 Service**
   - 构造服务实例：`BatchJobCreateService`（可以通过依赖注入或在路由内部构造）。
   - 调用：`job = await service.create_batch(session, job_type=data.job_type, query_params=data.query_params or {})`
3. **事务处理**
   - 成功：`await session.commit()`。
   - 异常：`await session.rollback()` + 记录日志。
4. **异常与错误码**
   - **业务异常（例如未知 job_type、参数不合法）**：
     - 由 Service 或 Handler 抛出自定义异常（例如 `UnknownJobTypeError`, `InvalidJobParamsError`），在路由层统一转换为 `HTTPException(status_code=400, detail=...)`。
   - **系统异常**：
     - 转成 500，日志中记录 `job_type` 和 `query_params` 摘要。

## 4. Service 设计：BatchJobCreateService

### 4.1 位置与职责

- 建议路径：`backend/domain/batch/batch_job_service.py`。
- 核心职责：
  - 维护 `job_type -> HandlerFactory` 的注册表；
  - 提供统一入口 `create_batch(session, job_type, query_params)`；
  - 将基础设施依赖（`BatchJobRepository`、`BatchTaskRepository` 等）注入给 Handler。

### 4.2 Handler 注册表

- 设计为「字符串常量 + 工厂函数」的形式，避免循环依赖。
- 示例：

```python
JOB_TYPE_PIPELINE_EMBEDDING = "pipeline_embedding"

HandlerFactory = Callable[[AsyncSession], BatchCreateTemplate]

class BatchJobCreateService:
    def __init__(self, session: AsyncSession) -> None:
        self._session = session
        self._registry: Dict[str, HandlerFactory] = {
            JOB_TYPE_PIPELINE_EMBEDDING: self._create_pipeline_embedding_handler,
            # 未来可在此处注册更多 job_type
        }
```

### 4.3 create_batch 流程

```python
async def create_batch(
    self,
    job_type: str,
    query_params: Optional[Dict[str, Any]] = None,
) -> BatchJobRecord:
    # 1. 查找 job_type 对应的 Handler 工厂
    factory = self._registry.get(job_type)
    if not factory:
        raise UnknownJobTypeError(job_type)

    # 2. 构造 Handler 实例
    handler = factory(self._session)

    # 3. 调用 Handler.create_batch
    job = await handler.create_batch(
        session=self._session,
        query_params=query_params or {},
    )
    return job
```

> 注意：这里完全不关心具体的 `query_params` 结构，校验责任由具体 Handler 承担。

### 4.4 Embedding Handler 工厂

- 在 `BatchJobCreateService` 内部定义：

```python
def _create_pipeline_embedding_handler(self, session: AsyncSession) -> BatchCreateTemplate:
    job_repo = BatchJobRepository(session)
    task_repo = BatchTaskRepository(session)
    return PipelineEmbeddingBatchHandler(job_repo=job_repo, task_repo=task_repo)
```

## 5. Domain 设计：PipelineEmbeddingBatchHandler（示例）

### 5.1 类位置与签名

- **文件路径**：`backend/domain/batch/pipeline_embedding_batch_handler.py`
- **类名**：`PipelineEmbeddingBatchHandler`
- **继承关系**：继承 `BatchCreateTemplate`（`backend/domain/batch/create_template.py`）
- **依赖注入**：
  - 构造函数参数：
    - `job_repo: BatchJobRepository`
    - `task_repo: BatchTaskRepository`
- **使用方式**：仅通过 `BatchJobCreateService`（`backend/domain/batch/batch_job_service.py`）内部工厂创建，不在其他地方直接实例化。

```python
class PipelineEmbeddingBatchHandler(BatchCreateTemplate):
    JOB_TYPE = "pipeline_embedding"

    def get_job_type(self) -> str: ...
    async def query_tasks_for_pre_create(
        self,
        session: AsyncSession,
        query_params: Optional[Dict[str, Any]] = None,
    ) -> List[TaskPreCreateItem]: ...
```

### 5.2 职责边界

- **只负责“查出要做 Embedding 的记录并转换为 TaskPreCreateItem 列表”**：
  - 不直接操作 `batch_job` / `batch_task` 表（由父类模版负责）。
  - 不直接执行 Embedding（由运行期的执行器 / Worker 负责）。
- **对自身 job_type 的参数进行业务校验**：
  - 确认 `source_dataset_id` 等关键字段是否存在。
  - 负责将“参数不合法”转化为明确的业务异常，便于 API 层返回 400。

### 5.3 query_params 约定与校验规则

#### 5.3.1 字段约定

对 `job_type="pipeline_embedding"`，`query_params` 的结构约定如下（JSON 示例）：

```json
{
  "source_dataset_id": "xxx",
  "scenario_type": "xxx",
  "sub_scenario_type": "xxx",
  "status": "success",
  "scenario_description": "...",
  "rewritten_question": "...",
  "rewritten_answer": "...",
  "rewritten_rule": "...",
  "source_item_id": "...",
  "batch_code": "...",
  "trace_id": "..."
}
```

含义与接口 `GET /data-cleaning/data-items-rewritten` 的查询参数保持一致：

- `source_dataset_id`：来源 DataSetsId，**精确匹配**（必填）。
- `scenario_type` / `sub_scenario_type`：场景类型 / 子场景类型（模糊匹配）。
- `status`：执行状态，枚举：`init` / `processing` / `success` / `failed`，默认 `success`。
- 其它字段：`scenario_description`、`rewritten_question`、`rewritten_answer`、`rewritten_rule`、`source_item_id`、`batch_code`、`trace_id`，全部为“包含匹配”。

#### 5.3.2 校验逻辑

在 `query_tasks_for_pre_create` 内部应做的校验：

- **必填项**：
  - 若缺失或为空：
    - `source_dataset_id`：直接抛业务异常（例如 `InvalidJobParamsError("source_dataset_id 不能为空")`）。
- **枚举值**：
  - `status` 若非空，必须在 `{"init", "processing", "success", "failed"}` 中。
- **类型与结构**：
  - `query_params` 必须是 `dict`：
    - 如果 `query_params` 不是字典（例如字符串 / 数组），直接抛业务异常。

### 5.4 查询实现：使用 DataItemsRewrittenRepository

#### 5.4.1 仓储方法建议

为避免分页，需要在 `DataItemsRewrittenRepository` 中新增一个“无分页查询”方法（在本设计只做约定，具体实现见仓储层）：

```python
async def get_all_for_embedding(
    self,
    *,
    source_dataset_id: str,
    scenario_description: Optional[str] = None,
    rewritten_question: Optional[str] = None,
    rewritten_answer: Optional[str] = None,
    rewritten_rule: Optional[str] = None,
    source_item_id: Optional[str] = None,
    scenario_type: Optional[str] = None,
    sub_scenario_type: Optional[str] = None,
    batch_code: Optional[str] = None,
    trace_id: Optional[str] = None,
    status: Optional[str] = None,
) -> List[DataItemsRewrittenRecord]:
    """
    无分页查询，筛选条件语义与 get_list_with_total 一致，
    用于“创建批次任务”场景下，一次性拉出本次批次要处理的全部记录。
    """
```

语义与 `get_list_with_total` 完全一致，只是去掉了 `limit/offset`，并按固定顺序（例如 `created_at` 升序或降序）返回。

#### 5.4.2 Handler 内部查询流程

`query_tasks_for_pre_create` 的推荐实现步骤：

1. **解析与校验 query_params**：
   - 确保为 `dict`。
   - 按 5.3.2 的规则校验必填字段与枚举值。
2. **构造仓储查询参数**：
   - 将 `query_params` 中的键安全地取出并映射给 `get_all_for_embedding`：
     - 字段对齐表：

       | query_params key      | 仓储方法参数名       |
       |----------------------|----------------------|
       | source_dataset_id    | source_dataset_id    |
       | scenario_description | scenario_description |
       | rewritten_question   | rewritten_question   |
       | rewritten_answer     | rewritten_answer     |
       | rewritten_rule       | rewritten_rule       |
       | source_item_id       | source_item_id       |
       | scenario_type        | scenario_type        |
       | sub_scenario_type    | sub_scenario_type    |
       | batch_code           | batch_code           |
       | trace_id             | trace_id             |
       | status               | status               |

3. **调用仓储**：

   - 创建仓储实例：`repo = DataItemsRewrittenRepository(session)`。
   - 调用：`records = await repo.get_all_for_embedding(...)`。
4. **空结果处理**：
   - 若 `records` 为空：
     - 可以直接返回空列表，让模版层抛出「query_tasks_for_pre_create 返回空列表」的异常，表示当前条件下没有可用于 Embedding 的数据。
5. **将记录转成 TaskPreCreateItem 列表（见 5.5）**：
   - 对于每条 `DataItemsRewrittenRecord`，生成两条 `TaskPreCreateItem`：
     - 一条用于「场景描述 + 改写问题」的 Embedding（类型记为 `Q`）；
     - 一条用于「场景描述 + 改写问答 + 规则」的 Embedding（类型记为 `QA`）。

### 5.5 TaskPreCreateItem 映射规则

`TaskPreCreateItem`（`backend/domain/batch/dto.py`）结构：

```python
@dataclass
class TaskPreCreateItem:
    source_table_id: Optional[str] = None
    source_table_name: Optional[str] = None
    runtime_params: Optional[Dict[str, Any]] = None
    redundant_key: Optional[str] = None
```

Embedding 业务建议的映射规则：

- `source_table_id`：
  - 对应 `DataItemsRewrittenRecord.id`（即 `pipeline_data_items_rewritten.id`）。
- `source_table_name`：
  - 固定写 `"pipeline_data_items_rewritten"`。
- `runtime_params`：
  - 仅保留与执行期强相关的最小字段：
    - `pipeline_data_items_rewritten_id`：与 `source_table_id` 相同；
    - `embedding_type`：Embedding 类型，取值为：
      - `"Q"`：表示按「场景描述 + 改写问题」进行 Embedding；
      - `"QA"`：表示按「场景描述 + 改写问答 + 规则」进行 Embedding。
- `redundant_key`：
  - 取单一的改写记录 id，便于排查和去重，例如：
    - `str(record.id)`。

伪代码示例：

```python
items: List[TaskPreCreateItem] = []
for rec in records:
    redundant_key = str(rec.id)

    # 记录 1：Q（场景描述 + 改写问题）
    items.append(
        TaskPreCreateItem(
            source_table_id=rec.id,
            source_table_name="pipeline_data_items_rewritten",
            runtime_params={
                "pipeline_data_items_rewritten_id": rec.id,
                "embedding_type": "Q",
            },
            redundant_key=redundant_key,
        )
    )

    # 记录 2：QA（场景描述 + 改写问答 + 规则）
    items.append(
        TaskPreCreateItem(
            source_table_id=rec.id,
            source_table_name="pipeline_data_items_rewritten",
            runtime_params={
                "pipeline_data_items_rewritten_id": rec.id,
                "embedding_type": "QA",
            },
            redundant_key=redundant_key,
        )
    )
return items
```

### 5.6 get_job_type 实现约定

- `get_job_type()` 必须返回与注册表中常量一致的字符串，例如：

```python
JOB_TYPE_PIPELINE_EMBEDDING = "pipeline_embedding"
```

- 要求：
  - `BatchJobCreateService` 的注册表 key 使用同一个常量；
  - `batch_job.job_type` 最终值即为该常量，用于运行期按类型路由到对应的执行器（Embedding 执行器）。

### 5.7 错误处理与日志建议

- **参数错误**：
  - 在 Handler 内部检测到 `query_params` 不符合约定（如缺少 `source_dataset_id`、status 非法等），抛出业务异常（例如 `InvalidJobParamsError`），Service 层将其转换为 HTTP 400。
- **查询异常**：
  - 仓储抛出的数据库异常向上冒泡，由上层（Service / API）统一处理并转为 HTTP 500。
- **日志**：
  - 建议在 Handler 中对关键路径打印 debug 日志（带上 `job_type`、`source_dataset_id` 以及命中的记录数量），便于后续排查：
    - 创建前：记录筛选条件摘要；
    - 创建后：记录 `len(task_items)`。

> 有了以上约定，你后续只需要：
> 1）按本节定义实现 `PipelineEmbeddingBatchHandler`；  
> 2）在 `BatchJobCreateService` 注册 `job_type="pipeline_embedding"` 对应的 Handler 工厂；  
> 即可完成 Embedding 批次创建链路的 Domain 部分实现。

## 7. 小结

- 本设计将「创建批次任务」从 Embedding 专用接口，抽象为：
  - **一个通用 HTTP 入口**：`POST /batch-jobs/create`；
  - **一个统一的 Domain Service**：`BatchJobCreateService`；
  - **一组按 job_type 扩展的 Handler**：例如 `PipelineEmbeddingBatchHandler`。
- 通过 `job_type + query_params` 的契约，使 API / Service 层与具体业务解耦，Embedding 只是其中一个实现，后续任意新批次类型都可以通过注册表扩展，而不需要再新增专用接口与 Schema。、
