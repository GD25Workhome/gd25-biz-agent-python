# 工业界数据清洗方案与问答案例清洗设计

本文档介绍工业界在 NLP / 大模型 / RAG 场景下的数据清洗与质量控制实践，并针对「将线上问答案例清洗成问答 Agent 回复案例」的需求，对自拟方案做优缺点点评与补充建议。

---

## 一、工业界数据清洗整体思路

### 1.1 通用数据 Pipeline 三阶段

工业界大模型/对话数据 pipeline 通常分为：

| 阶段 | 目标 | 常见手段 |
|------|------|----------|
| **数据预处理** | 去噪、统一格式、过滤低质/敏感内容 | 规则过滤、格式归一、PII/敏感词过滤、困惑度等质量指标 |
| **数据筛选与去重** | 提纯、控量、避免重复 | 精确/模糊/语义去重，质量打分、分类器筛选 |
| **数据增强与融合** | 多样性、覆盖度、可控规模 | 同义改写、负样本构造、多源混合、人工标注补充 |

你的方案里的「归档统一格式 → 模型粗提取 → 人工筛选 → 入库」与上述「预处理 → 筛选 → 融合/标注」是对齐的，整体方向正确。

### 1.2 质量优先原则

- **质量 > 数量**：小规模、高一致性的精选数据，往往优于大规模、噪声大的数据。
- **级联过滤**：先用轻量规则/启发式做粗筛（高效），再用模型或人工做细筛（高精度），平衡成本与效果。
- **多维度质量**：不仅看「有没有」，还看完整性、准确性、与业务目标的一致性（如是否适合做检索案例）。

### 1.3 人工与自动化分工（Human-in-the-Loop）

工业常见做法：

- **主动学习（Active Learning）**：让模型选出「最值得标注」的样本，减少人工量。
- **人工作为质量门控**：自动流水线产出后，用抽检/全检做质量门控，不合格回流或丢弃。
- **混合标注**：合成数据 + 自动打标 + 人工复核/纠错，迭代提升标注一致性与覆盖率。

你的「模型粗提取 + 人工筛选 + 二次标记与打分」正是典型的 HITL 设计，符合工业实践。

## 二、工业界在「对话/问答」与「RAG」场景的常见做法

### 2.1 对话/问答数据清洗

- **格式与字段统一**：多源日志（API、工单、客服）先映射到统一 schema（如：场景、用户话术、回复、会话轮次、渠道等）。
- **质量过滤**：  
  - 长度、语言、是否包含敏感信息；  
  - 困惑度/流畅度；  
  - 用分类器或 LLM 打「是否适合做示范案例」等标签。
- **去重**：问题级/会话级精确与模糊去重，避免重复案例占满向量空间。
- **场景与意图标签**：规则 + 模型做初标，人工做边界与难例校准。

和你的「归档 → 模型提取场景/提问/回复/标签 → 人工筛选」一致，工业界也会用模型做初标、人工做终审。

### 2.2 RAG / 检索用数据准备

- **内容标准**：  
  - 内容完整、准确、与业务强相关；  
  - 保留必要结构和元数据（来源、时间、场景、权限等）。
- **清洗**：去 HTML/乱码、去广告与无意义段落、结构还原、术语统一等。
- **分块与切片**：按段落/语义/问答对切分，常带重叠，避免语义断裂。
- **向量化与入库**：选合适的 Embedding 与向量库，并考虑多字段（如场景+提问+回复一起建索引或分索引）。

你计划「场景、提问、回复组装为自然语言再 embedding 入库」，属于「问答对/多字段组装为检索单元」的常见做法；若后续改为「回复生成规则」入库，只需在 schema 上增加规则字段即可，流程可复用。

### 2.3 使用方式与工业对齐

你描述的使用方式：

- 用户当前提问 + 基础信息 + 历史 → 模型做「场景识别 + 问题改写」→ 用改写结果做 embedding 查询向量库 → 取优质案例（或生成规则）→ 再生成最终回复。

工业界类似系统通常也会：

- **Query 侧**：对用户输入做归一化（改写、意图/场景分类），再用同一套 embedding 与索引做检索，保证「存什么就查什么」。
- **存储侧**：存「可复用单元」（完整案例或规则），并配合元数据（场景、标签、打分）做过滤或重排。

只要「写入时的组装方式」与「查询时的 query 表示」一致（例如都用「场景 + 改写问句」或「场景 + 问句 + 用户画像」），你的设计是合理的。

## 三、对你当前方案的逐段点评与建议

### 3.1 步骤 1：各类案例归档为同一种格式

**优点：**

- 多源数据统一 schema 是工业标准第一步，便于后续模型与人工在同一套字段上操作。
- 为步骤 2 的「模型统一提取」打好基础。

**建议：**

- 明确 schema：至少包含「原始渠道、原始问/答、会话标识、时间、可选元数据」等，便于回溯与审计。
- 预留扩展字段（如后续要加「回复规则」「适用人群」），避免后期大改表结构。
- 若数据量很大，可在归档时就做一层轻量去重与明显噪声过滤（如空回复、纯乱码），减轻后续压力。

---

### 3.2 步骤 2：模型初步提取（场景、改写提问、改写回复、多类标签）

**优点：**

- 用模型做场景识别、问句/回复改写，能大幅减少人工整理量，符合「自动化 + 人工精修」的级联思路。
- 标签设计（场景 / 提问 / 回复）有利于后续检索过滤与运营分析。

**建议：**

- **改写一致性**：改写后的提问/回复最好在风格、术语、长度上有明确规范（如写进 prompt 或示例），避免同一场景下表述差异过大，影响检索与人工评估。
- **标签体系**：先定一个小而稳的标签集（含场景、意图、质量档位等），再逐步扩展；避免一开始标签过多导致标注不一致。
- **质量初筛**：在模型输出后加一层规则或轻量模型过滤（如改写失败、过短、敏感词），再进入人工，可节省人工时间。
- **可解释性**：若模型能输出「改写依据」或「场景置信度」，有利于人工复核与纠错。

---

### 3.3 步骤 3：人工筛选、二次标记与打分

**优点：**

- 人工作为质量门控，能卡掉模型误判、边界案例和低质样本，保证入库案例「可示范」。
- 二次标记与打分为后续「按场景/质量检索」和运营分析提供依据，也便于做主动学习（优先标注模型不确定的样本）。

**建议：**

- **标注规范与一致性**：编写简明标注指南（含场景定义、打分标准、边界案例），并做少量多人标注一致性检查（如 Kappa），避免标准漂移。
- **抽样与主动学习**：若数据量大，可对模型高置信度样本做抽检，对低置信度/边界样本全检或优先检，提高人效。
- **打分维度**：除「是否优质」外，可考虑「可复用度」「与业务目标一致性」等，便于后续检索时按分数过滤或排序。

---

### 3.4 步骤 4：组装、Embedding、写入向量库

**优点：**

- 将「场景 + 提问 + 回复」组装成自然语言再 embedding，与「用场景 + 改写问句」去查询的使用方式匹配，检索逻辑清晰。
- 预留「回复案例 → 回复生成规则」的演进，兼容后续从「案例检索」到「规则检索」的平滑迁移。

**建议：**

- **分块与索引策略**：若单条内容较长，可考虑按「场景+问」「场景+问+答」分索引或分字段，便于按需检索（如只查问句相似度 vs 整条相似度）。
- **元数据入库**：场景、标签、人工打分等建议一并写入向量库或关联 DB，检索时可按场景/分数过滤，提高命中质量。
- **去重**：入库前对「语义近似」的组装结果做去重（如 embedding 相似度 + 阈值），避免重复案例占满 top-k。
- **版本与更新**：制定增量更新与下线策略（如按时间、版本号），便于后续迭代与 A/B 测试。

## 四、工业界常见补充点（可选增强）

| 维度 | 工业常见做法 | 你可考虑的落地 |
|------|--------------|----------------|
| **去重** | 精确 + 模糊 + 语义多级去重 | 在归档后、入库前各做一次，避免重复案例 |
| **敏感与合规** | PII、敏感词、合规策略统一过滤 | 在模型输出后、人工前加一道自动过滤 |
| **评估与迭代** | 检索准确率/召回率、人工抽检、Badcase 回流 | 定期用历史 query 测检索效果，Badcase 回流到标注或规则 |
| **可观测** | 记录数据来源、处理链路、版本 | 从归档到入库保留 trace（如 case_id、步骤、版本），便于排查与复现 |
| **规模与成本** | 先小规模闭环再放量 | 先做好单场景、单渠道的端到端闭环，再扩展到多场景、多源 |

## 五、小结

- 工业界数据清洗强调：**统一格式 → 级联过滤（规则 + 模型 + 人工）→ 质量优先与去重 → 明确使用方式与检索逻辑**。  
- 你的四步方案（归档 → 模型粗提取 → 人工筛选与打分 → 组装入库）与上述思路一致，且 HITL 设计合理。  
- 建议在「schema 与扩展性、改写与标签规范、人工规范与主动学习、元数据与去重、检索与使用方式一致」上稍作加强，即可在整体设计阶段把方案夯实，便于后续实现与迭代。
