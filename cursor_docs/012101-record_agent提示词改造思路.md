# record_agent 提示词改造思路

## 一、改造目标

在 `11-record_agent.md` 提示词中，增加数据记录完成后的问题加工逻辑，将 `10-before-rag.md` 的功能整合进来，实现：
1. 数据记录成功后，自动优化问题表达
2. 提取核心关键词（1-3个）
3. 提取疾病信息

## 二、当前状态分析

### 2.1 `11-record_agent.md` 当前结构

**核心功能**：
- 数据收集和记录（血压、药品、症状、健康事件）
- 记录成功后返回 `additional_fields`，包含：
  - `record_success`（布尔类型）：是否成功完成记录
  - `record_type`（字符串类型，可选）：实际调用的工具名称

**输出格式**：
```json
{
    "response_content": "回复内容",
    "reasoning_summary": "推理摘要",
    "additional_fields": {
        "record_success": true/false,
        "record_type": "工具名称"  // 可选
    }
}
```

### 2.2 `10-before-rag.md` 功能分析

**核心功能**：
- 优化问题表达：将患者发言改写为适合向量库检索的标准表达
- 提取核心关键词：从优化后的问题中提取1-3个核心关键词
- 提取疾病名：按照优先级顺序提取疾病信息

**输出格式**：
```json
{
    "query_text": "优化后的问题（字符串，必填，不能为空）",
    "keywords": ["关键词1", "关键词2", "关键词3"],
    "disease": "疾病名（字符串，如果未提及疾病则返回空字符串）"
}
```

## 三、改造方案

### 3.1 功能整合策略

**策略**：在数据记录成功后（`record_success == true`），自动执行问题加工逻辑。

**执行时机**：
- 当 `record_success == true` 时，必须执行问题加工
- 当 `record_success == false` 时，可以不执行（或返回空值）

### 3.2 提示词结构调整

#### 3.2.1 核心任务说明部分

在"核心任务说明"中增加：
```
- 调用相应的记录工具
- 确认记录成功
- **记录成功后，优化问题表达并提取关键信息**（新增）
```

#### 3.2.2 工作流程部分

在"第二步：确认记录"之后，增加"第三步：问题加工"：

```
### 第三步：问题加工（仅在记录成功时执行）
当 `record_success == true` 时，你需要：
1. **优化问题表达**：
   - 将用户的最新发言和对话上下文改写为精炼、标准的问题表达
   - 改写原则：保留核心语义，去除口语化表达，使用规范的医疗术语
   - 改写后的问题应该能够直接用于向量库检索

2. **提取核心关键词**：
   - 从优化后的问题中提取1-3个核心关键词
   - 关键词应该是独立的词或短语，能够代表问题的核心概念
   - 关键词应该包含：疾病名、症状、检查项目、操作类型等关键信息

3. **提取疾病名**：
   - 优先级顺序（从高到低）：
     1. 用户当前发言中明确关注或询问的疾病
     2. 用户最新发言中提到的疾病
     3. 历史对话中最重要的疾病
     4. 上下文信息中用户的疾病信息
   - 如果用户有多个疾病，优先选择用户当前最关注的疾病
   - 如果无法确定疾病名，返回空字符串 ""
```

#### 3.2.3 additional_fields 字段说明部分

在 `additional_fields` 中增加三个新字段，与 `record_success` 和 `record_type` 平级：

```
### 必须提供的字段

1. **record_success**（布尔类型，必填）
   - [保持原有说明不变]

2. **record_type**（字符串类型，可选）
   - [保持原有说明不变]

3. **query_text**（字符串类型，条件必填）
   - 含义：优化后的问题文本，用于向量库检索的主查询
   - 取值规则：
     - 当 `record_success == true` 时，必须提供此字段，不能为空
     - 当 `record_success == false` 时，可以不提供此字段（或设置为空字符串）
   - 格式要求：
     - 将用户的最新发言和对话上下文改写为精炼、标准的问题表达
     - 保留核心语义，去除口语化表达，使用规范的医疗术语
     - 改写后的问题应该能够直接用于向量库检索

4. **keywords**（数组类型，条件必填）
   - 含义：核心关键词数组，包含1-3个关键词，用于增强检索效果
   - 取值规则：
     - 当 `record_success == true` 时，必须提供此字段，不能为空数组
     - 当 `record_success == false` 时，可以不提供此字段（或设置为空数组）
   - 格式要求：
     - 从优化后的问题中提取1-3个核心关键词
     - 关键词应该是独立的词或短语，能够代表问题的核心概念
     - 关键词应该包含：疾病名、症状、检查项目、操作类型等关键信息
     - 示例：["高血压", "测量", "记录"]

5. **disease**（字符串类型，条件必填）
   - 含义：疾病名称
   - 取值规则：
     - 当 `record_success == true` 时，必须提供此字段
     - 如果未提及疾病，返回空字符串 ""
   - 提取优先级（从高到低）：
     1. 用户当前发言中明确关注或询问的疾病
     2. 用户最新发言中提到的疾病
     3. 历史对话中最重要的疾病
     4. 上下文信息中用户的疾病信息
   - 如果用户有多个疾病，优先选择用户当前最关注的疾病
   - 如果无法确定疾病名，返回空字符串 ""
```

#### 3.2.4 输出示例更新

更新所有输出示例，包含新增的三个字段：

**场景1：记录成功（需要包含问题加工字段）**
```json
{
    "response_content": "好的，已为您记录血压数据。",
    "reasoning_summary": "用户提供了完整的血压数据（收缩压120，舒张压80，心率70），已成功调用record_blood_pressure工具记录。同时优化了问题表达并提取了关键词和疾病信息。",
    "additional_fields": {
        "record_success": true,
        "record_type": "record_blood_pressure",
        "query_text": "血压测量值120/80是否正常",
        "keywords": ["血压", "测量", "正常值"],
        "disease": "高血压"
    }
}
```

**场景2：信息不完整，需要询问用户（不需要问题加工字段）**
```json
{
    "response_content": "请问您的舒张压是多少？",
    "reasoning_summary": "用户只提供了收缩压120，缺少舒张压，需要询问用户获取完整信息。",
    "additional_fields": {
        "record_success": false,
        "query_text": "",
        "keywords": [],
        "disease": ""
    }
}
```

**场景3：记录失败（不需要问题加工字段）**
```json
{
    "response_content": "抱歉，记录失败了，请稍后再试。",
    "reasoning_summary": "工具调用失败，无法完成记录。",
    "additional_fields": {
        "record_success": false,
        "query_text": "",
        "keywords": [],
        "disease": ""
    }
}
```

### 3.3 边界情况处理

在提示词中增加边界情况处理说明：

```
## 问题加工边界情况处理

1. **未提及疾病**：如果用户发言和上下文中没有涉及任何疾病（如问候、闲聊），`disease` 字段返回空字符串 ""
2. **问题不明确**：如果用户问题模糊不清，基于上下文尽可能推断，如果完全无法推断，保持原问题表达
3. **上下文矛盾**：如果历史聊天与当前发言矛盾，优先以当前发言为准
4. **无历史上下文**：如果历史聊天记录为空，仅基于当前发言进行分析
5. **记录失败时**：当 `record_success == false` 时，`query_text`、`keywords`、`disease` 可以设置为空值（空字符串或空数组）
```

## 五、注意事项

1. **执行时机**：问题加工逻辑仅在 `record_success == true` 时执行，确保不会在信息不完整或记录失败时执行不必要的加工
2. **字段一致性**：新增的三个字段必须与 `10-before-rag.md` 中的字段定义保持一致
3. **向后兼容**：确保改造后的提示词能够正确处理原有的记录逻辑，不影响现有功能
4. **格式要求**：所有新增字段的格式要求必须明确，避免 LLM 输出格式错误
5. **优先级规则**：疾病名提取的优先级规则必须与 `10-before-rag.md` 保持一致

## 六、验证要点

改造完成后，需要验证：
1. ✅ 记录成功时，是否正确返回 `query_text`、`keywords`、`disease` 三个字段
2. ✅ 记录失败时，这三个字段是否可以为空或不存在
3. ✅ 问题优化是否符合要求（精炼、标准、适合检索）
4. ✅ 关键词提取是否准确（1-3个，独立词或短语）
5. ✅ 疾病名提取是否符合优先级规则
6. ✅ 输出 JSON 格式是否正确，可以被系统正确解析
