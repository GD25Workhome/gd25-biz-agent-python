# optimization-agent 融合提示词改造思路（V6.3）

## 文档说明

**文档版本**：V1.0  
**创建时间**：2025-01-29  
**目标**：将 v6_1 / v6_2 的 `10-optimization-agent.md` 融合到 v6_3，采用「先意图识别 → 再通用信息提取 → 按意图选用专有逻辑」的三步结构，并明确输出多出的两个属性。

---

## 一、改造目标

- **来源**：`medical_agent_v6_1`、`medical_agent_v6_2` 的 `10-optimization-agent.md` 融合进 `medical_agent_v6_3/prompts/10-optimization-agent.md`。
- **流程角色**：本节点输出用于（1）下游 RAG 检索（scene_summary / optimization_question），（2）**路由到 core_agent**（intent + confidence），（3）元数据与文章检索（input_tags、disease_for_article）。
- **鲁棒性**：意图优先、通用提取与专有逻辑分离、同一目的下血压/QA 逻辑表述统一，并约定歧义与低置信度时的默认行为。

---

## 二、输出结构确认：比之前多两个属性

在原有 4 个字段基础上，**增加 2 个属性**（intent、confidence），与 v4 意图识别 agent 对齐（你引用的 91-92 行）。**不再包含 response_tags**（该属性为错误设计，已从提示词中移除）。

| 原有字段 | 说明 |
|----------|------|
| scene_summary | 场景摘要（1～3 句），用于向量检索 |
| optimization_question | 优化后问题，用于向量检索 |
| input_tags | 用户侧结构化标签数组 |
| disease_for_article | 疾病标签数组，用于科普文章检索 |

| 新增字段 | 说明 | 取值约定 |
|----------|------|----------|
| **intent** | 路由意图，用于 flow 条件边 | `"blood_pressure"`（明确记录血压）\| `"qa"`（其余均归此类） |
| **confidence** | 意图置信度 | 0.0～1.0 的浮点数，供路由或兜底逻辑使用 |

因此**输出 JSON 共 6 个属性**（4 个原有 + intent + confidence）。  
可选：若需可观测与排查，可再加 `reasoning`（意图判断理由），不参与路由时可不在 flow 条件中使用。

## 三、提示词逻辑三步（执行顺序）

整体顺序固定为以下三步，便于模型按顺序执行、减少遗漏与冲突：

1. **第一步：意图识别**  
   - 仅做一件事：根据当前用户输入与上下文，判断本条应走「血压记录专项」还是「QA」。
   - **准则（当前约定）**：**若用户发言中可以总结出「明确的记录血压」，则 intent = blood_pressure；否则均为 qa。**
   - **血压记录（blood_pressure）**：用户明确在「记录血压」——例如报数值、说「记录一下血压」「今天血压 120/80」等，以记录动作为主。
   - **QA（qa）**：除此以外均为 qa，包括：血压查询、血压/用药/疾病咨询、问候、其他健康问答等。
   - 输出：先在心里或中间步骤确定 `intent` 与 `confidence`，再进入第二步；无需复杂歧义规则，默认即「非明确记录血压 → qa」。

2. **第二步：通用信息提取（与意图无关）**  
   - 在**不依赖意图**的前提下，尽可能提取：
     - **scene_summary**：用 1～3 句自然语言概括「谁、在做什么、关键健康信息、是否有预警」；不在此步区分血压专有术语，仅做客观概括。
     - **optimization_question**：将用户最新一句改写为完整、清晰、可检索的问题句。
     - **input_tags**：用户侧标签（疾病类型、场景类型、其他特征等），先使用**通用枚举**（如疾病、场景类型、是否首次/连续/数据缺失等），不在此步引入血压专有评价术语。
     - **disease_for_article**：从对话中识别到的疾病名称数组，无则 `[]`。
   - 目的：保证无论最终 intent 是 blood_pressure 还是 qa，RAG 与下游都能拿到一致的、可用的基础字段。

3. **第三步：在已知意图下，选用专有逻辑（统一目的、鲁棒表述）**  
   - **目的统一**：两种专有逻辑的目的都是——在已有通用摘要与问题的基础上，**按场景补充或修正描述与标签**，使 scene_summary / optimization_question / input_tags 更贴合该意图下的下游需求（血压点评 vs 问答检索）。
   - **当 intent = blood_pressure**：  
     - 应用 v6_1 的**血压与心率规则**（三层：预警 → 常规评价 → 趋势/模式），对 scene_summary、input_tags 做**补充或改写**：  
       - scene_summary 中若涉及血压/脉搏数值，必须使用标准术语（如达标、轻度/中度/重度偏高、波动较大、需预警等）；  
       - input_tags 中可增加血压评价、心率状态、症状、预警标志等**标准化标签**（与 v6_1 现有标签集一致）。  
     - optimization_question 若为记录/查询血压，保持「明确动作对象」的表述。
   - **当 intent = qa**：  
     - 不引入血压评价与预警规则；  
     - 保持 scene_summary 为「客观场景 + 用户问题类型」即可；  
     - input_tags 使用通用标签（疾病、咨询类型等），不出现血压专有评价术语。  
   - **鲁棒性**：  
     - 若某一步信息不足（如缺少数值、缺少病史），则**仅填写能确定的部分**，不臆造；  
     - 专有逻辑只「在已有信息上做增强」，不改变第二步已确定的「客观事实」部分，避免与通用提取冲突。

这样，**先意图、再通用、再专有**，逻辑清晰；且专有逻辑仅作为「在已知意图下的增强」，两者目的统一，便于写提示词时用同一套话术框架（“在已有摘要/问题上，按当前意图做如下补充/修正……”）。

## 四、提示词文档结构建议（章节顺序）

建议的 md 结构，便于你后续直接落稿：

1. **角色定义**  
   - 一句话：本节点负责「意图识别 + 场景与问题提取 + 按意图的专有增强」，为 RAG 与下游 core_agent 路由提供统一输入。

2. **第一步：意图识别**  
   - **准则（当前约定）**：**若用户发言中可以总结出「明确的记录血压」，则 intent = blood_pressure；否则均为 qa。**  
   - 明确 `intent` 取值：仅 `blood_pressure` / `qa` 两种，用于 flow 条件边路由。  
   - 定义 **blood_pressure**：用户明确在「记录血压」——如报血压数值、说「记录血压」「今天血压 120/80」等，以记录动作为主。  
   - 定义 **qa**：除此以外均为 qa（含血压查询、血压/用药/疾病咨询、问候、其他健康问答等）。  
   - **confidence**：0.0～1.0 的浮点数，表示对上述意图判定的置信度；可选用于后续路由或兜底（如低置信度时仍按 intent 路由，或统一走 qa）。

3. **第二步：通用信息提取**  
   - 依次说明 scene_summary、optimization_question、input_tags（通用部分）、disease_for_article 的提取规则（**不包含 response_tags**）。  
   - 强调：本步**不区分意图**，使用通用术语与标签集；标签用标准枚举，不在此引入血压专有评价词。

4. **第三步：按意图的专有逻辑**  
   - **当 intent = blood_pressure**：  
     - 引用或内嵌 v6_1 的「血压与心率判断规则」（预警层 → 常规评价层 → 趋势/模式层）、预警触发规则；  
     - 说明如何用这些规则**补充/修正** scene_summary 与 input_tags（标准术语、预警标志等）；  
     - 不改变「客观事实」，只做术语与标签的标准化与增强。  
   - **当 intent = qa**：  
     - 说明：保持第二步的客观概括即可，不引入血压规则；  
     - 可选：补充 QA 场景下 input_tags 的推荐枚举（如咨询类型、科室等），与 v6_2 现有风格一致。  
   - 统一表述：两种分支的「目的」都是「在已有通用提取基础上，按意图做标准化与增强」，避免模型觉得两套逻辑目的不一致。

5. **边界处理**  
   - 无疾病信息、闲聊/问候、信息不足、上下文矛盾时的处理（与现有一致：不臆造、不添加未提及医学术语、以当前发言为准等）。  
   - 上述情况均归为 **qa** 意图；scene_summary / optimization_question 的默认写法（如「日常非医疗交互」、原句保留等）与现有一致。

6. **输出格式**  
   - 严格 JSON，列出全部 6 个字段（4 个原有 + intent + confidence），并注明类型与取值约束；**不包含 response_tags**。  
   - 可选：若保留 `reasoning`，写在说明中并注明「仅用于排查，不参与路由」。

## 五、鲁棒性设计要点

- **意图优先**：先定 intent，再做专有逻辑，避免先写摘要再反推意图导致的不一致。  
- **通用与专有分离**：第二步只做通用提取，第三步只在「已知意图」下做增强，减少两类逻辑互相覆盖。  
- **统一目的表述**：在提示词中显式写「血压专有逻辑与 QA 逻辑的目的相同：在通用提取基础上做标准化与增强」，便于模型统一理解。  
- **意图默认**：仅当「明确记录血压」时为 blood_pressure，其余均为 qa，无需复杂歧义规则。  
- **标签术语**：input_tags 使用固定枚举，血压分支再增加血压/心率/预警等标准术语，避免自由发挥。  
- **信息不足**：任一步信息不足时，只填能确定项，缺失项用空字符串或空数组，并在边界处理中写清。

---

## 六、与 flow 的衔接

- **条件边**：下游 flow 使用 `intent == 'blood_pressure'` 与 `intent == 'qa'`（及可选 `confidence >= 0.5` 等）做路由时，依赖本节点输出的 `intent`、`confidence` 写入 `state.edges_var`。  
- **rag_node 保留 intent**：按《012902-医疗流程v6_1与v6_2融合方案分析》结论，rag_node 写回 `edges_var` 时需**保留**本节点写入的 `intent`（及 `confidence`），否则条件边无法拿到意图。

---

## 七、小结

- **输出**：原有 4 个字段（不含 response_tags）+ **intent** + **confidence**（共 6 个属性）；与 v4 的 91-92 行一致，多出的就是 intent 与 confidence。  
- **意图准则**：若用户发言可总结为「明确的记录血压」→ blood_pressure；否则均为 qa。  
- **逻辑顺序**：意图识别 → 通用信息提取 → 按意图选用血压专有逻辑或 QA 逻辑（目的统一、专有逻辑只做增强）。  
- **文档结构**：角色 → 意图准则（明确记录血压 / 否则 qa）→ 通用提取 → 血压专有规则 / QA 逻辑 → 边界处理 → 输出格式。  
- **鲁棒性**：意图优先、通用/专有分离、统一目的表述、歧义与低置信度默认、标签术语标准化。
