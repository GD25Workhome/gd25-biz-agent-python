# QA 场景独立记录转知识库脚本 - 技术设计文档

## 1. 目标与背景

将 `cursor_docs/012802-QA场景独立记录/` 下以 `01-` 开头的场景独立记录 md 文档，通过「create_rag_agent」提示词调用大模型，得到结构化 `cases` 后写入 `knowledge_base` 表，用于后续向量检索与标注。

- **输入**：上述目录下指定的 md 文件（每个文件一条「场景独立记录」正文）。
- **输出**：每个 md 经模型输出一个 `{ "cases": [ ... ] }`，按 case 写入 `knowledge_base` 表对应字段。
- **约束**：脚本仅新建技术设计文档与脚本相关代码，不修改业务 flow 或模型定义；运行范围通过**代码内配置**指定（全部或部分 md），不使用启动参数切换范围。

## 2. 执行流程概览

```
获取待处理文件列表 → 并行：单文件处理(读文件→填提示词→调模型→解析 JSON→落库) → 汇总统计
```

1. **获取待处理文件路径列表**  
   扫描 `cursor_docs/012802-QA场景独立记录/`，筛选以 `01-`、`02-`、… 开头的 `.md` 文件（排除 `00-索引-xxx.md`）。  
   支持在代码中配置「只跑部分」：例如白名单 `["01-诊疗-场景1-疾病诊断与费用等.md", "05-药物-场景3-漏服错服过量服.md"]`，或「全部」不设白名单。

2. **单文件处理（可并行）**  
   对每个 md 文件：
   - 读取文件全文，作为 `scene_record_content`。
   - 读取提示词模板（见下），将 `{scene_record_content}` 替换为当前文件内容。
   - 调用大模型，得到一段文本。
   - 从文本中解析出 **仅包含 `cases` 的 JSON**（格式见 §4）。
   - 对 `cases` 中每个元素，映射到 `knowledge_base` 表字段并写入（见 §5）。

3. **并行策略**  
   参考 `scripts/embedding_import/run_embedding_import_parallel.py`：  
   使用 `asyncio.Semaphore` 控制最大并发数，`asyncio.gather` 并行执行多个「单文件处理」任务，避免一次全开造成限流或资源占满。

4. **模型与提示词**  
   - 提示词模板：使用 `scripts/create_rag_data/create_rag_agent.md`（与 `config/flows/rag_source_agent/prompts/create_rag_agent.md` 内容一致，脚本读独立副本即可）。  
   - 占位符：模板中 **`{scene_record_content}`**（约第 14 行）替换为当前 md 的完整正文。  
   - 模型：**采用当前项目框架的配置方式**——脚本通过 LangGraph 跑一个「单节点」流程，该流程仅包含一个 agent 节点；节点使用的模型、temperature、thinking 等均在 flow 的 yaml 中配置（与 `config/flows/medical_agent_v6_1/flow.yaml` 中 optimization_agent 的写法一致），换模型时只改 flow 配置或 `model_providers.yaml`，无需改脚本常量。

5. **解析与存储**  
   - 解析：模型输出可能带 markdown 代码块或前后说明，需从文本中抽取 **纯 JSON**（只含 `cases` 数组），再 `json.loads`。  
   - 存储：按 §5 映射到 `knowledge_base` 表字段；若一条 md 对应多个 case，则写入多条记录。

## 3. 目录与代码结构

- **脚本根目录**：`scripts/create_rag_data/`  
  - 主入口：`run_create_rag_data.py`（单文件，逻辑尽量简洁：加载配置 → 取文件列表 → 并行跑任务 → 打日志/统计）。  
  - 子包：`scripts/create_rag_data/core/`，放具体实现，便于维护和单测。

建议的 `core` 子模块划分：

| 模块 | 职责 |
|------|------|
| `config.py` | 常量：流程 key（单节点 flow 名）、提示词路径、场景记录目录、并发数、待处理文件白名单（空=全部）等；**不**在此配置 provider/model，由 flow 决定。 |
| `file_loader.py` | 扫描场景记录目录，返回待处理的 md 绝对路径列表（支持白名单过滤）；可提供「相对路径（项目根）」计算。 |
| `prompt_builder.py` | 可选：若 flow 内提示词占位符由 state 注入，则负责把 `scene_record_content` 放入 state 的 `edges_var["edges_prompt_vars"]`；若脚本自填模板再传入，则与现描述一致。 |
| `flow_runner.py` | 封装「加载单节点 LangGraph 流程 → 组装 state（含 scene_record_content）→ ainvoke → 从返回 state 中取出模型回复文本」。流程仅一个 agent 节点，配置参考 §6。 |
| `parser.py` | 从模型输出文本中抽取 JSON 并解析为 `List[Dict]`（即 `cases`）；处理代码块、前后噪音等。 |
| `repository.py` | 将单条 case（Dict）+ 当前 md 相对路径、原文，转为 `KnowledgeBaseRecord` 字段（含 `source_meta`、`raw_material_full_text`）并插入；封装异步 session 与事务。 |

主流程伪代码（在 `run_create_rag_data.py` 中）：

```text
async def main():
    加载 ProviderManager（model_providers.yaml）
    graph = FlowManager.get_flow(CREATE_RAG_AGENT_FLOW_KEY)  # 单节点 flow
    文件列表 = file_loader.get_md_paths()
    sem = asyncio.Semaphore(max_concurrent)
    tasks = [process_one_file(path, graph, sem, ...) for path in 文件列表]
    results = await asyncio.gather(*tasks, return_exceptions=True)
    统计成功/失败，写日志
```

单文件处理 `process_one_file` 内：读文件 → 组装 state（edges_prompt_vars.scene_record_content = 文件内容）→ flow_runner 执行 graph.ainvoke → 从 state 取模型回复文本 → parser 解析 → repository 按 case 写入（含 source_meta.source_file 相对路径、raw_material_full_text 原文）。

## 4. 模型输出格式与解析

- **约定格式**（与 `create_rag_agent.md` 中 124–135 行一致）：

```json
{
  "cases": [
    {
      "scene_summary": "string",
      "optimization_question": "string 或 [\"q1\", \"q2\", ...]",
      "reply_example_or_rule": "string",
      "scene_category": "string",
      "input_tags": ["tag1", "tag2", ...],
      "response_tags": ["tag1", "tag2", ...]
    }
  ]
}
```

- **解析要求**：  
  - 从模型返回的整段文本中定位 JSON（可先尝试去掉 ```json ... ``` 包裹，再取第一个 `{` 到匹配的 `}`）。  
  - 只解析根对象的 `cases` 数组；若根对象还含其它 key，可忽略。  
  - 解析失败时记录该文件路径与原始输出，便于排查；该文件视为失败，不写入 DB。

## 5. 表字段映射（knowledge_base）

写入时仅填充与 create_rag_agent 输出对应的字段（对应 `backend/infrastructure/database/models/knowledge_base.py` 约 25–59 行），以及来源与原文字段，其余字段可为空或默认值。

| 模型输出字段 | 表字段 | 说明 |
|-------------|--------|------|
| `scene_summary` | `scene_summary` | 直接字符串。 |
| `optimization_question` | `optimization_question` | 若为数组则序列化为字符串存储（如 `json.dumps`），否则直接存字符串。 |
| `reply_example_or_rule` | `reply_example_or_rule` | 直接字符串。 |
| `scene_category` | `scene_category` | 直接字符串。 |
| `input_tags` | `input_tags` | JSONB，保持数组。 |
| `response_tags` | `response_tags` | JSONB，保持数组。 |
| （无对应） | `raw_material_full_text` | **采用建议方案**：写入当前 md 的**完整原文**，便于追溯。 |
| （无对应） | `source_meta`（见下） | JSON 对象，存 `{"source_file": "相对路径"}`，相对路径以**项目根目录**为基准（如 `cursor_docs/012802-QA场景独立记录/01-诊疗-场景1-疾病诊断与费用等.md`）。 |

- **来源字段 `source_meta`**：在 `knowledge_base` 模型中、于 `technical_tag_classification` 附近（约 80–81 行）**新增一列**，类型 JSONB，用于存储来源信息；对象中 key 为 `source_file`，value 为上述相对路径。列名建议 `source_meta`，comment 如「来源信息（含 source_file 相对路径）」。脚本写入时构造 `{"source_file": "<相对路径>"}`。
- 主键 `id`：由表默认生成（如 ULID）。  
- `created_at` / `updated_at`：按表定义自动处理。  
- 其它 `raw_material_*`（除 `raw_material_full_text`）、`technical_tag_classification`、`business_tag_classification` 等本脚本可不填。

## 6. 配置方式（代码内）

- **待处理文件**  
  - 全部：白名单配置为空或 None，表示扫描目录下所有 `NN-*.md`（N 为数字）。  
  - 部分：在 `config.py` 中配置白名单列表，如 `INCLUDE_FILES = ["01-诊疗-场景1-疾病诊断与费用等.md", "16-危重症-01-胸痛.md"]`，只处理这些文件名。

- **模型（采用项目框架的 flow 配置）**  
  - 脚本**不**在代码里写死 provider/model，而是**直接使用一个 LangGraph 流程**：该流程仅包含**一个 agent 节点**，节点配置参考 `config/flows/medical_agent_v6_1/flow.yaml` 中的 `optimization_agent`（约 15–23 行），例如：
    - `type: agent`
    - `prompt`: 指向 create_rag_agent 的提示词（如 `prompts/create_rag_agent.md`，占位符 `{scene_record_content}` 由 state 的 `edges_prompt_vars` 注入）
    - `model`: `provider: doubao`，`temperature: 0.7`，`thinking: type: disabled`
  - 流程可放在现有 `config/flows/rag_source_agent/` 下（单独 yaml 或复用），或新建如 `config/flows/create_rag_agent/flow.yaml`；脚本通过 `FlowManager.get_flow(流程名)` 加载，换模型时只改该 flow 的 yaml 或 `model_providers.yaml`。
  - **单节点 flow 结构示意**：`name` / `entry_node` 指向该 agent 节点；`edges` 仅一条「该节点 → END，condition: always」；脚本在 ainvoke 前将 `scene_record_content` 放入 `state["edges_var"]["edges_prompt_vars"]`，ainvoke 后从 state 中取该节点输出或消息内容作为模型回复文本。

- **并发与路径**  
  - `MAX_CONCURRENT`：同时处理文件数。  
  - `CREATE_RAG_AGENT_FLOW_KEY`：单节点 flow 的 name/key，用于 `FlowManager.get_flow`。  
  - `PROMPT_TEMPLATE_PATH`：提示词 md 路径（默认 `scripts/create_rag_data/create_rag_agent.md`，若 flow 内已配置则可为只读/文档用）。  
  - `SCENE_RECORDS_DIR`：场景记录目录（默认 `cursor_docs/012802-QA场景独立记录`），均相对于项目根；写入 `source_meta.source_file` 时使用**相对项目根的路径**。

## 7. 错误与日志

- 单文件失败：捕获异常，打日志（文件路径 + 错误信息），该文件不写入 DB，计入失败数。  
- 解析失败：记录原始模型输出到日志（可截断长度），便于排查。  
- 汇总：结束后输出成功数、失败数、总耗时；若有失败可 `sys.exit(1)` 便于 CI/手工检查。

## 8. 依赖与运行环境

- 使用项目现有 conda 环境与依赖；如需读配置（如 `model_providers.yaml`），在入口处与其它脚本一致先加载 `ProviderManager`。  
- 数据库：使用项目已有 async session 与 `KnowledgeBaseRepository`/`KnowledgeBaseRecord`，保证与现有表结构一致。

## 9. 小结

- 输入：`cursor_docs/012802-QA场景独立记录/` 下以数字前缀开头的 md。  
- 提示词：`scripts/create_rag_data/create_rag_agent.md`，占位符 `{scene_record_content}` 填当前文件内容（通过 state 的 `edges_prompt_vars` 注入）。  
- 模型：**采用项目框架**——通过单节点 LangGraph 流程调用 agent，节点配置参考 medical_agent_v6_1 的 optimization_agent（provider、temperature、thinking 等在 flow yaml 中配置）。  
- 输出：解析 `cases` JSON，按 case 写入 `knowledge_base` 表（含 `scene_summary`、`optimization_question`、`reply_example_or_rule`、`scene_category`、`input_tags`、`response_tags`，以及 `raw_material_full_text` 存 md 完整原文、`source_meta` 存 `{"source_file": "相对路径"}`）。  
- 并行：asyncio + Semaphore，参考 `run_embedding_import_parallel.py`。  
- 范围：在代码内配置（白名单或全部）；模型与流程：由 flow yaml 与 ProviderManager 决定，无需在脚本中写死。

以上为脚本设计与实现所需的技术约定，实现时按此文档在 `scripts/create_rag_data/` 下组织主入口与 `core` 子模块即可。
