# 批次任务大批量创建：分批提交方案简述

当单次创建任务量极大（如 10 万条 task）时，若仍在一个事务内插入并一次 commit，会带来：长事务、大 undo、连接占用久、锁与内存压力大等问题。本文简述几种「分批提交」的改造思路，便于选型。

---

## 当前行为回顾

- **CreateTemplate._build_and_persist_batch**：先插入 1 条 `batch_job`，再循环插入全部 `batch_task`，**不**在模版内 commit。
- **API 层**：`create_batch()` 返回后执行一次 `session.commit()`。
- 因此：10 万条 task = 1 个事务、1 次提交。

---

## 方案一：模版内分批 flush，API 层仍一次 commit

**思路**：在 `_build_and_persist_batch` 里每插入 N 条 task（如 2000）调用一次 `session.flush()`，不提交。请求结束时仍由 API 做一次 `session.commit()`。

**改动点**：仅模版内部，在 for 循环里按批 `flush()`。

**效果**：  
- 减轻单次 flush 时的内存与锁压力，避免一次性生成 10 万条 INSERT 再一起 flush。  
- 事务边界不变，仍是「一个大事务、一次提交」，持锁/连接时间仍较长。

**适用**：希望改动小、先缓解内存/锁峰值，暂不拆事务时可用。

---

## 方案二：模版内先提交 job，再按批提交 task（同一 session）

**思路**：  
1. 先创建并插入 `batch_job`，**立即**对该 session 做一次 `commit()`（或使用独立小事务），以拿到 `job.id` 并持久化。  
2. 将 `task_items` 按固定大小分块，每块内循环 `task_repo.create(...)`，每块结束后 `session.commit()`，再继续下一块。

**改动点**：模版需要「先 commit job」再循环；循环内按块 commit；API 层不再对整次请求做一次总 commit（或仅做兜底）。

**效果**：  
- 多个小事务：1 个事务只含 1 个 job；后续每个事务只含一批 task（如 2000 条）。  
- 单次事务体积和持锁时间明显下降；同一请求内仍复用同一 session。

**注意**：若 job 与第一批 task 希望强一致（要么都有要么都没有），需在业务上权衡是否接受「job 已提交、部分 task 未提交」的中间状态；通常可接受，因 job 仅表示「已创建、总数为 10 万」，具体 task 可异步补全（见方案四）。

---

## 方案三：Repository 层提供 bulk 插入 + 模版内分批调用

**思路**：  
- 在 TaskRepository（或专门方法）提供 `bulk_create(job_id, task_items_chunk)`，内部用 `session.add_all()` 或 `bulk_insert_mappings` 等批量插入，减少 ORM 与 round-trip。  
- 模版内：先创建 job 并提交（或 flush）拿到 `job.id`，再将 `task_items` 分块，每块调用 `bulk_create`；每块后 `flush()` 或 `commit()`（视是否要拆事务，与方案一/二组合）。

**效果**：  
- 在「按批提交」的基础上，进一步降低每批的插入成本和 DB 往返次数，适合超大批量。

---

## 方案四：API 只创建 job，task 由后台异步分批写入

**思路**：  
- API 只创建并提交一条 `batch_job`（如 `total_count=100000`，`status=created` 等），**不**在本请求中插入任何 `batch_task`，立即返回 202 + `job_id` / `batch_code`。  
- 后台 worker 或定时任务：根据 `job_id` 或「待补全的 job」列表，按业务规则（如从 `query_params` 或源表）分批生成 `task_items`，每批插入 N 条并 `commit()`，直到达到 `total_count` 或业务结束。

**效果**：  
- 请求事务极短，只含 1 条 job 插入；10 万条 task 在多个小事务、多批中完成，不阻塞 API。  
- 需要：后台任务、可能的状态字段（如 job.status）、幂等与进度追踪设计。

**适用**：对 API 响应时间要求高、允许「job 先存在、task 逐步补全」的场景。

---

## 方案五：模版内每批使用独立 session/事务

**思路**：  
- 模版（或上层）能拿到 `session_factory`。先在当前请求的 session 上创建并提交 `batch_job`，拿到 `job.id`。  
- 将 `task_items` 分块，**每块**用 `session_factory()` 新建 session，插入该块所有 task 后 `commit()` 并关闭 session，再处理下一块。

**效果**：  
- 每批是独立事务，与主请求事务隔离；主请求也可在「只创建 job」后立即返回，或同步等所有块完成再返回（视产品需求）。  
- 需要依赖注入或传入 `session_factory`，模版或 Service 的构造会略复杂。

---

## 对比小结

| 方案 | 事务/提交方式 | 改动范围 | 典型用途 |
|------|----------------|----------|----------|
| 一   | 一大事务，分批 flush、一次 commit | 仅模版 | 缓解内存/锁峰值，不改事务边界 |
| 二   | 1 个 job 事务 + 多批 task 事务（同 session） | 模版 + 可能 API | 真正缩短每笔事务体积与持锁时间 |
| 三   | 同方案一或二，但用 bulk 插入 | repo + 模版 | 大批量时降低插入成本 |
| 四   | API 只 1 个 job 事务；task 后台多批多事务 | API + 后台任务 + 状态 | 请求快速返回，适合异步补全 |
| 五   | job 一事务；每批 task 独立 session/事务 | 模版/Service + 注入 | 与请求事务解耦，可同步可异步 |

实际选型可组合：例如「方案二 + 方案三」在模版内按批 bulk 插入并每批 commit；或「方案四 + 方案三」在后台用 bulk + 分批 commit 写入 task。
