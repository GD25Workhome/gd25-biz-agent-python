# 背景

## 业务背景
- **目标**：将线上Agent的标准聊天案例整理成few-shot模板，存入向量库
- **应用场景**：在`retrieval_node`节点根据上一个节点（`before_rag`）的特征提取，召回匹配度高的案例
- **预期效果**：提高`after_record_agent`节点的点评准确度

## 技术架构现状
- **向量库**：PostgreSQL + pgvector
- **Embedding模型**：moka-ai/m3e-base（768维）
- **当前embedding策略**：`user_input + agent_response` 的组合
- **检索流程**：
  1. `before_rag`节点：优化问题表达，提取`query_text`、`keywords`、`disease`
  2. `retrieval_node_v2`：使用`query_text + keywords`进行向量检索
  3. `after_record_agent`：接收检索到的案例，生成点评回复

## 核心难点
1. **难点一**：如何构建embedding内容，让召回更准确？
   - 是否要包含疾病、血压趋势、症状等结构化信息？
   - 这些信息是否会提升检索效果？

2. **难点二**：如何提取这些特征信息？
   - 人工标注 vs 自动提取
   - 提取哪些特征最有效？

# 需求

## 核心需求
- 设计能够将案例转为向量库的流程，将测试同学整理的案例或其它来源的优秀案例吸收到我们的rag中
- 设计embedding的方案，能够提取关键特征

## 具体需求

- 评估不同embedding策略的效果

### 2. 特征提取方案
- 设计自动特征提取流程，提取关键特征信息
- 确定需要提取的特征维度（疾病类型、血压评价、症状、预警等）
- 设计半自动提取+人工校验的流程

### 3. 数据导入流程
- 设计从Excel文件到向量库的完整导入流程
- 包括数据预处理、特征提取、embedding生成、批量导入等步骤
- 建立数据质量验证机制

### 4. 检索优化
- 优化检索逻辑，利用提取的特征信息提升召回效果
- 调整相似度阈值和Top-K参数
- 建立检索效果评估机制

# 设计
## 核心设计原则
- 向量检索的本质是语义相似度的匹配，不是精确的字段匹配

## embedding方案说明

### 推荐方案：结构化信息增强（⭐⭐⭐⭐⭐）

**核心原理**：将关键特征转换为自然语言描述，融入embedding文本，保持语义连贯性。

**构建embedding文本的标准格式**：
```
场景描述：{场景摘要（2-3句话）}
用户输入：{user_input}
医生回复：{agent_response}
关键特征：疾病={disease}，血压评价={bp_level}，症状={symptoms}，预警={alert}，趋势={trend}
上下文：{历史数据、用药等信息}
```

**优点**：
- ✅ 保留了语义信息，向量模型能理解这些特征
- ✅ 特征信息自然融入，不会破坏语义连贯性
- ✅ 检索时能匹配到相似场景的案例
- ✅ 格式清晰，便于后续维护和优化

**示例1：首次记录，血压轻度偏高**
```
场景描述：患者首次记录血压数据，血压值轻度偏高，无异常症状，医生给予正向反馈和监测建议。
用户输入：我今天的血压，高压129，低压70，脉搏72
医生回复：你能主动监测并告诉我血压情况，这个习惯非常棒！已经帮你记录了这次的血压数据。你的血压值129/70mmHg略高于正常范围...
关键特征：疾病=高血压，血压评价=轻度偏高，症状=无，预警=否，趋势=无
上下文：首次记录，无历史数据，用药=未提及
```

**示例2：重度偏高，有症状，需要预警**
```
场景描述：患者记录血压值重度偏高，伴有头晕症状，医生触发预警，建议尽快就医。
用户输入：今天量血压，高压180，低压110，有点头晕
医生回复：您本次测量的血压过高，且伴有不适症状，建议您尽快到医院就诊...
关键特征：疾病=高血压，血压评价=重度偏高，症状=头晕，预警=是，趋势=未分析
上下文：有症状，用药=未提及
```

### 备选方案：分层Embedding（⭐⭐⭐⭐）

**原理**：为不同类型的特征建立不同的embedding字段

**方案**：
- **主embedding**：`user_input + agent_response`（用于语义检索）
- **辅助metadata**：疾病、血压级别等存储在`tags`字段（用于后过滤）

**优点**：
- ✅ 检索灵活，可以先用语义检索，再用metadata过滤
- ✅ 可扩展性强，后续可以添加更多metadata

**缺点**：
- ⚠️ 需要修改检索逻辑，支持metadata过滤
- ⚠️ 实现复杂度更高

### 不建议的策略

#### ❌ 纯结构化字段embedding

**不建议原因**：
- 向量模型更适合处理自然语言文本，不适合结构化数据
- 如："高血压|轻度偏高|无症状" 这样的字段组合，语义信息丢失严重
- 检索时难以匹配到相似语义的场景

## 关键特征提取

### 特征提取方案：半自动提取 + 人工校验

**推荐方案**：使用LLM进行批量自动提取，然后人工校验关键字段，补充质量等级等主观评价。

#### 第一阶段：自动提取（使用LLM）
- 利用`embedding_agent`流程中的`stem_extraction_node`节点进行批量特征提取
- 提取内容涵盖输入特征和回复特征两个维度

#### 第二阶段：人工校验和标注
- 人工校验自动提取的准确性
- 补充难以自动提取的特征：
  - 回复质量等级（优秀/良好/一般）
  - 场景类型（首次记录/常规记录/异常记录等）
  - 特殊标签（特殊情况、特殊话术等）

### 关键特征列举

#### 输入的特征提取（自动提取、维度尽可能的全）

**必需字段（自动提取）**：
- **疾病类型**：高血压、冠心病、糖尿病、高脂血症等
- **当前血压评价级别**：
  - 达标、轻度偏高、中度偏高、重度偏高、低压偏低、高压偏低
  - 计算规则：
    - 达标：90<=收缩压<=收缩压目标值 且 舒张压<=舒张压目标值
    - 轻度偏高：收缩压、舒张压任一值超过对应设定目标值，但两者超过均不到10mmHg
    - 中度偏高：收缩压、舒张压任一值超过对应设定目标值+10mmHg，但未超过180/110
    - 重度偏高：收缩压>=180 or 舒张压>=110
    - 低压偏低：20mmHg<舒张压<=60mmHg 且 舒张压<=舒张压目标值
    - 高压偏低：40mmHg<=收缩压<90mmHg 且 收缩压<=收缩压目标值
- **当前心率情况**：
  - 脉搏较慢(BPM≤40)、脉搏偏慢(40<BPM<60)、脉搏正常(60≤BPM≤80)、脉搏偏快(80<BPM≤150)、脉搏较快(BPM>150)
- **两次血压对比**：
  - 波动较大（相比7天内上一次血压SBP升高/降低40mmHg及以上或DBP升高/降低20mmHg）、波动不大
- **血压趋势**：
  - 下降、上升、无显著变化、晨间高、夜间高、无突出模式
  - 需要满足点评条件：14天中数据天数≥6天，或最近7天数据≥3天
- **症状列表**：头晕、胸闷、胸痛、恶心、腹胀、呼吸急促、视力模糊、咳嗽、肢体麻木、夜尿增多、血尿、呕吐、烦躁不安、肌肉酸痛、肌肉不适、肌肉无力、憋喘等（数组）
- **用药情况**：
  - 正在服用药品：苯磺酸氨氯地平片等
  - 是否规律：不规律服药、规律服药
- **生活方式**：
  - 少喝酒、心情放松、睡眠良好、运动、少吃盐、不抽烟
- **是否需要预警**：是/否
  - 预警触发条件：
    - 血压评价为"重度偏高"或"高压偏低"或"低压偏低"
    - 脉搏较慢(BPM≤40)或脉搏较快(BPM>150)
    - 波动较大且带症状
    - 历史首次反馈特定症状且短期内未缓解（头昏头晕、胸闷、胸痛、呼吸急促、恶心腹胀、视力模糊、咳嗽、肢体麻木、夜尿增多、血尿）
    - 高血压患者血压不达标且带症状（头昏头晕、胸闷、胸痛、呼吸急促、恶心腹胀、视力模糊、呕吐、烦躁不安）
    - 高脂血症患者服用他汀带症状（肌肉酸痛、肌肉不适、肌肉无力）
    - 冠心病患者血压不达标且带症状（胸闷、胸痛、憋喘）
- **场景类型**：根据用户意图判断
  - record（记录数据）：用户想要记录各类健康数据（血压、症状、用药、健康事件等）
  - query（查询数据）：用户想要查询已记录的健康数据
  - qa（健康问答）：用户提出健康相关的咨询问题
  - greeting（打招呼）：用户进行问候、致谢、结束语等日常互动
- **患者发言风格**：简洁、详细、口语化、正式、情绪化等
- **其他重要特征**：首次记录、连续记录、异常值、数据缺失等

**推荐字段（自动提取+人工校验）**：
- **回复风格**：鼓励型、警示型、中性、引导型
- **是否包含趋势分析**：是/否
- **是否包含排班推送**：是/否
- **是否包含文章推送**：是/否
- **质量等级**：优秀、良好、一般（需要人工评估）

**可选字段（人工标注）**：
- **特殊标签**：特殊情况、特殊话术、最佳实践等


## 关键特征的提取的提示词设计

### 提示词设计概述

特征提取使用`embedding_agent`流程中的`stem_extraction_node`节点，该节点使用`prompts/20-ext_agent.md`提示词文件。

### 提示词核心功能

#### 角色定义
- 专业的问题加工助手
- 任务：分析患者的最新发言，结合历史聊天上下文，将患者的问题优化改写为适合向量库检索的标准表达，并提取核心关键词和疾病名

#### 核心任务
1. **问题优化**：将患者的最新发言优化改写为适合向量库检索的标准表达，结合历史聊天记录和上下文信息，使问题更规范、更完整
2. **输入关键词提取**：从用户信息、最新发言、历史信息中提取关键特征词，涵盖疾病、血压、症状、用药、生活方式、预警、场景等多个维度
3. **回复关键词提取**：从AI回复内容中提取关键特征词，包括回复风格、是否包含趋势分析、排班推送、文章推送等维度

#### 输入上下文信息
- 用户信息：{user_info}
- 最新发言：{current_message}
- 历史聊天记录：{history_messages}
- 回复内容：{ai_response}
- 人工标记的标签（用作keywords提取的参考）：{manual_ext}

#### 输出格式要求

所有回复必须严格按照以下JSON格式返回：

```json
{
    "optimization_question": "优化后的问题（字符串，必填，不能为空）",
    "input_keywords": ["关键词1", "关键词2", "关键词3"],
    "ai_response_keywords": ["关键词1", "关键词2", "关键词3"]
}
```

**字段说明**：
- `optimization_question`：优化后的问题文本，需要结合最新发言、历史聊天记录以及上下文中的其它信息进行提取，使其更规范、更适合向量检索
- `input_keywords`：从用户信息、最新发言、历史信息中提取的关键词数组，涵盖疾病、血压、症状、用药、生活方式、预警、场景等多个维度
- `ai_response_keywords`：从AI回复内容中提取的关键词数组，包括回复风格、是否包含趋势分析、排班推送、文章推送等维度

### 提取规则说明

#### input_keywords 提取原则
1. **全面性**：尽可能覆盖所有可提取的维度，但不要强行提取不存在的信息
2. **准确性**：关键词要准确反映实际情况，避免错误提取
3. **简洁性**：关键词应该是简洁的词或短句，避免过长的描述
4. **扩展性**：除了上述维度，还可以提取其他重要的特征关键词，如：首次记录、数据异常、时间信息等

#### ai_response_keywords 提取原则
1. **风格识别**：根据回复的语气、用词判断回复风格
   - 鼓励型：包含表扬、肯定、鼓励的词汇
   - 警示型：包含警告、提醒、紧急的词汇
   - 中性：客观陈述，无明显情感倾向
   - 引导型：包含建议、指导、引导的词汇
2. **内容识别**：识别回复中是否包含特定类型的内容
   - 趋势分析：包含血压趋势、达标率、变化趋势等分析内容
   - 排班推送：包含医生排班、门诊时间等信息
   - 文章推送：包含科普文章、健康文章等推荐内容
3. **其他特征**：提取回复中的其他重要特征，如回复结构、重点内容等

### 边界情况处理
1. **未提及疾病**：如果患者发言和上下文中没有涉及任何疾病（如问候、闲聊），`disease` 字段返回空字符串 ""
2. **问题不明确**：如果患者问题模糊不清，基于上下文尽可能推断，如果完全无法推断，保持原问题表达
3. **上下文矛盾**：如果历史聊天与当前发言矛盾，优先以当前发言为准
4. **无历史上下文**：如果历史聊天记录为空，仅基于当前发言进行分析

### 提示词使用流程

在`embedding_agent`流程中：
1. **stem_extraction_node**：使用该提示词进行特征提取，输出包含`optimization_question`、`input_keywords`、`ai_response_keywords`
2. **format_data_node**：将提取的特征格式化为embedding文本，保存到`embedding_str`
3. **embedding_node**：对`embedding_str`进行向量化，生成`embedding_value`
4. **insert_data_to_vector_db_node**：将向量数据导入向量库

# 实施建议

## 数据导入流程

### 步骤1：数据预处理
1. 从Excel文件读取案例数据
2. 清洗和验证数据（检查必需字段）
3. 使用LLM批量提取特征（调用`embedding_agent`流程）

### 步骤2：特征构建
1. 根据提取的特征，构建embedding文本（使用标准格式）
2. 将特征信息存储到`tags`字段（便于后续过滤和统计）

### 步骤3：向量化和导入
1. 使用moka-ai/m3e-base模型生成embedding
2. 批量导入到`record_examples`表
3. 验证导入结果

## 检索优化建议

### 当前检索逻辑
```python
# retrieval_node_v2.py
enhanced_query = f"{query_text} {' '.join(keywords)}"
query_embedding = embedding_cache.text_to_embedding(enhanced_query)
```

### 优化建议
1. **利用疾病信息**：在`before_rag`节点已经提取了`disease`，可以考虑：
   - 在检索前先用`disease`过滤相关表（如果表中有疾病标签）
   - 或者在embedding查询中加入疾病信息：`enhanced_query = f"{query_text} {disease} {' '.join(keywords)}"`

2. **相似度阈值调优**：
   - 当前阈值：0.7 → 0.6 → 0.5（降级策略）
   - 建议：根据实际召回效果调整，可能需要提高到0.75或降低到0.65

3. **Top-K调整**：
   - 当前：top_k=15
   - 建议：根据下游节点的需求调整，few-shot通常3-5个示例即可

## 技术风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| **特征提取不准确** | 导致embedding偏差，检索效果差 | 人工校验关键案例，建立校验流程 |
| **embedding文本过长** | 可能影响模型效果 | 控制文本长度，重点突出关键特征 |
| **案例质量不一致** | few-shot效果差 | 建立质量评估机制，只导入高质量案例 |
| **检索结果不相关** | 下游节点收到无关案例 | 调整相似度阈值，优化embedding策略 |

## 验证方案

1. **小规模验证**
   - 先导入50-100个案例
   - 用实际查询测试召回效果
   - 评估下游节点生成的回复质量

2. **A/B测试**
   - 对比有无few-shot案例的效果
   - 收集用户反馈，评估改进效果

3. **持续优化**
   - 监控检索日志，分析召回案例的相关性
   - 根据实际效果调整embedding策略和特征提取
