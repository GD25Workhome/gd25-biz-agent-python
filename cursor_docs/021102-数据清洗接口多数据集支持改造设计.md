# 数据清洗执行接口多数据集支持改造设计

## 1. 改造背景与目标

### 1.1 现状
- 当前 `POST /data-cleaning/datasets/{dataset_id}/items/rewritten/execute` 接口的 `dataset_id` 为路径参数，仅支持**单个**数据集。
- Step01 独立数据项页面（`step01_data_items_standalone.js`）已支持多选数据集查询，但「进行数据清洗」仅对第一个选中数据集生效（`effectiveDatasetId`）。

### 1.2 目标
- 将 `dataset_id` 从路径参数迁移至请求体 `RewrittenExecuteRequest`，并改为 `dataset_ids: List[str]`，支持**多个**数据集。
- 同步更新后端服务层、仓储层及前端调用逻辑。

## 3. API 变更详情

### 3.1 路由与路径

| 变更前 | 变更后 |
|--------|--------|
| `POST /data-cleaning/datasets/{dataset_id}/items/rewritten/execute` | `POST /data-cleaning/items/rewritten/execute` |

### 3.2 请求体 RewrittenExecuteRequest

| 字段 | 变更前 | 变更后 |
|------|--------|--------|
| `dataset_ids` | 无（路径参数） | **必填** `List[str]`，至少 1 个 |
| `item_ids` | 可选 | 保持不变 |
| `query_params` | 可选 | 保持不变 |

**约束**：
- `dataset_ids` 非空且至少 1 个
- 至少提供 `item_ids` 或 `query_params`（与现有一致）
- 若提供 `item_ids`，则不能为空列表

### 3.3 create_params（批次表 JSONB）

| 变更前 | 变更后 |
|--------|--------|
| `{ "dataset_id": "xxx", "mode": "item_ids" | "query_params", ... }` | `{ "dataset_ids": ["id1","id2"], "mode": "item_ids" | "query_params", ... }` |

## 4. 后端实现要点

### 4.1 多数据集下的 item_ids 模式
- 调用 `get_by_ids_multi(dataset_ids, item_ids)`：仅返回 `id in item_ids` 且 `dataset_id in dataset_ids` 的记录，避免跨数据集误操作。

### 4.2 多数据集下的 query_params 模式
- 调用 `get_list_by_conditions_multi(dataset_ids, query_params)`：在多个数据集范围内按条件筛选，无分页，返回所有命中记录。

### 4.3 create_init_batch 改造
- 入参移除 `dataset_id`，每条记录的 `source_dataset_id` 使用 `record.dataset_id`。
- 存在性检查：对所有 `(record.dataset_id, record.id)` 组合查询已存在的 init/processing/success 记录，跳过已存在的项。

## 5. 前端调用示例

### 5.1 Step01 独立页 - 批量清洗（多选数据集）
```javascript
const res = await axios.post(`${API_PREFIX}/items/rewritten/execute`, {
    dataset_ids: selectedDatasetIds.value,  // 多选的数据集 ID 列表
    query_params: queryParams
});
```

### 5.2 Step01 独立页 - 单条清洗
```javascript
const res = await axios.post(`${API_PREFIX}/items/rewritten/execute`, {
    dataset_ids: [row.dataset_id],
    item_ids: [row.id]
});
```

### 5.3 数据集 Tab 页 - 批量/单条
```javascript
// 批量
await axios.post(`${API_PREFIX}/items/rewritten/execute`, {
    dataset_ids: [props.datasetId],
    query_params: queryParams
});
// 单条
await axios.post(`${API_PREFIX}/items/rewritten/execute`, {
    dataset_ids: [props.datasetId],
    item_ids: [row.id]
});
```

## 6. 联动检查项（无代码改动）

| 检查项 | 说明 |
|--------|------|
| Worker 循环 | `rewritten_worker_loop` 拉取 `init` 记录并执行，每条记录已有 `source_dataset_id`，无需改动 |
| `run_one_rewritten` | 按 `source_dataset_id`、`source_item_id` 查原始项，逻辑不变 |
| RewrittenBatch 表 | `create_params` 已有 JSONB 设计，可存储 `dataset_ids`，模型无需改 |
| 其他路由 | `/datasets/{dataset_id}/items` 等 CRUD 仍为单数据集，不受影响 |

---

## 7. 测试建议

1. **单数据集**：`dataset_ids: ["id1"]`，与改造前行为一致。
2. **多数据集 query_params**：选中 2+ 数据集，执行批量清洗，验证跨数据集数据均被创建任务。
3. **多数据集 item_ids**：不同数据集下的 item，验证仅命中指定数据集内的 item。
4. **数据集不存在**：传入不存在的 `dataset_ids`，应返回 404 或明确错误提示。
5. **空 dataset_ids**：应返回 400。

---

## 8. 改造顺序建议

1. Schema：`RewrittenExecuteRequest` 新增 `dataset_ids`
2. 仓储：`get_by_ids_multi`、`get_list_by_conditions_multi`，以及 `create_init_batch` 改造
3. 服务层：`create_rewritten_batch` 与 `_build_create_params` 改造
4. API 路由：路径与参数调整
5. 前端：两个 JS 文件的请求体与 URL 更新
6. 设计文档：两篇 md 同步更新
