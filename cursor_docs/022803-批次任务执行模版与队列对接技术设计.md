## 1. 目标与约束

1. **执行侧模版化**：模版类提供「运行方法」的骨架（检查状态 → 乐观锁占位 → 调用业务实现 → 回写结果），实现类只实现具体业务（如 embedding）。
2. **与现有队列框架对接**：执行入口可对接到与 `rewritten_service.run_one_rewritten` + `rewritten_queue_service` 类似的「入队 → 多消费者拉取 → 单条执行」模式。
3. **按 job_type 路由执行器**：运行时根据 `batch_job.job_type` 找到对应业务执行器，与创建侧 `BatchJobCreateService` 按 `job_type` 找创建 Handler 的设计一致。

## 2. 与 step3 运行时设计的兼容性

step3 约定（摘要）：

- 核心入参（step3 原文）：`BatchTaskRecord`；本方案**执行入口**统一为队列元素 `BatchTaskQueueItem`，模版内部再加载 `BatchTaskRecord` 并用于乐观锁与业务执行。
- 运行时：  
  1. 检查 task 状态是否为 pending，用**乐观锁**将状态更新为 running；更新失败则打 error 日志并跳出。  
  2. 调用业务实现方法执行；成功则任务成功，否则失败，并将结果写回 `BatchTaskRecord`。

与 rewritten 框架对比：

| 维度           | Rewritten 方案                         | 本方案（batch_task 执行）                    |
|----------------|----------------------------------------|---------------------------------------------|
| 单条执行入口   | `run_one_rewritten(rec)`                | `run_one_batch_task(session, item: BatchTaskQueueItem)`（见下） |
| 乐观锁发生位置 | 消费者内：`update_status_to_processing_if_init(record_id)` 再执行 | 模版内：`update_status_to_running_if_pending(item.task_id)` 再执行业务 |
| 业务执行       | `run_one_rewritten(rec)` 内查库、调流程 | 模版内用 item.task_id 加载 task，再调用子类 `execute_task_impl(session, task_record)` |
| 结果回写       | 在 rewritten 的 repo/flow 内           | 模版内：成功则写 execution_result + execution_return_key + status=success；失败则写 execution_error_message + status=failed |

**结论**：step3 的「先乐观锁再执行业务、结果写回 BatchTaskRecord」与当前 rewritten 的「占位 + 单条执行」在**流程上兼容**。差异仅在于：  
- 乐观锁在 rewritten 里由**消费者**做；  
- 在本方案里由**执行模版**做（消费者只负责取队列元素、做 task_version 校验、按 `item.job_type` 取执行器、调 `run_one_batch_task(session, item)`）。  
这样仍满足「单条执行函数只做一件事、可被多消费者复用」的框架要求，兼容性成立。

## 3. 整体架构

### 3.1 分层关系

```
API 层（生产者）
  POST /batch-jobs/{job_id}/run  →  入参为 BatchJobRecord.id；将本批次下 pending 的 batch_task 入队

队列层（内存队列，参考 rewritten 的队列重新独立实现）
  元素：BatchTaskQueueItem（job_type、task_id、task_version）
  消费者：取元素 item → 用 item.task_id 查 task → 若 task.version != item.task_version 则舍弃；否则按 item.job_type 取执行器 → 调 run_one_batch_task(session, item)

领域层
  - 执行模版（ExecuteTemplate）：run_one_task(session, item) = 加载 task → 乐观锁 → execute_task_impl(session, task_record) → 成功则回写标准结果到 execution_result/execution_return_key/status，失败则回写 execution_error_message/status
  - 执行器实现（如 PipelineEmbeddingExecutor）：实现 execute_task_impl(session, task_record)；业务只关心 task 内容，入口统一为 item
  - 执行器注册表：job_type → 执行器工厂；由 BatchTaskExecuteService.run_one_batch_task(session, item) 按 item.job_type 解析执行器并调用 run_one_task(session, item)
```

### 3.2 与创建侧对称

| 阶段   | 创建侧                     | 执行侧（本方案）                     |
|--------|----------------------------|--------------------------------------|
| 模版   | `CreateTemplate`           | `ExecuteTemplate`（见下）            |
| 实现类 | `PipelineEmbeddingCreateHandler` | `PipelineEmbeddingExecutor`          |
| 注册   | `BatchJobCreateService._registry: job_type → CreateHandler 工厂` | `BatchTaskExecuteService._registry: job_type → ExecuteExecutor 工厂` |
| 入口   | `create_batch(job_type, query_params)` | `run_one_batch_task(session, item: BatchTaskQueueItem)`（按 `item.job_type` 解析执行器） |

## 4. 执行模版与执行器设计

### 4.1 执行模版（ExecuteTemplate）

- **职责**：定义「单条任务」执行流程，不关心具体业务；业务由子类实现。**任务执行的对象统一为队列元素 `BatchTaskQueueItem`**，与队列层衔接一致。
- **核心方法**：`run_one_task(session, item: BatchTaskQueueItem) -> None`  
  - 入参：当前 `AsyncSession`、**BatchTaskQueueItem**（含 `job_type`、`task_id`、`task_version`）；执行层不再依赖「查 job 取 job_type」，由队列元素直接携带。
- **模版步骤**（与 step3 一致）：
  1. **加载 task**：用 `item.task_id` 查询 `BatchTaskRecord`；可选再次校验 `task.version == item.task_version`，不一致则打日志并 return。
  2. **乐观锁**：调用 `BatchTaskRepository.update_status_to_running_if_pending(item.task_id)`；若返回 False，打 error 日志并 return，不执行业务。
  3. **业务执行**：调用子类实现的 `execute_task_impl(session, task_record)`，得到标准结果对象（见 4.2）；无异常即视为成功。
  4. **结果回写**（与 `batch_task` 表字段一一对应）：  
     - **成功**：将业务返回的**标准结果对象**序列化（如 JSON）写入 `batch_task.execution_result`（对应表字段 49）；将其中的 **execution_return_key** 写入 `batch_task.execution_return_key`（对应表字段 59）；将 `batch_task.status`（对应表字段 34）更新为 `"success"`。  
     - **失败**：捕获异常，将异常信息（含堆栈）写入 `batch_task.execution_error_message`（对应表字段 54），将 `batch_task.status` 更新为 `"failed"`。

说明：乐观锁由**仓储**提供（见 5.1），模版只调用接口；**整条链路从消费者到模版入参均为 `BatchTaskQueueItem`**，避免上下设计不一致。

### 4.2 执行结果标准对象与抽象业务方法

- **执行结果标准对象（BatchTaskExecutionResult）**  
  - 定义统一的结果类型，供模版回写到 `batch_task`：  
    - **必填字段**：`execution_return_key: str`，回执/标识 key，模版会将其写入 `batch_task.execution_return_key`（表字段 59）。  
    - **其它字段**：由业务自定（如 embedding 记录 id、条数等），与 `execution_return_key` 一起序列化后写入 `batch_task.execution_result`（表字段 49，Text，如存 JSON 字符串）。  
  - 实现时可为 dataclass、TypedDict 或带 `execution_return_key` 的基类 + 业务子类，保证模版能统一取到 `execution_return_key` 并写入对应列即可。

- **execute_task_impl(self, session: AsyncSession, task_record: BatchTaskRecord) -> BatchTaskExecutionResult**  
  - 子类实现：只做「当前任务」的业务逻辑（如：查 rewritten 记录 → 拼 embedding 串 → 调 embedding 模型 → 写 `pipeline_embedding_records`）。  
  - **返回值**：成功时返回标准结果对象（至少含 `execution_return_key`，其它字段业务自定）；模版负责将该对象写入 `execution_result`、`execution_return_key` 并更新 `status`。  
  - 不在此方法内改 task 状态；状态与结果由模版统一更新。失败时抛出异常，由模版捕获并写 `execution_error_message`、更新 `status="failed"`。

### 4.3 Pipeline Embedding 执行器（示例）

- 类名：`PipelineEmbeddingExecutor(ExecuteTemplate)`（或仅实现「执行器接口」并由模版组合，见 4.4）。
- **execute_task_impl** 逻辑（与 step3 一致）：
  1. 从 `task_record.runtime_params` 取 `pipeline_data_items_rewritten_id`、`embedding_type`（Q/QA）。
  2. 根据 `pipeline_data_items_rewritten_id` 查询改写记录。
  3. 按 `embedding_type` 拼接用于 embedding 的字符串（可参考 `BeforeEmbeddingFuncNode._format_embedding_str`，区分 Q 与 QA）。
  4. 调用 embedding 模型得到向量。
  5. 写入一条 `pipeline_embedding_records`。
- **返回**：成功时返回 `BatchTaskExecutionResult`（至少含 `execution_return_key`，如新写入的 `pipeline_embedding_record.id` 或业务约定的回执 key；其它字段如 `embedding_record_id` 等可自定并写入 `execution_result`）。异常抛出，由模版捕获并写 `execution_error_message`、更新 `status="failed"`。

### 4.4 执行器与模版的实现方式

采用**继承模版**：执行器类继承 `ExecuteTemplate`，实现 `execute_task_impl(session, task_record) -> BatchTaskExecutionResult`；`run_one_task(session, item: BatchTaskQueueItem)` 由基类实现（内部按 item 加载 task、乐观锁、调 execute_task_impl、按标准结果回写 execution_result / execution_return_key / status 或 execution_error_message / status）。与创建侧 `PipelineEmbeddingCreateHandler(CreateTemplate)` 风格一致。

## 5. 基础设施与扩展点

### 5.1 BatchTaskRepository 扩展

- **update_status_to_running_if_pending(task_id: str) -> bool**  
  - 条件更新：`WHERE id = :task_id AND status = 'pending'`，`SET status = 'running'`。  
  - 返回是否更新到行（rowcount > 0），用于乐观锁判断。  
  - 与 `DataItemsRewrittenRepository.update_status_to_processing_if_init` 语义一致；batch_task 有 version 时仍可仅用 status 条件简化实现，必要时再加强为 version 乐观锁。

### 5.2 按 job_type 解析执行器（与 batch_job_service 对称）

- **BatchTaskExecuteService**（或命名为 `BatchJobRunService`）：
  - 持有 `_registry: Dict[str, ExecutorFactory]`，例如 `job_type -> Callable[[], ExecuteTemplate]` 或 `job_type -> Callable[[AsyncSession], ExecuteTemplate]`。
  - 提供 **run_one_batch_task(session, item: BatchTaskQueueItem) -> None**：
    1. 从 `_registry[item.job_type]` 得到执行器实例（或工厂创建）；**不再用 task 查 job 取 job_type**，与队列元素结构一致。
    2. 调用执行器的 `run_one_task(session, item)`。
  - 未注册的 `item.job_type` 可抛 `UnknownJobTypeError`，与创建侧一致。

这样，**运行时根据队列元素的 `job_type` 找到对应业务类**，与 `BatchJobCreateService` 的 `create_batch(job_type, ...)` 设计对称；入参与队列层统一为 `BatchTaskQueueItem`。

## 6. 与 rewritten 队列框架的对接方式

### 6.1 生产者（API）

- 接口形态可类比 `POST /rewritten-batches/run`：  
  **POST /batch-jobs/{job_id}/run**  
  - 入参：`job_id` 为 **BatchJobRecord.id**（路径参数），不是 `job_code`。  
  - 逻辑：用 `job_id` 查 `batch_job`，将本批次下 `status = 'pending'` 的 `batch_task` 入队；入队时记录每条 task 的 `version`（见 6.2）。  
  - 返回：本次入队数量等。

### 6.2 队列与消费者

- **队列元素（新对象）**  
  - 定义专用结构（如 `BatchTaskQueueItem` 或 dataclass），包含：  
    - **job_type**：来自 `batch_job.job_type`，供消费者直接路由执行器，无需再为取 job_type 查 job（若已做 version 校验则已查 task）。  
    - **task_id**：`batch_task.id`。  
    - **task_version**：入队时 `batch_task.version` 的快照，作为乐观锁依据。  
  - 消费时：用 `item.task_id` 查 `BatchTaskRecord`；**若当前 `task.version != item.task_version`，说明 task 已被更新（如已被其他消费者处理、重跑等），该队列元素应舍弃，不执行**；若一致，按 `item.job_type` 取执行器，调用 `run_one_batch_task(session, item)`。**不再为取 job_type 查 job**。

- **方案 A：独立队列**  
  - 新建 `batch_task_queue_service`：队列元素为 `BatchTaskQueueItem`（job_type、task_id、task_version）。  
  - 消费者循环：`get() 得到 item → 用 item.task_id 查 BatchTaskRecord → 若 task.version != item.task_version 则舍弃；否则按 item.job_type 取执行器 → run_one_batch_task(session, item)`。  
  - `run_one_batch_task(session, item)` 内部按 `item.job_type` 取执行器并调用 `run_one_task(session, item)`；模版内再完成 task 级乐观锁 + execute_task_impl + 结果回写。

### 6.3 单条执行入口与 rewritten 的对应关系

- **rewritten**：消费者取 `record_id` → 查 `DataItemsRewrittenRecord` → `update_status_to_processing_if_init(record_id)` → `run_one_rewritten(rec)`。  
- **本方案**：消费者取队列元素 **item: BatchTaskQueueItem** → 用 `item.task_id` 查 `BatchTaskRecord` → **若 task.version != item.task_version 则舍弃** → 按 `item.job_type` 取执行器 → **run_one_batch_task(session, item)** → 执行器 **run_one_task(session, item)**；模版内部再「用 item.task_id 加载 task → update_status_to_running_if_pending → execute_task_impl 返回 BatchTaskExecutionResult → 成功则写 execution_result/execution_return_key/status，失败则写 execution_error_message/status」。  

因此，**step3 的运行时方法设计（先乐观锁再执行业务、结果写回 BatchTaskRecord）与当前 rewritten 的「占位 + 单条执行」框架兼容**；且**从队列到执行层入参统一为 BatchTaskQueueItem**，只需保证：  
- 消费者只负责「取 item → task_version 校验 → 按 item.job_type 取执行器 → 调 `run_one_batch_task(session, item)`」；  
- 乐观锁与结果回写集中在模版内，避免重复占位或漏写结果。

## 7. 实现清单（建议顺序）

1. **BatchTaskQueueItem**：定义队列/执行统一入参对象，含 `job_type`、`task_id`、`task_version`。
2. **BatchTaskExecutionResult**：定义执行结果标准对象，必含 `execution_return_key`；其它字段业务自定；模版将序列化结果写入 `batch_task.execution_result`，`execution_return_key` 写入 `batch_task.execution_return_key`，成功/失败写 `batch_task.status` 与 `execution_error_message`。
3. **BatchTaskRepository**：新增 `update_status_to_running_if_pending(task_id) -> bool`；以及按需提供 `update_execution_result(task_id, status, execution_result=..., execution_return_key=..., execution_error_message=...)` 供模版回写。
4. **ExecuteTemplate**：抽象类，定义 `run_one_task(session, item: BatchTaskQueueItem)`（内部加载 task、乐观锁、调 `execute_task_impl`、按 BatchTaskExecutionResult 或异常回写到 execution_result/execution_return_key/status/execution_error_message）与抽象方法 `execute_task_impl(session, task_record) -> BatchTaskExecutionResult`。
5. **PipelineEmbeddingExecutor**：继承 `ExecuteTemplate`，实现 `execute_task_impl(session, task_record) -> BatchTaskExecutionResult`（查 rewritten → 拼串 → 调 embedding → 写 `pipeline_embedding_records`，返回含 execution_return_key 的标准结果）。
6. **BatchTaskExecuteService**：注册表 `job_type -> 执行器工厂`，提供 `run_one_batch_task(session, item: BatchTaskQueueItem)`（按 item.job_type 取执行器并调用 `run_one_task(session, item)`）；与 `BatchJobCreateService` 对称。
7. **队列与 API**：实现「按 job_id（BatchJobRecord.id）将本批次 pending task 入队」的接口；入队时构造 `BatchTaskQueueItem`；消费者做 task_version 校验后调用 `BatchTaskExecuteService.run_one_batch_task(session, item)`。
8. **main.py / lifespan**：若采用独立消费者，启动 batch_task 消费者协程（可参考 rewritten 的 `start_consumers`）。

## 8. 小结

- **执行对象统一**：从队列元素到执行入口均使用 **BatchTaskQueueItem**（job_type、task_id、task_version），消费者与执行层不再传递 `BatchTaskRecord` 或查 job 取 job_type，避免上下设计割裂。  
- **模版/实现分离**：执行模版 `run_one_task(session, item)` 提供「加载 task → 乐观锁 + 调用业务 + 按标准结果回写」的骨架；业务返回 `BatchTaskExecutionResult`（必含 `execution_return_key`），模版写入 `execution_result`（49）、`execution_return_key`（59）、`status`（34），失败时写入 `execution_error_message`（54）、`status`（34）；实现类只实现 `execute_task_impl(session, task_record) -> BatchTaskExecutionResult`，与现有 `CreateTemplate` / `PipelineEmbeddingCreateHandler` 一致。  
- **与 step3 兼容**：先检查 pending、再乐观锁更新为 running、再执行业务并写回结果，完全符合 step3 第 27～28 行及 30 行的描述。  
- **与 rewritten 框架兼容**：通过单一入口 `run_one_batch_task(session, item)` 对接「生产者入队 + 消费者拉取 + 单条执行」模式；乐观锁在模版内完成，不破坏现有队列语义。  
- **按 job_type 路由**：通过队列元素的 `item.job_type` 在 `BatchTaskExecuteService` 的注册表中解析执行器，与 `BatchJobCreateService` 的 `create_batch(job_type, ...)` 设计一致。
