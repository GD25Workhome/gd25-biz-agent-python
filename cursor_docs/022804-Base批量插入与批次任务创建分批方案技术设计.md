# Base 批量插入与批次任务创建分批方案技术设计

## 1. 背景与问题

- **现状**：`BaseRepository.create()` 每次插入后都会 `await self.session.flush()`，保证单条创建后能拿到自增/ULID 等数据库生成字段。
- **问题**：在 `CreateTemplate._build_and_persist_batch()` 中，对 `task_items` 循环调用 `_task_repo.create()`。当 task 数量达到 10 万级时，会产生 **10 万次连续 flush**，带来：
  - 大量网络往返与协议开销；
  - PG 侧重复的解析/规划/执行路径、WAL 与锁竞争、I/O 碎片化。
- **目标**：在 Base 层提供批量插入能力（按 chunk 分组、每组一次 flush），业务层对「整批 list」按大小分批（如每批 1 万条）调用 Base，从而将 flush 次数从 N 降为「分批次数 × 每批内 chunk 数」，显著减少与数据库的交互。

## 2. 设计目标

- Base 层：提供 **传入业务对象列表 → 底层补充必要字段 → 按固定条数（默认 1000）一组创建并每组 flush** 的批量插入方法。
- 业务层（batch_template）：**先构建完整 list**，再按「每批最多 1 万条」拆分，每批调用 Base 的批量方法；超过 1 万条时循环调用，避免单次 list 过大导致内存与事务体积问题。

## 3. 分层职责

| 层级 | 职责 | 关键参数/行为 |
|------|------|-------------------------------|
| **Base（或 AuditBase）** | 接收「单表」业务数据列表；补全审计/默认字段；按 chunk 插入并 flush | chunk_size 默认 1000；每 chunk 一次 flush |
| **Task Repository** | 提供 `create_many(job_id, items, chunk_size=1000)`，将业务 DTO 转成 kwargs 列表并调用 Base | 固定 job_id、status 等，items 为 TaskPreCreateItem 或等价结构 |
| **CreateTemplate** | 先得到 task_items list；按 max_batch_size=10000 分批；每批调用 task_repo.create_many | 单批 ≤ 1 万条；多批顺序调用，同一事务内 |

## 4. Base 层批量插入设计

### 4.1 方法签名与语义

- **建议方法名**：`create_many`（与单条 `create` 对应）。
- **入参**：
  - `items: List[Dict[str, Any]]`：每项为「可传入 model 构造的键值对」，由具体 Repository 从业务 DTO 转成。
  - `chunk_size: int = 1000`：每多少条为一组，每组执行一次 `add_all` + `flush()`。
- **行为**：
  - 若 `items` 为空，直接返回，不 flush。
  - 按 `chunk_size` 将 items 切分为多组；对每组：用 `self.model(**kwargs)` 构造实例并 `session.add()`，然后 `await self.session.flush()`。
  - **不 commit**，由上层（如 CreateTemplate 所在事务）统一 commit。
- **返回值**：可选。若不需要批量插入后的主键列表，可返回 `None` 或插入条数；若需要，可在每 chunk flush 后收集 id（PG 支持 RETURNING，但需结合实现）。当前设计优先「只写不读」，返回插入条数即可。

### 4.2 审计字段（AuditBaseRepository）

- 若 Base 的 `create_many` 在 `AuditBaseRepository` 中实现（或由子类调用 Base 的通用实现并包一层），则对每条 item 在传入 `self.model()` 前补全：
  - `id`（未传则 `generate_ulid()`）
  - `version=0`、`is_deleted=False`、`create_time=_now()`
- 与现有单条 `create()` 的审计逻辑保持一致。

### 4.3 与现有 create() 的关系

- 单条 `create()` 保持现状：单条 insert + flush，适合需要立即拿到 `instance.id` 的场景。
- `create_many()` 仅用于「批量写入、无需逐条拿 id」的场景，避免在 Base 内混用两套 flush 策略导致理解成本。

## 5. Task Repository 层

- 新增方法：`create_many(self, job_id: str, items: List[TaskPreCreateItem], chunk_size: int = 1000) -> int`。
- 将每个 `TaskPreCreateItem` 转成 `create()` 所用的 kwargs（含 `job_id`、`status="pending"` 等），得到 `List[Dict]`，再调用 Base 的 `create_many(items=dict_list, chunk_size=chunk_size)`。
- 返回插入条数（或 Base 的返回值），便于日志/监控。

## 6. CreateTemplate 业务层改造

### 6.1 当前逻辑（简述）

- `_build_and_persist_batch()`：先 `_job_repo.create()` 得到 `job`（需要 flush 以拿到 `job.id`），再循环 `task_items` 逐条 `_task_repo.create()`。

### 6.2 改造后逻辑

- **Step 1**：创建 job（不变）。  
  `job = await self._job_repo.create(...)`，保证 `job.id` 可用。
- **Step 2**：将 `task_items` 按「每批最多 1 万条」切分。  
  常量如 `TASK_LIST_BATCH_SIZE = 10_000`，`batches = [task_items[i:i+TASK_LIST_BATCH_SIZE] for i in range(0, len(task_items), TASK_LIST_BATCH_SIZE)]`。
- **Step 3**：对每一批调用 `await self._task_repo.create_many(job_id=job.id, items=batch, chunk_size=1000)`。  
  Base 内部会再按 1000 条一组 flush，即每 1 万条最多 10 次 flush，而不是 1 万次。

### 6.3 边界

- `task_items` 为空：已在模版入口 `create_batch()` 中 `raise ValueError`，无需在 `_build_and_persist_batch` 内重复处理。
- 单批 0 条：若切分后某批为空（理论上不会，因按固定 size 切分），`create_many` 内对空 list 直接返回即可。

## 7. 数据流小结

```
CreateTemplate.create_batch()
  → query_tasks_for_pre_create() 得到 task_items（如 10 万条）
  → _build_and_persist_batch(task_items)
       → _job_repo.create()  [1 次 flush]
       → 按 1 万条切分：batch1(1~10000), batch2(10001~20000), ...
       → for batch in batches:
            _task_repo.create_many(job_id, batch, chunk_size=1000)
              → Base.create_many(items=dict_list, chunk_size=1000)
                   每 1000 条：add_all → flush  [每批 1 万条 = 10 次 flush]
  → 同一 session commit（由上层统一）
```

- **10 万条示例**：10 批 × 每批 10 次 flush = 100 次 flush（再加 job 的 1 次），相对 10 万次 flush 大幅减少。

## 8. 风险与注意事项（供你检查漏洞）

### 8.1 事务与一致性

- 所有 `create_many` 与 job 的 `create` 应在 **同一 AsyncSession/事务** 内执行；若中间抛异常，整体可回滚。需确认 CreateTemplate 的调用方是否保证同一 session 贯穿 `_build_and_persist_batch` 整段。

### 8.2 内存

- 「先构建完整 list 再分批」意味着 10 万条 `TaskPreCreateItem` 会常驻内存；若未来规模到百万级，可考虑流式/分页从 `query_tasks_for_pre_create` 取数并按批写入，避免一次性拉全量。当前 1 万/批的 cap 主要控制单次调用 Base 的规模与事务内 flush 次数，不是内存上限。

### 8.3 chunk_size 与 1 万的关系

- **chunk_size=1000**：Base 层「每 1000 条 flush 一次」，控制单次 flush 的 SQL 量和 PG 压力。
- **max_batch_size=10000**：业务层「单次传给 create_many 的 list 最多 1 万条」，控制单次调用和单段事务内的数据量。
- 两者独立：例如可将 chunk_size 设为 2000，则每批 1 万条只会触发 5 次 flush；需根据 PG 与驱动表现做调优。

### 8.4 返回值与幂等

- 当前设计不依赖「批量插入后返回每条 task 的 id」。若后续需要（如落库后立刻发 MQ 带 task_id），可再在 Base 或 Repository 层为每 chunk 使用 RETURNING 收集 id 并返回；否则保持「只返回插入条数」即可。
- 幂等仍由业务层 `redundant_key` 等保证，与单条 create 一致。

### 8.5 空列表与异常

- Base：`create_many(items=[])` 不做任何 flush，直接返回。
- 若某 chunk 插入时 PG 报错（如唯一约束），异常会向上抛，整事务回滚；需在调用链上保留清晰日志（job_id、batch 序号、chunk 范围）便于排查。

### 8.6 测试建议

- 单条 create 与 create_many 混用同一 session 的兼容性（如先 create job，再 create_many tasks）。
- 边界：0 条、1 条、恰好 1000/10000 条、10001 条等，验证切分与 flush 次数符合预期。
- 大数量（如 10 万）的集成测试或压测，观察 PG 连接数、WAL、锁等待是否在预期内。

## 9. 小结

- **Base**：新增 `create_many(items, chunk_size=1000)`，底层按 chunk 补充字段并分组 flush；AuditBase 中对每条补 id/version/is_deleted/create_time。
- **Task Repository**：新增 `create_many(job_id, items, chunk_size=1000)`，将 TaskPreCreateItem 转成 dict 列表后调 Base。
- **CreateTemplate**：先构建完整 task_items list，按 max_batch_size=10000 分批，每批调用 `_task_repo.create_many`，从而在保证「job 先写、再写 tasks」的前提下，将 flush 次数从 N 降为约 N/1000（再乘以分批系数），并便于后续按 PG 表现调整 chunk_size 与 max_batch_size。

若你希望把 chunk_size / max_batch_size 做成配置项或从上层传入，可在接口上增加可选参数，文档中已预留该扩展点（调优处）。
