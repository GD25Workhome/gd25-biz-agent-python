# 知识库表与前端查询界面技术设计文档

## 1. 概述

本文档描述**知识库**业务的数据库表设计与前端查询界面设计，包括表结构、字段说明、查询能力（分页 + 左模糊查询）及接口约定。

---

## 2. 业务名称与范围

- **业务名称**：知识库
- **用途**：存储 RAG/向量话术场景下的场景摘要、优化问题、回复示例与规则、场景分类及各类标签，支持按多字段左模糊查询与分页展示。

---

## 3. 表结构设计

### 3.1 表名约定

- **技术表名**（建议）：`knowledge_base` 或与项目现有命名规范一致（如 `kb_entry`）。
- **业务名称**：知识库。

### 3.2 字段列表

| 序号 | 业务含义           | 技术字段名               | 类型           | 说明 |
|------|--------------------|--------------------------|----------------|------|
| 1    | 主键               | id                       | STR (VARCHAR)  | 与系统一致：ULID，String(50)，default=generate_ulid |
| 2    | 场景摘要           | scene_summary            | TEXT           | 对应 create_rag_agent 输出 scene_summary |
| 3    | 优化问题           | optimization_question    | TEXT           | 可为单字符串或数组 `["q1","q2",...]` 的序列化存储 |
| 4    | 回复示例或规则     | reply_example_or_rule    | TEXT           | 对应 reply_example_or_rule |
| 5    | 场景分类           | scene_category          | VARCHAR        | 对应 scene_category |
| 6    | 输入标签           | input_tags               | JSONB          | 数组 `["tag1","tag2",...]` |
| 7    | 回复标签           | response_tags            | JSONB          | 数组 `["tag1","tag2",...]` |
| 8    | 原始资料-全量文字   | raw_material_full_text   | TEXT           | 原始资料全文，文本 |
| 9    | 原始资料-场景简介   | raw_material_scene_summary | TEXT         | 原始资料中的场景简介，文本 |
| 10   | 原始资料-提问      | raw_material_question    | TEXT           | 原始资料中的提问，文本 |
| 11   | 原始资料-回答      | raw_material_answer      | TEXT           | 原始资料中的回答，文本 |
| 12   | 原始资料-其它      | raw_material_other       | TEXT           | 原始资料其它内容，文本 |
| 13   | 技术标记分类       | technical_tag_classification | JSONB      | 技术侧标记/分类，JSON 结构由业务约定 |
| 14   | 业务标记分类       | business_tag_classification  | JSONB      | 业务侧标记/分类，JSON 结构由业务约定 |
| 15   | 创建时间           | created_at               | TIMESTAMP      | 可选，便于排序与审计 |
| 16   | 更新时间           | updated_at               | TIMESTAMP      | 可选 |

### 3.3 与 create_rag_agent 输出的对应关系

create_rag_agent 输出（123–128 行）与表字段对应如下：

- `scene_summary` → `scene_summary`
- `optimization_question` → `optimization_question`（存字符串或 JSON 数组）
- `reply_example_or_rule` → `reply_example_or_rule`
- `scene_category` → `scene_category`
- `input_tags` → `input_tags`
- `response_tags` → `response_tags`

扩展字段：

- **技术标记分类**：`technical_tag_classification`（JSON）
- **业务标记分类**：`business_tag_classification`（JSON）

### 3.4 索引

- 暂不建除主键外的索引，数据量不大；后续若查询变慢再按需增加。

## 4. 查询能力设计

### 4.1 查询方式

- **分页**：与当前框架一致，使用 `limit`、`offset`（参见 `blood_pressure`、`users` 路由）；前端需总条数时由响应中的 `total` 提供。
- **左模糊查询（LEFT LIKE）**：对以下字段支持「左匹配」模糊查询（即 `LIKE 'keyword%'` 或前端等价输入）：
  - `scene_summary`（场景摘要）
  - `optimization_question`（优化问题）
  - `scene_category`（场景分类）
  - `input_tags`（输入标签：若为 JSONB 数组，可按数组元素左模糊或转成文本后左模糊，由实现决定）
  - `response_tags`（回复标签：同上）

说明：用户输入的「左 like」在语义上一般为「以关键词开头」；若产品希望「包含即匹配」，则可用 `LIKE '%keyword%'`，但无法利用普通 B-tree 索引，需权衡性能。

### 4.2 查询参数汇总

| 参数名               | 类型   | 说明 |
|----------------------|--------|------|
| scene_summary        | str, Optional | 场景摘要左模糊，Query(None, description="...") |
| optimization_question | str, Optional | 优化问题左模糊 |
| scene_category       | str, Optional | 场景分类左模糊 |
| input_tags           | str, Optional | 输入标签左模糊 |
| response_tags        | str, Optional | 回复标签左模糊 |
| limit                | int    | 限制数量，默认 100（与现有路由一致） |
| offset               | int    | 偏移量，默认 0 |

### 4.3 接口约定（对齐当前框架）

与现有 API 风格一致：FastAPI、路由挂载在 `/api/v1` 下，路径 kebab-case，列表用 Query 参数 + `limit`/`offset`，响应用 Pydantic 模型。

- **列表查询**：`GET /api/v1/knowledge-base`
  - **路由定义**：在 `backend.app.api.routes` 中新增模块（如 `knowledge_base.py`），在 `routes/__init__.py` 中 `include_router(knowledge_base_router, prefix="/api/v1", tags=["知识库"])`，故实际路径为 `/api/v1/knowledge-base`。
  - **Query 参数**：`scene_summary`、`optimization_question`、`scene_category`、`input_tags`、`response_tags`（均为 `Optional[str] = Query(None, ...)`），`limit: int = 100`，`offset: int = 0`。
  - **响应**：Pydantic 模型，例如 `KnowledgeBaseListResponse`，字段：`total: int`（总条数，便于前端分页）、`items: list[KnowledgeBaseRecordResponse]`（当前页数据）；单条用 `KnowledgeBaseRecordResponse`，与表字段一致，`Config.from_attributes = True`。
  - **风格参考**：`backend.app.api.routes.blood_pressure.list_blood_pressure`（Query + limit/offset）、`backend.app.api.schemas.users.UserListResponse`（列表包装为 `users` 字段）；本接口用 `items` + `total` 以支持分页展示。

## 5. 前端界面设计

### 5.1 页面功能

- **列表展示**：以表格或卡片展示知识库记录，支持分页（上一页/下一页或页码）。
- **查询区域**：提供与左模糊查询对应的输入项：
  - 场景摘要
  - 优化问题
  - 场景分类
  - 输入标签
  - 回复标签
- **交互**：输入条件后点击「查询」或「搜索」，列表刷新并回到第一页；分页切换时保持当前查询条件。

### 5.2 查询表单字段（与后端参数一致）

| 表单项标签     | 对应参数               | 说明 |
|----------------|------------------------|------|
| 场景摘要       | scene_summary          | 单行输入，左模糊 |
| 优化问题       | optimization_question | 单行输入，左模糊 |
| 场景分类       | scene_category        | 单行输入，左模糊 |
| 输入标签       | input_tags            | 单行输入，左模糊（可提示「多个标签以逗号分隔」等） |
| 回复标签       | response_tags         | 单行输入，左模糊 |

### 5.3 列表列建议

- 主键 id（可选，用于调试或详情跳转）
- 场景摘要（截断展示）
- 优化问题（截断展示）
- 场景分类
- 输入标签 / 回复标签（可合并为一列或分两列，以标签形式展示）
- 技术标记分类 / 业务标记分类（可折叠或详情页展示，避免列表过长）
- 创建时间 / 更新时间（可选）

### 5.4 分页组件

- 展示：当前页、总条数、每页条数、上一页/下一页及可选页码。
- 请求：切换页时仅改变 `offset`（或 `limit`），其余查询参数不变；与接口 `limit`/`offset` 一致。

## 6. 实现注意事项

1. **optimization_question**：若存 JSON 数组，左模糊可对序列化后的字符串做 `LIKE 'keyword%'`，或对数组内元素做「任一元素左匹配」。
2. **input_tags / response_tags**：数组左模糊可统一为「序列化后 like」或「数组包含以 keyword 开头的元素」，需与产品统一语义并在文档中写明。
3. **技术/业务标记分类**：JSON 结构需与上下游约定（例如层级、键名），便于后续扩展与检索。
4. **权限与审计**：若知识库为多租户或多角色，需在接口层增加租户/权限校验；重要变更可记审计日志。
