# Base 批量插入与批次任务创建详细技术设计方案

> 依据设计文档：`cursor_docs/022804-Base批量插入与批次任务创建分批方案技术设计.md` 展开的详细技术方案，含接口约定、实现步骤、代码位置与测试要点。

| 项目 | 说明 |

| 目标 | 在 Base 层提供批量插入（按 chunk flush），业务层按批调用，将 10 万级 task 的 flush 次数从 N 降为约 N/1000 |
| 关联设计 | 022804-Base批量插入与批次任务创建分批方案技术设计.md |

## 2. 现状与问题（简要）

- **Base 现状**：`BaseRepository.create()` 单条 `session.add()` 后立即 `await self.session.flush()`，保证自增/ULID 等可读。
- **业务现状**：`CreateTemplate._build_and_persist_batch()` 对 `task_items` 循环调用 `_task_repo.create()`，每条一次 flush。
- **问题**：10 万条 → 10 万次 flush，带来网络往返、PG 解析/规划/执行与 WAL、锁、I/O 压力过大。
- **目标**：Base 层按 chunk（默认 1000 条）分组插入并每组一次 flush；业务层按「每批最多 1 万条」拆分后每批调用 Base 批量接口，整体 flush 次数降为「批次数 × 每批内 chunk 数」。

---

## 3. 分层职责与常量

### 3.1 常量定义（建议集中配置）

| 常量名 | 建议值 | 作用域 | 说明 |
|--------|--------|--------|------|
| `DEFAULT_CHUNK_SIZE` | 1000 | Base 层 | 每多少条执行一次 `add_all` + `flush` |
| `TASK_LIST_BATCH_SIZE` | 10_000 | CreateTemplate | 单次传给 `create_many` 的 task 列表上限 |

后续若需调优，可改为从配置或上层入参传入，文档预留该扩展点。

### 3.2 分层职责表

| 层级 | 文件/类 | 职责 | 关键行为 |
|------|---------|------|----------|
| Base | `repository/base.py` → `BaseRepository` | 通用「dict 列表 → 按 chunk 插入并 flush」 | 不补审计字段；不 commit |
| AuditBase | `repository/base.py` → `AuditBaseRepository` | 在 Base 语义上对每条补 id/version/is_deleted/create_time | 与现有单条 `create()` 一致 |
| Task Repository | `repository/batch/batch_task_repository.py` → `BatchTaskRepository` | 将 `TaskPreCreateItem` 转为 kwargs 列表，调 Base/AuditBase 的批量方法 | 固定 job_id、status=pending |
| CreateTemplate | `domain/batch/batch_template.py` → `CreateTemplate` | 先得到完整 task_items；按 1 万条分批；每批调用 `task_repo.create_many` | 与 job 创建同 session、同事务 |

## 4. Base 层详细设计

### 4.1 方法签名（BaseRepository）

**新增方法**：`create_many`

```python
async def create_many(
    self,
    items: List[Dict[str, Any]],
    chunk_size: int = 1000,
) -> int:
    """
    批量创建记录：按 chunk_size 分组，每组 add_all 后执行一次 flush，不 commit。

    Args:
        items: 每项为可传入 self.model(**kwargs) 的键值对。
        chunk_size: 每多少条为一组执行一次 flush，默认 1000。

    Returns:
        实际插入条数。
    """
```

- **入参**：`items` 由具体 Repository（如 BatchTaskRepository）从业务 DTO 转成；`chunk_size` 默认 1000，可选。
- **行为**：
  - `items` 为空：直接返回 `0`，不执行 flush。
  - 按 `chunk_size` 切分为多组；对每组：循环 `self.model(**kwargs)` 并 `session.add(instance)`，然后 `await self.session.flush()`。
  - 不调用 `session.commit()`，由上层统一提交。
- **返回值**：插入条数（当前不返回主键列表；若后续需要可再扩展 RETURNING 或逐 chunk 收集 id）。

### 4.2 AuditBaseRepository 对 create_many 的增强

在 **AuditBaseRepository** 中实现（或重写）`create_many`，在将每条 item 传入 `self.model()` 之前补全审计字段，与现有单条 `create()` 一致：

- `id`：未传则 `generate_ulid()`
- `version`：未传则 `0`
- `is_deleted`：未传则 `False`
- `create_time`：未传则 `_now()`

实现方式二选一：

- **方案 A**：在 `AuditBaseRepository` 中直接实现 `create_many`（对每条 item 补字段后调用 `super().create_many()` 需注意 Base 的 `create_many` 接收的是已补全的 dict）。
- **方案 B**：在 `BaseRepository` 中实现「裸」`create_many`（不补字段），在 `AuditBaseRepository` 中重写 `create_many`，对 `items` 逐条补全后再 `await super().create_many(items, chunk_size)`。

推荐 **方案 B**：Base 保持「只做按 chunk 插入+flush」，AuditBase 负责审计字段补全，职责清晰。

### 4.3 与现有 create() 的关系

- 单条 `create()` 保持现状，用于需要立即拿到 `instance.id` 的场景。
- `create_many()` 仅用于「批量写入、无需逐条拿 id」的场景；同一 session 内可与 `create()` 混用（如先 `create` job，再 `create_many` tasks）。

### 4.4 代码位置与实现要点

| 文件 | 类 | 改动 |
|------|-----|------|
| `backend/infrastructure/database/repository/base.py` | `BaseRepository` | 新增 `create_many(items, chunk_size=1000) -> int`，按 chunk 循环 add + flush |
| `backend/infrastructure/database/repository/base.py` | `AuditBaseRepository` | 重写 `create_many`：对 items 逐条补 id/version/is_deleted/create_time 后调用 `super().create_many()` |

## 5. Task Repository 层详细设计

### 5.1 方法签名（BatchTaskRepository）

**新增方法**：`create_many`

```python
async def create_many(
    self,
    job_id: str,
    items: List[TaskPreCreateItem],
    chunk_size: int = 1000,
) -> int:
    """
    批量创建子任务。将 TaskPreCreateItem 转为 kwargs 列表后调用 Base 的 create_many。

    Args:
        job_id: 批次 job 的 id。
        items: 预创建项列表（来自 CreateTemplate.query_tasks_for_pre_create）。
        chunk_size: 每多少条一组 flush，默认 1000。

    Returns:
        插入条数。
    """
```

### 5.2 转换规则（TaskPreCreateItem → Dict）

与现有单条 `create()` 保持一致：

- `job_id`：由参数传入
- `source_table_id`：`item.source_table_id`
- `source_table_name`：`item.source_table_name`
- `status`：固定 `"pending"`
- `runtime_params`：`item.runtime_params`
- `redundant_key`：`item.redundant_key`

审计字段由 AuditBaseRepository 的 `create_many` 补全，此处不传。

### 5.3 代码位置

| 文件 | 类 | 改动 |
|------|-----|------|
| `backend/infrastructure/database/repository/batch/batch_task_repository.py` | `BatchTaskRepository` | 新增 `create_many(job_id, items, chunk_size=1000) -> int`，转换后调用 `super().create_many(dict_list, chunk_size)` |
| `backend/domain/batch/dto.py` | （无改动） | `TaskPreCreateItem` 已存在，无需修改 |

## 6. CreateTemplate 业务层改造

### 6.1 当前逻辑（简述）

- `_build_and_persist_batch()`：先 `_job_repo.create()` 得到 `job`，再 `for item in task_items: await self._task_repo.create(...)`。

### 6.2 改造后逻辑（步骤）

1. **创建 job（不变）**  
   `job = await self._job_repo.create(job_type=..., code=..., total_count=len(task_items), query_params=...)`，保证 `job.id` 可用。

2. **按批切分 task_items**  
   使用常量 `TASK_LIST_BATCH_SIZE = 10_000`：
   ```python
   batches = [
       task_items[i : i + TASK_LIST_BATCH_SIZE]
       for i in range(0, len(task_items), TASK_LIST_BATCH_SIZE)
   ]
   ```

3. **每批调用 create_many**  
   对每个 `batch`：  
   `await self._task_repo.create_many(job_id=job.id, items=batch, chunk_size=1000)`  
   Base 内部再按 1000 条一组 flush，即每批 1 万条最多 10 次 flush。

### 6.3 边界与异常

- `task_items` 为空：已在 `create_batch()` 入口 `raise ValueError`，无需在 `_build_and_persist_batch` 内重复处理。
- 某批为空：按固定 size 切分一般不会出现；若出现，`create_many(items=[])` 直接返回 0 即可。
- 异常：任一批或任一块插入失败（如唯一约束），异常向上抛，整个事务回滚；建议在调用链上打日志（job_id、batch 下标、chunk 范围）便于排查。

### 6.4 代码位置

| 文件 | 类/方法 | 改动 |
|------|---------|------|
| `backend/domain/batch/batch_template.py` | `CreateTemplate` | 在文件顶部或类附近定义 `TASK_LIST_BATCH_SIZE = 10_000`；在 `_build_and_persist_batch` 中先 create job，再按批切分并循环调用 `_task_repo.create_many(job_id, batch, chunk_size=1000)` |

## 7. 数据流与调用链

```
CreateTemplate.create_batch(session, query_params)
  → query_tasks_for_pre_create()           # 得到 task_items（如 10 万条）
  → _build_and_persist_batch(...)
       → _job_repo.create(...)             # 1 次 flush，得到 job.id
       → batches = 按 1 万条切分 task_items
       → for batch in batches:
            await _task_repo.create_many(job_id, batch, chunk_size=1000)
              → AuditBaseRepository.create_many(dict_list, 1000)
                   补审计字段 → BaseRepository.create_many(...)
                     每 1000 条：add 若干实例 → flush   # 每批 1 万条 = 10 次 flush
  → 由上层统一 session.commit()
```

**10 万条示例**：10 批 × 每批 10 次 flush = 100 次 flush（加 job 的 1 次），相对 10 万次 flush 大幅减少。

## 9. 风险与注意事项

### 9.1 事务与一致性

- 所有 `create_many` 与 job 的 `create` 必须在同一 `AsyncSession`/同一事务内执行；若中间抛异常，整体回滚。
- 需确认：CreateTemplate 的调用方（如 `batch_job_service` 或路由）是否保证同一 session 贯穿 `_build_and_persist_batch` 整段。

### 9.2 内存

- 「先构建完整 list 再分批」意味着 10 万条 `TaskPreCreateItem` 会常驻内存；若未来规模到百万级，可考虑流式/分页从 `query_tasks_for_pre_create` 取数并按批写入。当前 1 万/批主要控制单次调用与事务内 flush 次数，不是内存上限。

### 9.3 chunk_size 与 TASK_LIST_BATCH_SIZE

- **chunk_size**：Base 层「每多少条 flush 一次」，控制单次 flush 的 SQL 量与 PG 压力。
- **TASK_LIST_BATCH_SIZE**：业务层「单次传给 create_many 的 list 上限」，控制单次调用与单段事务内数据量。
- 两者独立；可根据 PG 与驱动表现调优（例如 chunk_size 改为 2000，则每批 1 万条仅 5 次 flush）。

### 9.4 返回值与幂等

- 当前不依赖「批量插入后返回每条 task 的 id」；若后续需要（如落库后立刻发 MQ 带 task_id），可在 Base 或 Repository 层为每 chunk 使用 RETURNING 收集 id 并返回。
- 幂等仍由业务层 `redundant_key` 等保证，与单条 create 一致。

### 9.5 空列表与异常

- Base：`create_many(items=[])` 不执行任何 flush，返回 0。
- 若某 chunk 插入时 PG 报错（如唯一约束），异常向上抛，整事务回滚；建议保留清晰日志（job_id、batch 序号、chunk 范围）便于排查。

## 10. 测试建议

### 10.1 单元/集成测试

| 场景 | 说明 |
|------|------|
| Base.create_many 空列表 | 不 flush，返回 0 |
| Base.create_many 1 条、999 条、1000 条、1001 条 | 验证 chunk 切分与 flush 次数 |
| AuditBase.create_many | 验证审计字段自动补全且与单条 create 一致 |
| 同 session 先 create(job) 再 create_many(tasks) | 兼容性：混用单条与批量 |
| BatchTaskRepository.create_many | 转换 TaskPreCreateItem → dict 正确，条数一致 |
| CreateTemplate._build_and_persist_batch | 0 条由入口报错；1 条、1 万条、10001 条切分正确，总 task 数与 job.total_count 一致 |

### 10.2 边界与压力

- 边界：0 条、1 条、恰好 1000/10000 条、10001 条，验证批数与 chunk 数符合预期。
- 大数量：如 10 万条集成测试或压测，观察 PG 连接数、WAL、锁等待是否在可接受范围内。

- 测试文件放在 `cursor_test` 下，命名如 `test_batch_create_many.py`、`test_base_create_many.py`，覆盖上述场景。

## 12. 小结

- **Base**：新增 `create_many(items, chunk_size=1000)`，按 chunk 插入并 flush；AuditBase 对每条补 id/version/is_deleted/create_time 后调用 Base。
- **Task Repository**：新增 `create_many(job_id, items, chunk_size=1000)`，将 TaskPreCreateItem 转成 dict 列表后调 Base。
- **CreateTemplate**：先构建完整 task_items，按 max_batch_size=10000 分批，每批调用 `_task_repo.create_many`，在保证「job 先写、再写 tasks」的前提下，将 flush 次数从 N 降为约 N/1000（再乘以批次数），并便于后续按 PG 表现调整 chunk_size 与 max_batch_size。

本方案在保持与现有单条 `create()` 及审计逻辑一致的前提下，给出可落地的接口、实现位置与测试要点，可直接用于开发与代码评审。
