# 提示词拆分为向量话术知识库与轻量提示词可行性分析

> 基于医疗 QA 提示词（如 `50-QA_agent.md`）的改造思路总结。  
> 场景：原提示词将「边界场景定义 + 患者示例 + 回复话术」融合在一处，导致单提示词庞大、回复话术固定、维护成本高。

---

## 一、拆分思路总览

| 拆分目标 | 内容归属 | 说明 |
|---------|----------|------|
| **向量话术知识库** | 场景定义、匹配标准、患者问题示例、回复话术 | 按「场景 + 条件」建 chunk，用 RAG 按用户问题检索，只注入相关片段 |
| **轻量提示词** | 角色、核心原则、使用方式、回复风格、硬性规则、输出格式 | 固定模板，体量控制在数十行内，不含具体场景与话术枚举 |

---

## 二、适宜进入「向量话术知识库」的内容

原提示中 **「边界场景定义及回复话术」** 整段（约 14–328 行）适合向量化：

| 内容类型 | 用途 |
|----------|------|
| **场景定义 + 匹配标准** | 如「诊疗-疾病诊断」「药物-用药剂量」「危重症-胸痛」等，作为 chunk 的「适用条件」描述，便于检索时按意图/场景匹配 |
| **患者问题示例** | 典型问法，作为同一 chunk 的补充或单独示例库，提升「用户问法 → 场景」的召回 |
| **回复话术** | 每类场景下的标准话术或可选话术，作为该场景的「标准回复素材」，检索到即得到话术，便于单独维护、多版本、多风格 |

**结论**：  
凡属于「在什么条件下、对什么问题、该用什么话术」的枚举型知识，都放入向量库，由 RAG 在对话时按需注入，而不是全部写进主提示词。

---

## 三、适宜保留在「轻量提示词」的内容

以下属于「怎么用检索结果、怎么组织回复」的元规则，留在主提示中并尽量精简：

- **角色与核心原则**：医疗 QA 助手、安全边界优先等（可压缩为几句话）
- **使用方式**：根据「检索到的话术知识」+ 用户当前问题生成回复；多场景命中时取更保守的一侧
- **流程说明**：理解问题 → 用检索结果匹配场景与话术 → 若有占位符（如 `[症状描述]`）则用用户原话填充 → 按风格要求组织成自然回复
- **回复风格**：自然、友好、口语化，不机械列举、不照读条款（可保留 1–2 个正反例）
- **硬性规则**：紧急情况优先、个人信息保护、非诊断非治疗、儿童问题必须引导家长等（用短条列即可）
- **输出格式**：沿用现有占位符（如 `{end_llm_resopnse}`）说明

**目标**：轻量提示词只承载「角色 + 原则 + 如何用检索结果 + 风格与约束」，不承载具体场景与话术原文。

---

## 四、可行性结论

### 1. 单提示词过大 → 可解决

- **现状**：整份提示约 397 行，大部分是「场景 + 示例 + 话术」枚举。
- **拆分后**：提示词收敛到「角色 + 原则 + 用法 + 风格 + 约束 + 输出」，体量可压到几十行；具体「有哪些场景、每类怎么说」由 RAG 按当前问题检索后注入，只带入围的相关知识，上下文更短、更聚焦。

### 2. 回复话术固定 → 可缓解

- **现状**：话术写死在提示里，调整需改大段文本，也不方便做多版本、多风格。
- **拆分后**：话术存在向量库（或关联的结构化表）中，可按「场景 ID + 版本 + 类型（标准/简短/温和）」维护多条；检索时可按场景、用户情绪或紧急程度选用不同话术，或在提示中写「在符合安全边界的前提下，可对检索到的话术做适度口语化改写」，在合规与灵活之间取得平衡。

**综合判断**：从「减小单提示词体量」和「让话术可配置、可扩展」两方面看，拆成「向量话术知识库 + 轻量提示词」是可行且值得推进的。

## 五、实施要点与注意事项

### 1. 危重症等「分支多」的场景

危重症段有大量「是否伴随高危」的细分（如胸痛是否冠心病、是否高危体征等），同一症状下存在多条不同话术或动作（如打 120、去急诊、联系医生等）。

**建议**：

- **知识库**：按「场景粒度」存，例如「危重症-胸痛-冠心病用户」「危重症-胸痛-非冠心病-伴高危」「危重症-胸痛-非冠心病-非高危」各成一条或一个 chunk，每条包含「判定要点（短）」+「对应话术」。
- **轻量提示词**：写一句显式规则，如「若检索结果属于危重症类，需根据用户是否冠心病、是否描述高危体征等，选用检索到的对应分支话术；若有多个匹配，优先采用更安全、更紧急的那一条。」

既把话术内容从提示词里迁出，又保留「按条件选话术」的决策逻辑在提示词中，避免模型在长枚举中迷失。

### 2. 话术中的占位符（如 `[症状描述]`）

当前有「将患者描述代入话术」的用法（如紧急情况模板里的 `[症状描述]`）。

**建议**：在轻量提示词中固定写一条：「若检索到的话术中含有 `[症状描述]`、`[具体症状]` 等占位符，用用户原话或简短归纳填充，保持语句通顺。」知识库条目可保留占位符，不必在入库前展开成无数种说法。

### 3. 多场景同时命中时的优先级

同一句用户问题可能同时命中「药物-剂量」「诊疗-用药调整」或「危重症」等。

**建议**：在轻量提示词里写一条显式规则，如「若一次检索命中多个场景，以更严格、更保守的场景为准（例如：危重症 > 药物/诊疗）；若为同一层级，可合并话术要点，但不得弱化安全边界。」

### 4. 检索质量与 chunk 设计

- **chunk 设计**：以「场景 + 条件」为最小单位，例如「药物-用药剂量-话术」「危重症-胸痛-非冠心病-伴高危-话术」。每个 chunk 内包含：场景 ID、适用条件/判定要点（一两句话）、患者示例问题（1–3 句）、该条件下的回复话术（可多句）。
- **查询**：可直接用「用户当前问题」作 query；若有意图识别，可用「意图 + 问题」拼接后再查，提高命中率。
- **评估**：用一批「用户问题 → 期望场景 → 期望话术类型」的测试用例，评估检索 top-k 是否包含正确场景，以及最终回复是否仍符合安全边界、语气是否符合要求。

## 六、其它可选建议

1. **话术版本化**：在知识库中为每条话术加「版本号、生效时间、下线时间」，便于灰度、回滚和合规审计，而不必改提示词。
2. **同一场景多版话术**：同一场景下存多条话术（如「标准版 / 简短版 / 更温和版」），在检索或路由时按「用户情绪、问题紧急度、产品策略」选择一条，再交给模型做适度口语化，既保留「边界固定」，又减少「回复千篇一律」感。
3. **分层检索（可选）**：若场景非常多，可先按「大类」（诊疗/药物/危重症/院内/其他）做一次粗筛（规则或小模型），再在该类下用向量检索「场景 + 话术」，兼顾准确率与延迟。
4. **保留极简「核心清单」在提示词**：若希望模型在无检索结果或检索不确定时仍有兜底，可在轻量提示词里保留极简的「场景清单 + 一句话原则」（如「危重症 → 必须先引导拨打 120/就医」），不展开具体话术，具体话术仍以检索结果为准。

---

## 七、建议实施顺序

1. **设计 chunk 结构**：按「场景 + 条件」划分，把现有文档中的场景与话术整理成可入库的条目（可先从诊疗/药物等相对简单的开始）。
2. **编写轻量提示词**：含「如何使用检索结果、占位符替换、多场景优先级、风格与硬性规则」。
3. **建设与评估**：用一组标注好的「问题 → 应有场景/话术」做检索与端到端测试，再迭代 chunk 设计和提示词表述。
4. **危重症细粒度拆分**：把危重症按「症状 + 是否高危/是否冠心病等」拆成多条知识，在提示词里用 1–2 条规则说清「按条件选用检索到的话术」即可。
