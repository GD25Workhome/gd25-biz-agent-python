# 医疗流程 V6.1 与 V6.2 融合方案分析

## 文档说明

**文档版本**：V1.0  
**创建时间**：2025-01-29  
**目标**：将 `medical_agent_v6_1`（血压专项）与 `medical_agent_v6_2`（QA）融合为 `medical_agent_v6_3`，梳理多方案并给出优缺点与评分。

---

## 一、待融合流程现状

### 1.1 共同设计思路

两个流程均为「问答型」设计，整体链路一致：

1. **builder_prompt_context_node**：查询用户基础信息（当前均为 `query_list: [blood_pressure]`）。
2. **optimization_agent**：词干/场景提取，输出供 RAG 使用的摘要与优化问题。
3. **rag_node**：用词干做向量检索，得到相似案例与（可选）科普文章。
4. **core_agent**：在 RAG 上下文基础上做问答、数据记录等核心动作。

### 1.2 差异点

| 项目 | V6.1（血压） | V6.2（QA） |
|------|--------------|------------|
| optimization_agent 提示词 | `10-optimization-agent.md`（v6_1 目录，含血压预警/评价规则） | `10-optimization-agent.md`（v6_2 目录，通用场景摘要与优化问题） |
| core_agent 提示词 | `50-core_agent.md`（血压点评、记录） | `50-QA_agent.md`（通用问答） |
| core_agent 工具 | `record_blood_pressure` 等 | 无工具配置 |

结论：**可合并的理由成立**——差异集中在 optimization_agent 与 core_agent 的提示词（及 core 侧工具），流程骨架一致，适合在一个图中通过「意图 + 条件路由」合并。

---

## 二、融合方案汇总

### 方案一：optimization_agent 增加意图 + 双 core_agent 条件路由（用户方案）

**思路**：保留一个 optimization_agent，在其输出中增加「意图」字段（如 `intent: "qa" | "blood_pressure"`）；RAG 之后由 LangGraph 根据 `intent` 做条件边，路由到两个不同的 core_agent 节点（不同 prompt + 不同工具）。

**流程结构示意**：

```
builder_prompt_context_node
    → optimization_agent（输出含 scene_summary / optimization_question / intent）
    → rag_node
    → [条件边] intent == 'qa'           → core_agent_qa（50-QA_agent.md）
    → [条件边] intent == 'blood_pressure' → core_agent_bp（50-core_agent.md + record_blood_pressure）
```

**要点**：

- optimization_agent 提示词需融合 v6_1 / v6_2 的职责，并明确输出 `intent`。
- 条件边从 `rag_node` 出发，依赖 `state.edges_var["intent"]`。当前实现中 **rag_node 会整体覆盖 `edges_var`**，因此需要：**rag_node 写回状态时保留上游的 `intent`**（例如合并 `edges_var` 而非整体覆盖），否则需改 builder/状态结构以在条件边处拿到意图。

**优点**：

- 与现有「先查用户信息 → 词干 → RAG → 核心 agent」的思路一致，易理解。
- 意图与词干在同一步完成，只多一个输出字段，改动集中。
- 两个 core_agent 完全独立（提示词、工具可分别维护），互不干扰。
- 本项目的 `flow.yaml` + `ConditionEvaluator` 已支持基于 `edges_var` 的条件边，扩展新意图只需加节点和边。

**缺点**：

- 需修改 optimization_agent 提示词并保证 `intent` 稳定输出；若识别不准，会路由到错误 agent。
- 需保证 rag_node 对 `edges_var` 的写回策略（保留 intent），或改状态设计，有一处实现成本。

---

### 方案二：独立「路由节点」+ 双 core_agent

**思路**：optimization_agent 只负责词干与可选的结构化标签；在 rag_node 之后增加一个**纯函数节点**（如 `route_by_intent_node`），该节点只做一件事：根据当前 state（如上一节点写入的 `edges_var` 或专门字段）解析或赋值 `intent`，不调用 LLM。条件边仍从该路由节点出发，指向 core_agent_qa / core_agent_bp。

**流程结构示意**：

```
builder_prompt_context_node
    → optimization_agent（仅词干 + 可选 intent 或其它标签）
    → rag_node
    → route_by_intent_node（从 state 读 intent，必要时用规则/小模型补全）
    → [条件边] intent == 'qa'           → core_agent_qa
    → [条件边] intent == 'blood_pressure' → core_agent_bp
```

**优点**：

- 意图逻辑与「词干提取」解耦：optimization 专注摘要与优化问题，意图可放在路由节点用规则或轻量模型处理。
- 若未来意图来源变化（如从别处 API 获取），只需改路由节点，不动 optimization 与 core。
- 便于测试：路由节点可单测，条件边行为清晰。

**缺点**：

- 多一个节点和一条边，图略复杂。
- 若意图仍由 optimization_agent 输出，则与方案一类似，仍需保证 intent 能传递到路由节点（如 rag_node 合并 edges_var）。

---

### 方案三：单 core_agent + 动态提示词（按 intent 选 prompt）

**思路**：只保留一个 core_agent 节点，在**构建图时**根据 state 中的 intent 选择不同的 prompt 文件（如 `50-QA_agent.md` 或 `50-core_agent.md`）注入。即「一个节点，多套提示词，运行时按意图选一套」。

**流程结构示意**：

```
builder_prompt_context_node
    → optimization_agent（输出含 intent）
    → rag_node
    → core_agent（内部/框架层按 state.intent 选择 prompt 与 tools）
```

**优点**：

- 图中只有一个 core 节点，结构最简单。
- 新增意图类型时只需增加 prompt 与分支逻辑，不增加节点数。

**缺点**：

- 当前 flow 是 YAML 驱动、按节点静态配置 prompt，**没有「根据 state 动态选 prompt」的现成机制**，需要扩展 builder 或节点创建逻辑（例如 agent 节点支持 `prompt_by_intent` 或从 state 读 prompt key）。
- 工具绑定通常也是按节点配置的，要支持「同一节点不同 intent 用不同 tools」，也要扩展配置或代码。
- 实现成本和侵入性相对高，且与现有「一节点一 prompt」的模型不一致。

---

### 方案四：单 core_agent + 合并大提示词（意图分支写进 prompt）

**思路**：只保留一个 core_agent 节点，使用**一份合并后的提示词**，在 system 里用自然语言或结构化指令写清：「若意图为 QA 则执行……；若意图为血压则执行……（含点评规则与记录要求）」。意图仍由 optimization_agent 输出并传入 state。

**流程结构示意**：

```
builder_prompt_context_node
    → optimization_agent（输出含 intent）
    → rag_node
    → core_agent（50-merged_agent.md，内部分段描述 QA vs 血压）
```

**优点**：

- 无需改图结构，也不需要条件边或新节点。
- 实现快，不依赖框架扩展。

**缺点**：

- 单 prompt 体积大、易冲突，后续维护成本高（血压规则 + QA 规则混在一起）。
- 模型容易混淆两种模式，不利于专项优化与评测。
- 工具配置仍需在节点上同时挂载 QA 与血压两类，或在大 prompt 里约束「按意图只调用某类工具」，可控性不如分开节点。

---

### 方案五：子流程编排（主流程按意图调 v6_1 / v6_2 子图）

**思路**：主流程只做「意图识别 + 选子流程」；子流程 A = 当前 v6_1 的整图，子流程 B = 当前 v6_2 的整图。主图入口 → 意图节点 → 调用子图 A 或子图 B。

**流程结构示意**：

```
主流程：
  entry → intent_node（或 optimization_agent 仅做意图）
       → [条件边] intent == 'blood_pressure' → 子流程 v6_1（builder → optimization → rag → core_bp）
       → [条件边] intent == 'qa'              → 子流程 v6_2（builder → optimization → rag → core_qa）
```

**优点**：

- v6_1 / v6_2 完全不动，复用现有 YAML 与提示词。
- 适合「多业务线、多独立流程图」的长期架构。

**缺点**：

- 需要框架支持「子图」或「嵌套图」的加载与调用，当前项目是否支持尚不明确，实现量最大。
- 每个请求会走两套「用户信息查询 + optimization + RAG」中的一套，与「先共用一个 optimization + RAG，再分 core」相比，没有节省公共段；若未来要共用 RAG 再分 core，又要再拆一层。
- 对「只是把两个 core 能力合并到一个图」的目标来说，偏重。

## 三、方案对比与评分

| 维度 | 方案一（意图+双 core） | 方案二（路由节点+双 core） | 方案三（单 core+动态 prompt） | 方案四（单 core+合并 prompt） | 方案五（子流程编排） |
|------|------------------------|----------------------------|-------------------------------|-------------------------------|----------------------|
| 与现有 YAML/条件边兼容性 | 高（仅需保留 intent） | 高 | 低（需扩展 builder/节点） | 高 | 低（需子图能力） |
| 实现与改造成本 | 中（改 optimization 提示词 + rag 写回策略） | 中高（多一节点） | 高 | 低 | 最高 |
| 可维护性（提示词与工具） | 高（两个 core 独立） | 高 | 中 | 低（大 prompt 难维护） | 高（子图独立） |
| 意图与路由清晰度 | 高 | 最高 | 中 | 中 | 高 |
| 扩展新意图（如用药、复诊） | 容易（加节点+边+prompt） | 容易 | 需改动态逻辑 | 需改大 prompt | 需加子图 |
| 对现有代码侵入 | 小（rag 合并 edges_var 或等价改动） | 小 | 大 | 最小 | 大 |

**综合评分（1–5 分，5 为最佳）**：

| 方案 | 兼容性 | 实现成本 | 可维护性 | 扩展性 | 综合评分 |
|------|--------|----------|----------|--------|----------|
| 方案一 | 5 | 4 | 5 | 5 | **4.75** |
| 方案二 | 5 | 3.5 | 5 | 5 | **4.5** |
| 方案三 | 2 | 2 | 3 | 3 | **2.5** |
| 方案四 | 5 | 5 | 2 | 2 | **3.5** |
| 方案五 | 2 | 1 | 5 | 4 | **3** |

## 四、推荐结论与落地要点

### 4.1 推荐方案

- **首选：方案一（optimization_agent 增加意图 + 双 core_agent 条件路由）**  
  与你的设想一致，在不引入子图或动态 prompt 的前提下，把「意图识别」与「词干提取」放在同一节点，用现有条件边机制路由到两个功能 agent，实现简单、易维护、易扩展。

- **备选：方案二（独立路由节点 + 双 core_agent）**  
  若希望「意图」与「词干」完全解耦，或意图来源将来可能从别处来，可采用方案二；多一个节点，逻辑更清晰，但实现稍多一步。

### 4.2 方案一落地要点（v6_3）

1. **optimization_agent 提示词融合与意图**  
   - 在现有「场景摘要 + 优化问题」基础上，增加输出字段：`intent`，取值如 `"qa"`、`"blood_pressure"`（必要时可加 `"unclear"` 并约定默认路由）。  
   - 若 v6_1 的 optimization 提示词含有血压专用规则，可保留为「当判定为血压相关时遵循的规则」，并在同一份提示词中说明：何时填 `intent: "blood_pressure"`，何时填 `intent: "qa"`。

2. **状态与条件边**  
   - 条件边从 `rag_node` 出发，依赖 `state.edges_var["intent"]`。  
   - **必须保证**：rag_node 在写回 `edges_var` 时**保留** optimization_agent 写入的 `intent`（例如在写 `edges_prompt_vars` 时合并原有 `edges_var`，而不是整体替换）。若当前实现是整体覆盖，需在 rag_agent 节点或 builder 中做一次合并逻辑。

3. **flow.yaml（v6_3）结构建议**  
   - **节点**：  
     - `builder_prompt_context_node`、`optimization_agent`、`rag_node` 各一个。  
     - 两个 core 节点：如 `core_agent_qa`（prompt: `50-QA_agent.md`，无工具）、`core_agent_bp`（prompt: `50-core_agent.md`，tools: `record_blood_pressure` 等）。  
   - **边**：  
     - `builder_prompt_context_node` → `optimization_agent`（always）  
     - `optimization_agent` → `rag_node`（always）  
     - `rag_node` → `core_agent_qa`（condition: `intent == 'qa'`）  
     - `rag_node` → `core_agent_bp`（condition: `intent == 'blood_pressure'`）  
   - 若存在 `intent` 为其它或缺失，可在条件边中增加默认分支（如指向 `core_agent_qa` 或单独 fallback 节点）。

4. **提示词与目录**  
   - 将 v6_3 的 optimization 提示词放在 `config/flows/medical_agent_v6_3/prompts/` 下（如 `10-optimization-agent.md`），并在其中约定 `intent` 的取值与规则。  
   - core 侧继续引用 `50-QA_agent.md` 与 `50-core_agent.md`（可从 v6_2 / v6_1 复制或复用）。

按上述要点即可在 `medical_agent_v6_3` 中完成融合，并保留后续加新意图（如用药、复诊）时仅增节点与边的扩展方式。
