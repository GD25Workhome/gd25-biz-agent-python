# Feishu 导入设计模式评审与清洗转换分层架构

> 评审当前解析策略设计模式的局限性，提出「数据清洗与格式转换分层」架构，解决不同 Excel/Sheet 数据转换逻辑差异化问题。

---

## 一、问题来源

在实现「患者无数据+历史会话+历史Action」Sheet 的改造时，020303 文档提出在 `parsers/base.py` 中新增 `parse_history_qa_to_messages`。该方案存在设计问题：

- **不应在 base 层增加 Sheet 专属解析逻辑**：`parse_history_qa_to_messages` 仅服务于特定 Sheet 的「历史会话」Q/A 格式，放入 base 会污染通用层。
- **扩展性差**：每增加一种 Sheet 格式，就要在 base 或现有解析器中塞入新逻辑，导致 base 膨胀、策略类职责混杂。

**核心诉求**：不同 Excel、Sheet 的数据清洗逻辑各异，但希望架构简单、易扩展——各清洗逻辑独立实现，清洗后输出统一格式，再由统一转换逻辑生成 DataSetItem。

## 二、当前设计模式评审

### 2.1 现有架构

```
Excel 行 (pd.Series)
  → Parser.extract_items_from_row(row, df)  → List[Dict] (row_data)
  → FieldProcessorChain.process(row_data)    → DataSetItemData
  → ItemWriter.write_item()                 → Langfuse
```

### 2.2 设计模式

| 模式 | 实现 | 职责 |
|------|------|------|
| 策略模式 | `BaseSheetParser` 子类（LskParser、Sh1128Parser、Sh1128MultiParser） | 按 Sheet 选择解析策略 |
| 责任链 | `FieldProcessorChain`（Normalize → Assemble） | 字段规范化 + 组装 DataSetItem |

### 2.3 局限性分析

#### 2.3.1 清洗与转换未分离

当前 `Parser` 同时承担：

1. **数据清洗**：从 Excel 原始行提取、清洗、转换（列映射、空值、Q/A 拆分、历史会话解析等）
2. **中间结构**：产出 `row_data`（Dict），结构由 base 约定
3. **格式转换**：`to_dataset_item(row_data)` 将 row_data 转为 DataSetItemData

问题：清洗逻辑与「row_data → DataSetItem」的转换逻辑耦合在 Parser 与 base 中。新增 Sheet 时，要么改 base（如加 `parse_history_qa_to_messages`），要么在子类中重写 `extract_row_data`、`extract_items_from_row`、`to_dataset_item`，导致：

- base 被 Sheet 专属逻辑污染
- 子类职责过重，难以复用

#### 2.3.2 row_data 结构过载

`row_data` 需同时满足：

- 多种 Sheet 的提取结果（history_session/history_response、history_messages、current_session、response 等）
- `to_dataset_item` 的输入约定
- NormalizeHandler 的假设（current_session、response 等）

结果：row_data 成为「万能字典」，不同 Sheet 往不同 key 塞数据，`to_dataset_item` 通过分支（如 `history_messages` 是否存在）处理，可读性和可维护性下降。

#### 2.3.3 扩展新 Sheet 的成本高

以「患者无数据+历史会话+历史Action」为例：

| 方案 | 问题 |
|------|------|
| 在 base 加 `parse_history_qa_to_messages` | 污染 base，违反单一职责 |
| 新增 Sh1128HistoryQAParser，重写 extract_row_data | 需理解 base 的 row_data 约定，并在子类中手工组装 history_messages |
| 在 to_dataset_item 中根据 history_session 格式分支 | 转换逻辑与清洗逻辑混在一起，base 继续膨胀 |

#### 2.3.4 空字符 "-" 等通用规则的位置

`-` 视为空属于**通用清洗规则**，适合放在 base 的工具函数中。但「历史会话 Q/A 解析」属于**某 Sheet 专属规则**，不应放在 base。当前设计未区分「通用」与「Sheet 专属」，容易把两者都塞进 base。

## 三、分层架构：清洗 → 统一格式 → 转换

### 3.1 核心思想

**分离关注点**：

1. **数据清洗（Sheet 专属）**：各 Sheet 独立实现，从 Excel 行 → 清洗后的**规范格式**
2. **格式转换（通用）**：仅负责 规范格式 → DataSetItem，逻辑唯一、稳定
3. **规范格式**：作为清洗与转换之间的契约，固定、明确

### 3.2 数据流

```
Excel 行 (pd.Series)
  → [Sheet Cleaner] 清洗（各 Sheet 独立实现）
  → List[CanonicalItem]  规范格式
  → [Unified Converter] 规范格式 → DataSetItemData（唯一实现）
  → ItemWriter.write_item()
  → Langfuse
```

### 3.3 规范格式（CanonicalItem）

定义统一的中间结构，所有 Sheet 清洗后必须输出此格式：

```python
@dataclass
class CanonicalItem:
    """清洗后的规范格式，所有 Sheet 清洗器统一输出"""
    current_msg: str                          # 当前用户输入
    history_messages: List[Dict[str, str]]    # [{"type":"human","content":"..."},{"type":"ai","content":"..."}, ...]
    response_message: str                    # 期望的 AI 回复
    message_id: Optional[str] = None
    patient_id: Optional[str] = None
    context: Dict[str, Any] = field(default_factory=dict)   # age, disease, symptom 等
    ext: Optional[str] = None
```

特点：

- 字段语义清晰，与 Input/Output Schema 一一对应
- `history_messages` 已是最终形态，无需在转换层再解析 Q/A
- 各 Sheet 清洗器负责把原始数据（含 Q/A 文本）转成 `history_messages`

### 3.4 职责划分

| 层级 | 职责 | 实现位置 | 修改频率 |
|------|------|----------|----------|
| Sheet Cleaner | 从 Excel 行提取、清洗，输出 CanonicalItem | 各 Sheet 专属模块 | 新增/调整 Sheet 时 |
| Unified Converter | CanonicalItem → DataSetItemData | 单一转换器 | 仅当 Schema 变更 |
| 通用工具 | 空值判断（含 "-"）、类型转换等 | base 或 utils | 通用规则变更时 |

### 3.5 与 020303 的对比

| 项目 | 020303 方案 | 分层架构 |
|------|-------------|----------|
| parse_history_qa_to_messages | 放在 base.py | 放在「患者无数据+历史会话+历史Action」的 Cleaner 内 |
| 历史会话 Q/A 解析 | 通过新 Parser 调用 base 函数 | Cleaner 内部完成，输出已是 history_messages |
| 空字符 "-" | 修改 base 的 convert_* | 修改通用工具函数，Cleaner 复用 |
| 扩展新 Sheet | 新 Parser + 可能改 base | 新 Cleaner，输出 CanonicalItem，不改转换器 |

## 四、实现方案

### 4.1 目录结构建议

```
parsers/
├── base.py                    # DataSetItemData、通用工具（convert_to_text、_is_empty_like 等）
├── canonical.py               # CanonicalItem 定义、canonical_to_dataset_item 转换器
├── chain.py                   # 可选：若仍需对 CanonicalItem 做链式处理
└── cleaners/                  # 各 Sheet 清洗器
    ├── __init__.py
    ├── base.py                # Cleaner 接口：clean(row, df) -> List[CanonicalItem]
    ├── lsk.py                 # 4.1 lsk 清洗器
    ├── sh1128.py              # sh1128 通用清洗器（父类，列映射、空行判断等基础逻辑）
    ├── sh1128_multi.py        # 继承 Sh1128Cleaner，重写 clean()：常见问题及单轮（一行→多 Item）
    └── sh1128_history_qa.py   # 继承 Sh1128Cleaner，重写 clean()：患者无数据+历史会话+历史Action（历史会话 Q/A 解析）
```

**继承关系**：`Sh1128Cleaner` 作为 sh1128 系列父类，提供列映射（会话输入、供应商响应()、历史会话、message_id 等）、空行判断、基础字段提取。`Sh1128MultiCleaner`、`Sh1128HistoryQACleaner` 继承父类，仅重写与具体 Sheet 相关的逻辑（如 `clean()` 或 `_parse_history()`），复用父类公共能力。

#### 4.1.1 Sh1128 系列父子类设计

| 类 | 继承 | 重写方法 | 定制逻辑 |
|----|------|----------|----------|
| Sh1128Cleaner | BaseSheetCleaner | — | 1 行→1 Item，history_session/history_response 直接作为单条 human/ai |
| Sh1128MultiCleaner | Sh1128Cleaner | `clean()` | 1 行→多 Item，会话输入/供应商响应 按 Q/A 拆分，逐轮产出 CanonicalItem |
| Sh1128HistoryQACleaner | Sh1128Cleaner | `clean()` 或 `_parse_history_messages()` | 历史会话字段为 Q/A 格式，解析为多轮 history_messages 后产出 1 个 CanonicalItem |

子类通过重写 `clean()` 实现 Sheet 专属逻辑；若仅历史解析不同，可抽 `_parse_history_messages(raw: str) -> List[Dict]` 供子类重写，父类 `clean()` 调用该方法组装 `history_messages`。

### 4.2 Cleaner 接口

```python
class BaseSheetCleaner(ABC):
    """Sheet 清洗器基类：Excel 行 → List[CanonicalItem]"""

    @abstractmethod
    def clean(self, row: pd.Series, df: pd.DataFrame) -> List[CanonicalItem]:
        """清洗单行，返回 0 个或多个 CanonicalItem"""
        pass

    @abstractmethod
    def is_empty_row(self, row: pd.Series, df: pd.DataFrame) -> bool:
        """是否为空行（跳过）"""
        pass
```

### 4.3 主流程改造

```python
# import_feishu_excel.py 主循环
cleaner = get_cleaner_by_type(parser_type)  # 替代 get_parser_by_type
converter = CanonicalToDatasetItemConverter()  # 单一转换器，无 parser 依赖

for idx, row in df.iterrows():
    if cleaner.is_empty_row(row, df):
        sheet_stats["skipped"] += 1
        continue
    canonical_items = cleaner.clean(row, df)
    for item in canonical_items:
        dataset_item = converter.convert(item)
        writer.write_item(dataset_name, dataset_item)
        sheet_stats["success"] += 1
```

### 4.4 迁移策略

| 阶段 | 内容 |
|------|------|
| 1 | 定义 CanonicalItem、实现 canonical_to_dataset_item |
| 2 | 实现 BaseSheetCleaner，将现有 LskParser、Sh1128Parser 逻辑迁移为 LskCleaner、Sh1128Cleaner（父类） |
| 3 | 实现 Sh1128MultiCleaner、Sh1128HistoryQACleaner（继承 Sh1128Cleaner，重写 clean 或 _parse_history_messages） |
| 4 | 主流程切换为 Cleaner + Converter |
| 5 | 废弃或保留 Parser 作为兼容层（可选） |

## 五、设计模式局限性总结

| 局限性 | 表现 | 分层架构的应对 |
|--------|------|----------------|
| 清洗与转换耦合 | Parser 既清洗又转换，base 含 Sheet 专属逻辑 | 清洗在 Cleaner，转换在 Converter，base 仅通用工具 |
| row_data 过载 | 万能 Dict，多分支处理 | CanonicalItem 固定结构，语义清晰 |
| 扩展成本高 | 新 Sheet 需改 base 或重写多方法 | 新 Sheet = 新 Cleaner，输出 CanonicalItem |
| 通用与专属混淆 | "-" 与 Q/A 解析混在一起 | 通用规则在工具函数，专属逻辑在 Cleaner |

---

## 六、与 020303 的衔接

- **空字符 "-" 兼容**：在通用工具（如 `convert_to_text`、`_is_empty_like`）中实现，所有 Cleaner 复用。
- **历史会话 Q/A 解析**：在 `Sh1128HistoryQACleaner` 内实现 `parse_history_qa_to_messages`（或等价逻辑），输出 `CanonicalItem.history_messages`，**不再放入 base**。
- **020303 中的 Sh1128HistoryQAParser**：可对应为 Sh1128HistoryQACleaner，职责仅为「清洗 → CanonicalItem」，不涉及 base 的 to_dataset_item 或 parse 函数。
