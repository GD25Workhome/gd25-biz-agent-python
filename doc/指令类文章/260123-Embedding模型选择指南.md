# Embedding模型选择指南：输入长度限制与向量维度考虑

## 一、问题背景

在RAG检索方案开发过程中，发现embedding模型的选择需要考虑两个关键因素：
1. **输入token长度限制**：不同模型对输入文本的最大长度有严格限制
2. **向量维度**：影响存储空间、检索性能和模型能力

本文档总结了在实际项目开发中如何正确选择和使用embedding模型。

---

## 二、输入Token长度限制：关键约束

### 2.1 问题发现

在构建embedding文本时，我们设计了包含完整信息的格式：
```
场景描述：{场景摘要（2-3句话）}
用户输入：{user_input}
医生回复：{agent_response}
关键特征：疾病={disease}，血压评价={bp_level}，症状={symptoms}，预警={alert}，趋势={trend}
上下文：{历史数据、用药等信息}
```

**问题**：这种格式的文本长度可能达到：
- 最短：约260字 ≈ 390-520 tokens
- 最长：约900字 ≈ 1350-1800 tokens

而常用的embedding模型（如m3e-base）的**最大输入长度仅为512 tokens**，会导致文本被截断，丢失重要信息。

### 2.2 常见模型的Token限制

| 模型 | 最大Token长度 | 维度 | 说明 |
|------|-------------|------|------|
| **moka-ai/m3e-base** | 512 | 768 | 当前项目使用，中文优化 |
| **moka-ai/m3e-large** | 512 | 768 | 性能更好，但长度限制相同 |
| **text2vec-bge-large-chinese** | 256 | 1024 | 长度限制更严格 |
| **BGE-large-zh** | 512 | 1024 | 性能好但长度限制 |
| **Jina-embeddings-v2-base-zh** | 8,192 | 768 | 支持长文本 |
| **Jina-embeddings-v3** | 8,192 | 1024（可调） | 支持长文本，性能更好 |
| **tao-8k** | 8,192 | 1024 | 中文优化，长文本支持 |
| **Doubao-embedding** | 4,096 | 2,048 | 在线API，支持长文本 |
| **Doubao-embedding-large** | 128,000 | 4,096（可降维） | 在线API，超长文本支持 |

### 2.3 长度限制的影响

**文本被截断的后果**：
1. **信息丢失**：超出限制的部分被丢弃，可能丢失关键信息
2. **语义不完整**：截断后的文本可能无法完整表达原始语义
3. **检索效果下降**：不完整的embedding导致检索准确率降低

**实际案例**：
- 如果医生回复有500字，加上其他字段，总长度可能超过512 tokens
- 使用m3e-base时，后面的内容会被截断
- 检索时可能无法匹配到完整的案例语义

## 三、解决方案对比

### 3.1 方案一：精简Embedding文本（推荐用于短文本模型）

**策略**：只保留核心语义信息，去掉冗余内容

**精简格式**：
```
场景：{场景摘要（1句话，20-30字）}
用户输入：{user_input}
特征：疾病={disease}，血压={bp_level}，症状={symptoms}，预警={alert}
```

**优点**：
- ✅ 文本长度控制在200-300字以内（约300-450 tokens，在512限制内）
- ✅ 保留核心语义信息
- ✅ 无需更换模型，无需修改数据库schema
- ✅ 完整内容仍存储在数据库其他字段中

**缺点**：
- ⚠️ 需要调整embedding文本构建逻辑
- ⚠️ 可能丢失部分细节信息（但核心语义保留）

### 3.2 方案二：使用长文本模型（推荐用于完整信息保留）

#### 2.1 在线API方案：Doubao-embedding

**规格**：
- 最大tokens：4,096（doubao-embedding）或 128,000（doubao-embedding-large）
- 维度：2,048（可降维到512/1024）
- 支持中文

**优点**：
- ✅ 无需本地GPU，部署简单
- ✅ 支持长文本（4K-128K tokens）
- ✅ 无需管理模型文件

**缺点**：
- ⚠️ 需要API密钥和费用
- ⚠️ 需要网络连接（有延迟）
- ⚠️ 需要修改代码（API调用）
- ⚠️ 需要修改数据库schema（768 → 2048维）

**适用场景**：
- 没有GPU或不想本地部署
- 需要快速上线
- 可以接受API调用成本

#### 2.2 本地模型方案：Jina-embeddings-v2-base-zh（推荐）

**规格**：
- 最大tokens：8,192
- 维度：768（与当前数据库schema兼容）
- 参数量：~161M
- 显存需求：4-8 GB（GPU）或 8-12 GB（CPU）

**优点**：
- ✅ 维度768，无需修改数据库schema
- ✅ 支持8K tokens，完全满足需求
- ✅ 中文优化，性能好
- ✅ 本地部署，无API成本
- ✅ 社区支持好，文档完善

**缺点**：
- ⚠️ 需要GPU（推荐）或足够内存（CPU推理）
- ⚠️ CPU推理速度较慢（2-5秒/次）

**硬件要求**：
- **GPU方案**：RTX 3060 12GB 或更高（推荐8-12GB显存）
- **CPU方案**：Mac M1 32G 或同等配置（推理速度较慢）

**适用场景**：
- 有GPU或足够内存
- 需要保留完整embedding文本格式
- 希望本地部署，无API成本

## 四、向量维度考虑

### 4.1 维度对性能的影响

| 维度 | 存储空间（每条） | 检索速度 | 语义表达能力 | 适用场景 |
|------|----------------|---------|-------------|---------|
| **256** | ~1 KB | 快 | 较低 | 简单场景，大规模数据 |
| **512** | ~2 KB | 较快 | 中等 | 一般检索任务 |
| **768** | ~3 KB | 中等 | 良好 | **推荐**，平衡性能与效果 |
| **1024** | ~4 KB | 较慢 | 很好 | 复杂语义任务 |
| **2048** | ~8 KB | 慢 | 优秀 | 高精度需求 |

### 4.2 维度选择原则

**1. 与数据库Schema匹配**
- 如果数据库已定义为768维，优先选择768维模型
- 修改维度需要数据库迁移，成本较高

**2. 任务复杂度**
- **简单任务**（关键词匹配、短文本相似度）：256-512维足够
- **中等任务**（语义检索、文档相似度）：768维推荐
- **复杂任务**（细粒度语义区分、多语言）：1024-2048维

**3. 数据规模**
- **小规模**（<10万条）：可以使用高维度（1024+）
- **中等规模**（10万-100万）：推荐768维
- **大规模**（>100万）：考虑512-768维，平衡效果与性能


**4. 硬件资源**
- **GPU显存有限**：选择较低维度模型
- **存储空间有限**：选择较低维度模型
- **计算资源充足**：可以选择高维度模型

### 4.3 当前项目建议

**推荐维度：768**

**理由**：
1. ✅ 与当前数据库schema匹配（无需迁移）
2. ✅ 对于医疗对话RAG检索，768维已足够表达语义
3. ✅ 存储和检索性能平衡
4. ✅ 常见开源模型标准维度（m3e-base、Jina-v2-base-zh等）

**何时需要更高维度**：
- 数据量很大（百万级以上）
- 语义区分度要求极高
- 需要捕获更细粒度的语义差异

## 五、模型选择决策树

```
开始选择Embedding模型
    │
    ├─ 文本长度需求
    │   ├─ < 512 tokens → 使用 m3e-base（当前模型）
    │   ├─ 512-4000 tokens → 考虑精简文本 或 Doubao-embedding API
    │   └─ > 4000 tokens → 必须使用长文本模型
    │
    ├─ 部署方式
    │   ├─ 本地部署
    │   │   ├─ 有GPU（8GB+）→ Jina-v2-base-zh（推荐）
    │   │   └─ 无GPU，有内存（32GB+）→ Jina-v2-base-zh（CPU推理）
    │   └─ 在线API
    │       └─ Doubao-embedding API
    │
    ├─ 维度要求
    │   ├─ 数据库已定义768维 → 选择768维模型（Jina-v2-base-zh）
    │   ├─ 可以修改数据库 → 可选择1024-2048维模型
    │   └─ 大规模数据 → 优先考虑768维，平衡性能
    │
    └─ 成本考虑
        ├─ 预算有限 → 本地模型（一次性成本）
        └─ 快速上线 → 在线API（按量付费）
```

## 六、实际项目建议

### 6.1 当前项目推荐方案

**方案A：保持m3e-base + 精简文本（快速方案）**
- 优点：无需改动模型和数据库
- 缺点：需要调整embedding文本格式
- 适用：快速验证，文本可以精简

**方案B：升级到Jina-v2-base-zh（推荐方案）**
- 优点：支持长文本，维度匹配，效果更好
- 缺点：需要本地部署，需要GPU或足够内存
- 适用：有硬件资源，需要保留完整文本格式

**方案C：使用Doubao-embedding API（备选方案）**
- 优点：部署简单，支持长文本
- 缺点：需要API成本，需要修改代码和数据库
- 适用：无硬件资源，快速上线

### 6.2 实施步骤

**如果选择Jina-v2-base-zh**：

1. **下载模型**：
   ```python
   from sentence_transformers import SentenceTransformer
   model = SentenceTransformer('jinaai/jina-embeddings-v2-base-zh')
   ```

2. **修改代码**：
   - 更新 `EmbeddingModelCache` 中的模型名称
   - 确保 `trust_remote_code=True`（如果需要）

3. **测试验证**：
   - 使用实际文本测试长度限制
   - 验证embedding质量和检索效果

4. **性能优化**（Mac M1）：
   - 考虑使用ONNX后端加速
   - 控制批量大小（batch_size ≤ 4）
   - 限制最大长度（如果不需要完整8K）

## 七、关键要点总结

### 7.1 必须注意的事项

1. **⚠️ 输入长度限制是硬约束**
   - 必须评估实际文本长度
   - 超过限制会导致信息丢失
   - 选择模型时优先考虑长度支持

2. **⚠️ 维度与数据库Schema匹配**
   - 修改维度需要数据库迁移
   - 优先选择与现有schema匹配的模型
   - 768维是常见标准，兼容性好

3. **⚠️ 存储格式 vs 查询格式**
   - 存储时可以包含完整信息（包括回复）
   - 查询时只包含可用信息（无回复）
   - 这是合理的RAG设计模式

### 7.2 最佳实践

1. **文本长度评估**：在设计embedding文本格式时，先估算长度
2. **模型选择**：根据文本长度需求选择合适模型
3. **维度匹配**：优先选择与数据库schema匹配的维度
4. **性能测试**：在实际数据上测试模型效果
5. **渐进优化**：先使用简单方案验证，再逐步优化
