# 向量查询原理讲解：query_text 与 keywords 双参数设计

## 概述

本文档详细讲解 RAG 向量检索中 `query_text` 和 `keywords` 两个查询参数的设计原理、使用场景以及底层实现机制。

## 一、为什么需要两个查询条件？

### 1.1 设计背景

在 RAG（Retrieval-Augmented Generation）系统中，向量检索是核心环节。为了提高检索的准确性和相关性，系统采用了**双参数查询设计**：

- **`query_text`**：语义查询文本（来自上游节点）
- **`keywords`**：关键词列表（可选，用于增强检索）

### 1.2 两个参数的作用差异

#### `query_text` - 语义查询文本

**特点**：
- 包含完整的语义信息
- 通常来自上游节点（如意图识别节点）的输出
- 是用户原始查询的语义化表示
- 可能经过 LLM 处理，包含上下文信息

**示例**：
```python
query_text = "患者询问血压测量方法和注意事项"
```

#### `keywords` - 关键词列表

**特点**：
- 提取的关键实体、概念、术语
- 用于补充和增强语义查询
- 可以包含领域特定的重要词汇
- 帮助向量模型更好地理解查询意图

**示例**：
```python
keywords = ["血压", "测量", "注意事项", "健康管理"]
```

### 1.3 为什么需要两者结合？

1. **语义完整性**：`query_text` 提供完整的语义上下文
2. **关键信息强化**：`keywords` 突出重要实体和概念
3. **检索精度提升**：两者结合可以同时利用语义相似度和关键词匹配的优势
4. **灵活性**：`keywords` 是可选的，可以根据场景动态调整

## 二、实际语法原理

### 2.1 查询文本增强

在 `vector_db_search` 函数中，两个参数会被合并成一个增强的查询文本：

```python
# 代码位置：backend/infrastructure/rag/retrieval.py:229-233

# 关键词增强（可选）
if keywords:
    enhanced_query = f"{query_text} {' '.join(keywords)}"
else:
    enhanced_query = query_text
```

**处理逻辑**：
- 如果提供了 `keywords`，将其与 `query_text` 用空格拼接
- 如果未提供 `keywords`，直接使用 `query_text`

**示例**：
```python
query_text = "患者询问血压测量方法"
keywords = ["血压", "测量", "注意事项"]

# 增强后的查询文本
enhanced_query = "患者询问血压测量方法 血压 测量 注意事项"
```

### 2.2 向量化过程

增强后的查询文本会被转换为向量（embedding）：

```python
# 代码位置：backend/infrastructure/rag/retrieval.py:235-237

# 向量化
embedding_cache = EmbeddingModelCache()
query_embedding = np.array(embedding_cache.text_to_embedding(enhanced_query))
```

**技术细节**：
- **模型**：使用 `moka-ai/m3e-base` 中文语义向量模型
- **维度**：生成 768 维向量
- **归一化**：使用 `normalize_embeddings=True` 进行归一化处理
- **格式**：转换为 numpy 数组

**向量化模型配置**：
```python
# 代码位置：backend/infrastructure/rag/data_import.py:52

_model_name: str = "moka-ai/m3e-base"
```

### 2.3 向量相似度查询

向量化后的查询向量会在 PostgreSQL + pgvector 数据库中执行相似度搜索：

```python
# 代码位置：backend/infrastructure/rag/retrieval.py:54-67

sql = f"""
    SELECT 
        id,
        user_input,
        agent_response,
        tags,
        quality_grade,
        1 - (embedding <=> %s::vector) AS similarity_score,
        %s AS source
    FROM {full_table_name}
    WHERE 1 - (embedding <=> %s::vector) >= %s
    ORDER BY embedding <=> %s::vector
    LIMIT %s
"""
```

#### pgvector 操作符详解

**`<=>` 操作符**：
- **名称**：余弦距离操作符（Cosine Distance Operator）
- **语法**：`vector1 <=> vector2`
- **返回值**：0 到 2 之间的浮点数
  - `0`：完全相似（向量方向相同）
  - `1`：正交（向量垂直）
  - `2`：完全相反（向量方向相反）

**相似度计算**：
```sql
-- 余弦距离转换为相似度分数
1 - (embedding <=> query_vector) AS similarity_score
```

**转换逻辑**：
- 余弦距离越小 → 相似度越高
- `similarity_score = 1 - cosine_distance`
- 相似度分数范围：`-1` 到 `1`（归一化向量情况下为 `0` 到 `1`）

#### SQL 查询执行流程

1. **WHERE 子句**：过滤相似度低于阈值的结果
   ```sql
   WHERE 1 - (embedding <=> %s::vector) >= %s
   ```
   - `%s`（第一个）：查询向量
   - `%s`（第二个）：相似度阈值（默认 0.7）

2. **ORDER BY 子句**：按相似度排序（升序，距离越小越相似）
   ```sql
   ORDER BY embedding <=> %s::vector
   ```

3. **LIMIT 子句**：限制返回结果数量
   ```sql
   LIMIT %s
   ```

### 2.4 多表检索与结果合并

系统支持在多个表中并行检索，然后合并结果：

### 2.5 降级策略

如果检索结果不足，系统会自动降低相似度阈值重试：

```python
# 代码位置：backend/infrastructure/rag/retrieval.py:156-193

def search_with_fallback(
    query_embedding: np.ndarray,
    ...
) -> List[Dict]:
    """带降级的检索（如果结果不足，降低阈值重试）"""
    thresholds = [0.7, 0.6, 0.5]  # 降级阈值列表
    
    for threshold in thresholds:
        results = multi_table_search(..., similarity_threshold=threshold)
        if len(results) >= min_results:
            return results
    
    return results  # 返回已有结果（即使数量不足）
```

## 三、完整执行流程

### 3.1 调用链路

```
RetrievalNode.execute()
    ↓
vector_db_search(query_text, keywords, ...)
    ↓
[1] 查询文本增强：enhanced_query = query_text + keywords
    ↓
[2] 向量化：query_embedding = embedding_model.encode(enhanced_query)
    ↓
[3] 多表检索：search_with_fallback(query_embedding, ...)
    ↓
[4] 相似度过滤与排序：SQL 查询（使用 pgvector <=> 操作符）
    ↓
[5] 结果合并与排序：按相似度降序排列
    ↓
[6] 返回 Top-K 结果
```

## 四、技术要点总结

### 4.1 核心优势

1. **语义理解**：通过向量化实现语义级别的相似度匹配
2. **关键词强化**：通过 `keywords` 参数突出重要实体和概念
3. **多表检索**：支持在多个表中并行检索，提高覆盖率
4. **智能降级**：自动降低阈值确保返回足够结果
5. **性能优化**：使用 pgvector 索引加速向量查询

### 4.2 关键技术

| 技术 | 说明 |
|------|------|
| **向量模型** | moka-ai/m3e-base（中文语义向量模型） |
| **向量维度** | 768 维 |
| **相似度度量** | 余弦相似度（Cosine Similarity） |
| **数据库** | PostgreSQL + pgvector 扩展 |
| **查询操作符** | `<=>`（余弦距离操作符） |
| **归一化** | L2 归一化（normalize_embeddings=True） |

### 4.3 参数设计原则

1. **`query_text` 必填**：提供完整的语义上下文
2. **`keywords` 可选**：根据场景动态增强
3. **拼接策略**：简单空格拼接，保持语义连贯性
4. **向量化统一**：两个参数合并后统一向量化，保证一致性

## 五、最佳实践

### 5.1 何时使用 keywords？

**推荐使用场景**：
- 包含重要领域术语的查询
- 需要强调特定实体的场景
- 意图识别节点提取了关键实体

**示例**：
```python
query_text = "患者询问血压相关问题"
keywords = ["血压", "高血压", "测量"]  # 强化关键概念
```

### 5.2 参数调优建议

1. **相似度阈值**：
   - 默认 0.7，可根据业务需求调整
   - 阈值越高，结果越精确但可能数量不足
   - 阈值越低，结果越多但可能包含不相关内容

2. **Top-K 设置**：
   - 默认 15，可根据下游节点需求调整
   - 过多结果可能增加 LLM 上下文长度
   - 过少结果可能无法覆盖所有相关场景

3. **关键词选择**：
   - 选择领域核心术语
   - 避免过于通用的词汇
   - 保持关键词数量适中（3-5个）

## 六、常见问题

### Q1: 为什么不能只用 query_text 或只用 keywords？

**A**: 
- 只用 `query_text`：可能丢失关键实体信息，导致检索不准确
- 只用 `keywords`：缺乏语义上下文，无法理解完整意图
- 两者结合：既保证语义完整性，又强化关键信息

### Q2: keywords 的顺序会影响结果吗？

**A**: 不会。因为：
1. 关键词会被拼接成字符串：`' '.join(keywords)`
2. 整个字符串统一向量化
3. 向量模型会理解整体语义，不依赖顺序

### Q3: 为什么使用余弦相似度而不是欧氏距离？

**A**: 
- **余弦相似度**：关注向量方向（语义相似性），适合文本向量
- **欧氏距离**：关注向量距离（数值差异），更适合数值型数据
- 文本语义匹配更关注"意思是否相似"，而非"数值是否接近"

### Q4: 向量维度为什么是 768？

**A**: 
- 这是 `moka-ai/m3e-base` 模型的输出维度
- 768 维是 BERT 系列模型的标准维度
- 维度越高，表达能力越强，但计算成本也越高
- 768 维在精度和性能之间取得了良好平衡
