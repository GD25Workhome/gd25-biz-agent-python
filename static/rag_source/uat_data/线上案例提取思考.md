# 线上案例提取思考：RAG检索方案技术可行性分析

## 一、问题背景

### 1.1 业务需求
- **目标**：将线上Agent的标准聊天案例整理成few-shot模板，存入向量库
- **应用场景**：在`retrieval_node`节点根据上一个节点（`before_rag`）的特征提取，召回匹配度高的案例
- **预期效果**：提高`after_record_agent`节点的点评准确度

### 1.2 技术架构现状
- **向量库**：PostgreSQL + pgvector
- **Embedding模型**：moka-ai/m3e-base（768维）
- **当前embedding策略**：`user_input + agent_response` 的组合
- **检索流程**：
  1. `before_rag`节点：优化问题表达，提取`query_text`、`keywords`、`disease`
  2. `retrieval_node_v2`：使用`query_text + keywords`进行向量检索
  3. `after_record_agent`：接收检索到的案例，生成点评回复

### 1.3 核心难点
1. **难点一**：如何构建embedding内容，让召回更准确？
   - 是否要包含疾病、血压趋势、症状等结构化信息？
   - 这些信息是否会提升检索效果？

2. **难点二**：如何提取这些特征信息？
   - 人工标注 vs 自动提取
   - 提取哪些特征最有效？

## 三、Embedding策略设计建议

### 3.1 核心原则

**关键洞察**：向量检索的本质是**语义相似度匹配**，不是精确字段匹配。

### 3.2 推荐的Embedding策略

#### 策略一：结构化信息增强（推荐）⭐⭐⭐⭐⭐

**原理**：将关键特征转换为自然语言描述，融入embedding文本

**构建embedding文本的格式**：
```
[场景描述] 用户输入：{user_input} 医生回复：{agent_response}
[关键特征] 疾病：{disease}，血压评价：{blood_pressure_level}，症状：{symptoms}，是否有预警：{has_alert}
[上下文信息] 历史数据：{has_history}，用药情况：{medication_status}
```

**优点**：
- ✅ 保留了语义信息，向量模型能理解这些特征
- ✅ 特征信息自然融入，不会破坏语义连贯性
- ✅ 检索时能匹配到相似场景的案例

**示例**：
```
[场景描述] 用户输入：我今天的血压，高压129，低压70，脉搏：72 医生回复：你能主动监测并告诉我血压情况，这个习惯非常棒！已经帮你记录了这次的血压数据。你的血压值129/70mmHg，略高于正常范围...
[关键特征] 疾病：高血压，血压评价：轻度偏高，症状：无，是否有预警：否
[上下文信息] 历史数据：无，用药情况：未提及
```

#### 策略二：分层Embedding（备选）⭐⭐⭐⭐

**原理**：为不同类型的特征建立不同的embedding字段

**方案**：
- **主embedding**：`user_input + agent_response`（用于语义检索）
- **辅助metadata**：疾病、血压级别等存储在`tags`字段（用于后过滤）

**优点**：
- ✅ 检索灵活，可以先用语义检索，再用metadata过滤
- ✅ 可扩展性强，后续可以添加更多metadata

**缺点**：
- ⚠️ 需要修改检索逻辑，支持metadata过滤
- ⚠️ 实现复杂度更高

### 3.3 不建议的策略

#### ❌ 策略三：纯结构化字段embedding

**不建议原因**：
- 向量模型更适合处理自然语言文本，不适合结构化数据
- 如："高血压|轻度偏高|无症状" 这样的字段组合，语义信息丢失严重
- 检索时难以匹配到相似语义的场景

## 四、特征提取方案设计

### 4.1 推荐方案：**半自动提取 + 人工校验**⭐⭐⭐⭐⭐

#### 4.1.1 提取流程

**第一阶段：自动提取（使用LLM）**
- 利用`before_rag`节点的能力，对案例进行批量特征提取
- 提取内容：
  - 疾病类型（高血压/冠心病/糖尿病等）
  - 血压评价级别（达标/轻度偏高/中度偏高/重度偏高/低压偏低/高压偏低）
  - 症状列表（头晕/胸闷/胸痛等）
  - 是否有预警（是/否）
  - 是否包含趋势分析（是/否）
  - 是否包含排班推送（是/否）
  - 是否包含文章推送（是/否）
  - 回复风格标签（鼓励型/警示型/中性等）

**第二阶段：人工校验和标注**
- 人工校验自动提取的准确性
- 补充难以自动提取的特征：
  - 回复质量等级（优秀/良好/一般）
  - 场景类型（首次记录/常规记录/异常记录等）
  - 特殊标签（特殊情况、特殊话术等）

### 4.2 备选方案对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **人工标注** | 准确性高，质量可控 | 成本高，耗时 | ⭐⭐⭐ |
| **自动提取** | 快速，成本低 | 准确性可能不足 | ⭐⭐⭐⭐ |
| **半自动+人工校验** | 平衡效率和准确性 | 需要两轮工作 | ⭐⭐⭐⭐⭐ |

## 五、具体实施方案

### 5.1 Embedding文本构建规范

#### 标准格式
```
场景描述：{场景摘要（2-3句话）}
用户输入：{user_input}
医生回复：{agent_response}
关键特征：疾病={disease}，血压评价={bp_level}，症状={symptoms}，预警={alert}，趋势={trend}
上下文：{历史数据、用药等信息}
```

#### 示例1：首次记录，血压轻度偏高
```
场景描述：患者首次记录血压数据，血压值轻度偏高，无异常症状，医生给予正向反馈和监测建议。
用户输入：我今天的血压，高压129，低压70，脉搏72
医生回复：你能主动监测并告诉我血压情况，这个习惯非常棒！已经帮你记录了这次的血压数据。你的血压值129/70mmHg略高于正常范围...
关键特征：疾病=高血压，血压评价=轻度偏高，症状=无，预警=否，趋势=无
上下文：首次记录，无历史数据，用药=未提及
```

#### 示例2：重度偏高，有症状，需要预警
```
场景描述：患者记录血压值重度偏高，伴有头晕症状，医生触发预警，建议尽快就医。
用户输入：今天量血压，高压180，低压110，有点头晕
医生回复：您本次测量的血压过高，且伴有不适症状，建议您尽快到医院就诊...
关键特征：疾病=高血压，血压评价=重度偏高，症状=头晕，预警=是，趋势=未分析
上下文：有症状，用药=未提及
```

### 5.2 特征提取字段清单
#### 必需字段（自动提取）
- **疾病类型**：高血压、冠心病、糖尿病、高脂血症等
- **当前血压评价级别**：达标、轻度偏高、中度偏高、重度偏高、低压偏低、高压偏低
- **症状列表**：头晕、胸闷、胸痛、恶心等（数组）
- **是否有预警**：是/否
- **场景类型**：首次记录、常规记录、异常记录、澄清询问

#### 推荐字段（自动提取+人工校验）
6. **回复风格**：鼓励型、警示型、中性、引导型
7. **是否包含趋势分析**：是/否
8. **是否包含排班推送**：是/否
9. **是否包含文章推送**：是/否
10. **质量等级**：优秀、良好、一般（需要人工评估）

### 5.3 数据导入流程

#### 步骤1：数据预处理
1. 从Excel文件读取案例数据
2. 清洗和验证数据（检查必需字段）
3. 使用LLM批量提取特征（调用特征提取API）

#### 步骤2：特征构建
1. 根据提取的特征，构建embedding文本（使用标准格式）
2. 将特征信息存储到`tags`字段（便于后续过滤和统计）

#### 步骤3：向量化和导入
1. 使用moka-ai/m3e-base模型生成embedding
2. 批量导入到`record_examples`表
3. 验证导入结果

### 5.4 检索优化建议

#### 当前检索逻辑（已实现）
```python
# retrieval_node_v2.py
enhanced_query = f"{query_text} {' '.join(keywords)}"
query_embedding = embedding_cache.text_to_embedding(enhanced_query)
```

#### 优化建议
1. **利用疾病信息**：在`before_rag`节点已经提取了`disease`，可以考虑：
   - 在检索前先用`disease`过滤相关表（如果表中有疾病标签）
   - 或者在embedding查询中加入疾病信息：`enhanced_query = f"{query_text} {disease} {' '.join(keywords)}"`

2. **相似度阈值调优**：
   - 当前阈值：0.7 → 0.6 → 0.5（降级策略）
   - 建议：根据实际召回效果调整，可能需要提高到0.75或降低到0.65

3. **Top-K调整**：
   - 当前：top_k=15
   - 建议：根据下游节点的需求调整，few-shot通常3-5个示例即可

## 六、技术风险与应对

### 6.1 风险点

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| **特征提取不准确** | 导致embedding偏差，检索效果差 | 人工校验关键案例，建立校验流程 |
| **embedding文本过长** | 可能影响模型效果 | 控制文本长度，重点突出关键特征 |
| **案例质量不一致** | few-shot效果差 | 建立质量评估机制，只导入高质量案例 |
| **检索结果不相关** | 下游节点收到无关案例 | 调整相似度阈值，优化embedding策略 |

### 6.2 验证方案

1. **小规模验证**
   - 先导入50-100个案例
   - 用实际查询测试召回效果
   - 评估下游节点生成的回复质量

2. **A/B测试**
   - 对比有无few-shot案例的效果
   - 收集用户反馈，评估改进效果

3. **持续优化**
   - 监控检索日志，分析召回案例的相关性
   - 根据实际效果调整embedding策略和特征提取

## 七、总结与建议

### 7.1 方案可行性结论

✅ **方案完全可行**，技术架构支持，实现难度中等。

### 7.2 核心建议

1. **Embedding策略**：推荐使用**结构化信息增强**方案
   - 将关键特征转换为自然语言描述
   - 融入embedding文本，保持语义连贯性
   - 格式：场景描述 + 用户输入 + 医生回复 + 关键特征 + 上下文

2. **特征提取方案**：推荐**半自动提取 + 人工校验**
   - 第一阶段：使用LLM批量自动提取特征
   - 第二阶段：人工校验关键字段，补充质量等级等主观评价
   - 平衡效率和准确性

3. **提取的关键特征**：
   - **必需**：疾病类型、血压评价级别、症状列表、是否有预警、场景类型
   - **推荐**：回复风格、是否包含趋势/排班/文章推送、质量等级
   - **可选**：特殊标签

4. **实施步骤**：
   1. 设计特征提取提示词
   2. 批量处理Excel案例数据
   3. 人工校验和补充标注
   4. 构建embedding文本
   5. 导入向量库
   6. 小规模验证效果
   7. 全量导入和持续优化

### 7.3 关键成功因素

1. **特征提取的准确性**：直接影响embedding质量
2. **embedding文本的质量**：决定检索效果
3. **案例质量**：few-shot的效果依赖于案例质量
4. **持续优化**：根据实际效果不断调整策略
