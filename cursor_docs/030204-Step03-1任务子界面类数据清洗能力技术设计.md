## 2. 执行逻辑说明

单行「执行」的流程为：**先将该 task 状态改为 pending，再将其添加到队列，由消费者统一运行**。与批量（按 job 整批入队）的差异在于：批量仅将 job 下**已是 pending** 的 task 入队（见 `batch_task_queue_service.enqueue_batch_by_job_id`，按 `status=STATUS_PENDING` 查询），**不会**先把 failed/success/running 等改为 pending；单行执行则对当前行 task **先置 pending 再入队**，故任意状态的 task 均可从界面触发执行。

| 步骤 | 说明（单行执行） |
|------|------------------|
| 1. 置 pending | 将指定 task 的 status 更新为 pending（与当前 status 无关）。 |
| 2. 入队 | 将对应 task 构造为 `BatchTaskQueueItem` 加入内存队列。 |
| 3. 消费者执行 | 现有消费者从队列取任务，按 job_type 调用执行模版（`BatchTaskExecuteService.run_one_batch_task`）。 |

**乐观锁与多次入队**：消费者在执行时已通过乐观锁做数据管理——执行模版中会调用 `update_status_to_running_if_pending(task_id)`，**仅当当前 status 为 pending 时**才更新为 running 并继续执行业务，否则直接跳过。因此同一 task 若被多次入队，队列中可能有多条待处理项，但**实际只会有一笔被成功执行**（先抢到乐观锁的那次），其余取到该 task 的消费者会因「非 pending」而跳过。无需在入队侧做额外去重。

## 3. 单行任务执行 — 功能需求

### 3.1 交互需求

| 项目 | 说明 |
|------|------|
| 行内入口 | 在表格操作列增加「执行」按钮（风格参考 Step01 行内「数据清洗」）；与 status 无关，任意行均可点击；点击后调用后端对该行 task 执行（置 pending 并入队），成功后刷新列表。 |
| 加载与防重复 | 行内执行时该行按钮 loading 或禁用，防重复点击。 |

### 3.2 后端能力需求

| 项目 | 说明 |
|------|------|
| 语义 | 「执行」= 将指定 task 的 status 置为 pending，并将其入队（若当前无按 task_id 入队接口，则需新增）。与 status 无关，任意状态的 task 均可触发。 |
| 入参 | 仅请求体中的 task_id；后端根据 task_id 查 task，将其置为 pending 并入队，返回 enqueued 数量。无需 job_id（避免过度设计）。 |

### 3.3 状态与幂等

- 与 **status 无关**：pending / running / success / failed 均可点击「执行」；后端统一将该 task 置为 pending 并入队，不做「已是 pending 或已在队中」的检查与跳过。
- 同一 task 执行后变为 pending 并入队，可再次被消费；消费者侧乐观锁保证实际只会有一笔被成功执行，入队侧不做额外去重。

## 4. 接口设计

### 4.1 单行任务执行接口

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/batch-jobs/tasks/execute` | 对**单条**任务触发执行（置 pending 并入队）；入参仅 task_id |

**请求体**（JSON）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| task_id | string | 是 | 要执行的任务 id |

**约束**：

- 入参仅 **task_id**，无路径参数；后端根据 task_id 查 task，将其置为 pending 并入队，不做「已在队中」等检查。
- task_id 对应记录必须存在，否则 404 或 400。

**响应**（200）：

```json
{
  "success": true,
  "message": "已入队执行",
  "enqueued": 1
}
```

- `enqueued`：本次入队数量（1 表示成功入队）。

### 4.2 后端实现要点

| 模块 | 职责 |
|------|------|
| `batch_task_repository` | 新增 `update_status_to_pending(task_id: str)`：根据 task_id 将该 task 的 status 更新为 pending（与当前 status 无关）；返回是否更新成功。 |
| `batch_task_queue_service` | 新增 `enqueue_task_by_id(task_id: str, session)`：根据 task_id 查询 task（含 job_type、version），构造 `BatchTaskQueueItem` 入队；返回 1 或 0。 |
| `batch_jobs` 路由 | 新增 `POST /tasks/execute`（无 job_id 路径）：校验 body.task_id；根据 task_id 查 task 存在后，调用 repository 置 pending → commit；调用 queue_service 入队；返回 enqueued。路由需注册在 `GET /{job_id}/tasks` 之前，以免 `tasks` 被当作 job_id。 |

**注意**：批量入队接口 `enqueue_batch_by_job_id` 只入队「已是 pending」的 task，不会改状态。本接口（单行执行）流程：先 **update status → pending**，commit 后再 **enqueue_task_by_id**。

## 5. 前端设计

### 5.1 组件与入口

| 位置 | 说明 |
|------|------|
| 文件 | `frontend/js/pipeline_batch_job_tasks.js` |
| 表格操作列 | 在现有列后增加「执行」按钮；与 status 无关，任意行均可点击。风格参考 Step01 行内「数据清洗」按钮。 |

### 5.2 行内执行逻辑

1. 点击行内「执行」→ 请求 `POST ${BATCH_JOBS_API_PREFIX}/tasks/execute`，body：`{ "task_id": scope.row.id }`。
2. 成功：`ElMessage.success('已入队执行')`，刷新列表。
3. 失败：`ElMessage.error(detail)`。
4. 执行中：该行「执行」按钮 loading 或禁用，防重复点击。

### 5.3 与 Step01 数据清洗的对比

| 项目 | Step01 数据清洗（行内） | Step03-1 任务执行 |
|------|------------------------|-------------------|
| 行内按钮文案 | 数据清洗 | 执行 |
| API | POST .../datasets/{id}/items/rewritten/execute | POST .../batch-jobs/tasks/execute |
| 请求体 | item_ids: [row.id] | task_id: row.id |
| 后端行为 | 创建 rewritten 批次与 init 记录，Worker 执行 flow | 将 task 置 pending 并入队，消费者执行 batch_task |
