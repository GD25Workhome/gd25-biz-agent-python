## 1. 概述

### 1.1 目标

在 Step01 阶段的数据项详情界面（`pipeline_dataset_items.js`）中，新增「数据清洗」入口，支持：

1. **批量清洗**：在「新建数据项」前新增「进行数据清洗」按钮，对当前查询条件下的所有数据项执行清洗
2. **单行清洗**：在表格每一行增加「数据清洗」按钮，对单条数据项执行清洗
3. **统一接口**：两个入口共用同一后端 API，支持「单行 ID 传入」或「查询参数传入」两种模式

### 1.2 设计原则

- 与现有 `data_cleaning` 路由、`pipeline_step2` 流程保持一致
- 复用 `DataSetsItemsRepository`、`FlowManager`、`insert_rewritten_data_func` 已有能力
- 模型供应商配置在 Demo 启动时已由 `main.py` 加载，无需在 service 内重复调用

## 3. 接口设计

### 3.1 请求定义

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/data-cleaning/datasets/{dataset_id}/items/rewritten/execute` | 执行数据清洗 |

**路径参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| dataset_id | string | 是 | 数据集 ID |

**请求体**（JSON）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| item_ids | string[] | 否 | 指定要清洗的数据项 ID 列表。若提供，则忽略 query_params |
| query_params | object | 否 | 筛选条件（不含分页），用于按条件查询后清洗。命中多少条，就执行多少条 |

**说明**：`max_concurrent`（最大并发数）为后端常量，默认 4，不开放给前端。

**query_params 结构**（仅筛选条件，不含分页）：

| 字段 | 类型 | 说明 |
|------|------|------|
| status | int | 1=激活，0=废弃 |
| unique_key | string | unique_key（包含） |
| source | string | source（包含） |
| keyword | string | 关键词（在 input/output/metadata 中搜索） |

**请求示例**：

```json
// 模式一：单条/多条指定 ID
{
  "item_ids": ["01ARZ3NDEKTSV4RRFFQ69G5FAV", "01ARZ3NDEKTSV4RRFFQ69G5FBW"]
}

// 模式二：按查询条件（命中多少执行多少）
{
  "query_params": {
    "status": 1,
    "unique_key": "",
    "source": "",
    "keyword": ""
  }
}
```

### 3.2 响应定义

**成功（200）**：

```json
{
  "success": true,
  "message": "数据清洗完成",
  "stats": {
    "total": 10,
    "success": 8,
    "failed": 2
  }
}
```

**失败（4xx/5xx）**：

```json
{
  "detail": "错误信息"
}
```

### 3.3 参数校验规则

| 场景 | 处理 |
|------|------|
| 未传 `item_ids` 且未传 `query_params` | 400：至少提供 item_ids 或 query_params |
| 传入 `item_ids` 且为空数组 | 400：item_ids 不能为空 |
| 传入 `query_params` 但查询结果为 0 条 | 200：返回 `total=0, success=0, failed=0` |
| `dataset_id` 不存在 | 404：数据集合不存在 |
| `item_ids` 中某 ID 不属于该 dataset | 校验通过，执行时跳过或记录失败（建议：仅清洗属于该 dataset 的 item，其余记录为 failed） |

## 4. 后端实现设计

### 4.1 模块职责划分

| 模块 | 职责 |
|------|------|
| `backend/app/api/routes/data_cleaning.py` | 新增 POST 路由，参数校验，调用 `rewritten_service` |
| `backend/app/api/schemas/data_cleaning.py` | 新增 `RewrittenExecuteRequest`、`RewrittenExecuteResponse` |
| `backend/pipeline/rewritten_service.py` | 核心逻辑：查询数据、构建 state、并发执行 flow |
| `backend/infrastructure/database/repository/data_sets_items_repository.py` | 新增 `get_by_ids` 方法（按 ID 列表查询） |

### 4.2 rewritten_service 核心流程

```
1. 加载流程图（FlowManager.get_flow("rewritten_data_service_agent")）
   - 注意：不调用 ProviderManager.load_providers()，Demo 启动时已加载

2. 解析入参，确定待清洗的 DataSetsItemsRecord 列表
   - 若 item_ids 非空：DataSetsItemsRepository.get_by_ids(dataset_id, item_ids)
   - 否则：DataSetsItemsRepository 按 query_params 查询，**命中多少条就返回多少条**（无分页）

3. 若 records 为空，直接返回 total=0, success=0, failed=0

4. 并行执行批次处理（参考 run_embedding_import_qa_parallel.run_batch_parallel）
   - 使用 asyncio.Semaphore 控制并发数（后端常量 MAX_CONCURRENT=4）
   - 对每条 record 调用 run_one(record, graph)
   - run_one：构建 state → graph.ainvoke

5. 统计 success/failed，返回 stats
```

### 4.3 State 构建约定

从 `DataSetsItemsRecord` 构建 `FlowState`，需映射到 `rewritten_agent.md` 提示词变量：

| 提示词变量 | 来源 | 说明 |
|------------|------|------|
| q_context | `item.metadata.content_info.user_info` | 发言相关上下文。若路径不存在则不填充 |
| current_message | item.input.current_msg | 当前患者发言 |
| history_messages | item.input.history_messages | 历史聊天记录 |
| manual_ext | `item.metadata.content_info.ext` 与 `item.metadata.content_info.user_info.response_tags` | 参考标签/人工标注，合并两者内容 |
| response_message | item.output.response_message | 回复案例 |
| response_rule | item.output.response_rule | 回复规则 |

**persistence_edges_var**：需包含 `source_dataset_id`、`source_item_id`，供 `insert_rewritten_data_func` 使用。这两个值**仅存储在 persistence_edges_var 中**，不放入 prompt_vars。

**注意**：`DataSetsItemsRecord` 的 `input`、`output`、`metadata` 结构取决于导入时的 cleaner 及 schema，需兼容字段缺失（空字符串、空列表、None）。上述路径若不存在，则不填充或使用空值。

### 4.4 与 run_embedding_import_qa 的差异

| 项目 | run_embedding_import_qa_parallel | rewritten_service |
|------|----------------------------------|-------------------|
| 模型供应商配置 | 脚本内显式调用 ProviderManager.load_providers | 不调用，Demo 启动时已加载 |
| 数据源 | knowledge_base 表 | pipeline_data_sets_items 表 |
| 流程 | embedding_knowledge_agent | rewritten_data_service_agent |
| State 构建 | build_initial_state_from_record(KnowledgeBaseRecord) | 从 DataSetsItemsRecord 映射到 prompt_vars |
| 入口 | 命令行脚本 | REST API |

### 4.5 DataSetsItemsRepository 扩展

**新增/扩展方法**：

```python
async def get_by_ids(
    self,
    dataset_id: str,
    item_ids: List[str],
) -> List[DataSetsItemsRecord]:
    """
    按 dataset_id 和 item_ids 列表查询数据项。
    仅返回属于该 dataset 的 record，且 id 在 item_ids 中。
    """

# 按 query_params 查询时，需支持「无分页」模式
# 方式一：新增 get_list_by_conditions(dataset_id, status, unique_key, source, keyword) 返回全量
# 方式二：扩展 get_list_with_total，当 limit 为 None 或传入特殊值时返回全量
```

## 5. 前端实现设计

### 5.1 批量按钮

**位置**：在「新建数据项」前（与「清理所有数据」同行）

**布局示意**：

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ {{ datasetName }}   [进行数据清洗] [新建数据项] [清理所有数据]                    │
└──────────────────────────────────────────────────────────────────────────────┘
```

**逻辑**：

1. 点击「进行数据清洗」→ 弹出确认框：「将对当前查询条件下的全部数据项执行数据清洗，是否继续？」
2. 确认后，调用 `POST ${API_PREFIX}/datasets/${datasetId}/items/rewritten/execute`，请求体为：

```json
{
  "query_params": {
    "status": queryStatus,
    "unique_key": queryUniqueKey,
    "source": querySource,
    "keyword": queryKeyword
  }
}
```

3. 后端按 query_params 查询，命中多少条就执行多少条，无分页限制。

4. 成功：`ElNotification.success('数据清洗完成，成功 X 条，失败 Y 条')`，可选刷新列表
5. 失败：`ElMessage.error(detail)`

### 5.2 行内按钮

**位置**：操作列，与「编辑」「删除」并列

**逻辑**：

1. 点击行内「数据清洗」→ 调用接口，请求体为 `{ "item_ids": [row.id] }`
2. 成功/失败提示同上
3. 可选：执行中禁用按钮，防止重复点击

### 5.3 加载状态与防重复

- 批量清洗：执行中显示 loading，禁用「进行数据清洗」按钮
- 行内清洗：可对当前行按钮 diabled，或全局 loading（按实现复杂度选择）

## 6. 流程时序

单条与批量清洗**共用同一流程**，差异仅在于**查询条件**不同（`item_ids` 或 `query_params`），后续执行逻辑完全一致。

### 6.1 通用流程

```
用户点击「进行数据清洗」（批量）或 行内「数据清洗」（单条）
  → 批量时：确认弹窗
  → POST /datasets/{dataset_id}/items/rewritten/execute
      body: { item_ids: [...] } 或 { query_params: {...} }
  → rewritten_service.execute_rewritten(dataset_id, item_ids=..., query_params=...)
  → 根据入参确定 records：
      - 若 item_ids 非空 → get_by_ids(dataset_id, item_ids)
      - 否则 → 按 query_params 查询（命中多少取多少）
  → FlowManager.get_flow("rewritten_data_service_agent")
  → run_batch_parallel(records, graph, max_concurrent=4)
  → 每条 record: build_state(record) → graph.ainvoke
  → 返回 stats
  → 前端提示，可选刷新
```

### 6.2 入参差异说明

| 入口 | 请求体 | 查询方式 |
|------|--------|----------|
| 批量「进行数据清洗」 | `{ query_params: { status, unique_key, source, keyword } }` | 按筛选条件查询，命中多少执行多少 |
| 行内「数据清洗」 | `{ item_ids: [row.id] }` | 按 ID 精确查询 |

## 8. 设计缺失与待确认点

### 8.1 已识别的设计缺失

| 项目 | 说明 | 建议 |
|------|------|------|
| **DataSetsItemsRecord → prompt_vars 映射** | `input`/`output`/`metadata` 结构因 cleaner 而异（LSK、Sh1128 等），字段名可能不统一 | 在 `rewritten_service` 中实现健壮的映射函数，兼容多种结构；缺失字段用空字符串/空列表填充 |
| **异步执行** | 大批量清洗可能耗时较长，是否改为异步任务+轮询 | 首版建议同步执行，超时时间可配置；若后续需要异步，可增加「任务提交 + 轮询状态」模式 |

### 8.2 风险记录

| 风险 | 说明 | 处理策略 |
|------|------|----------|
| **大数据量内存溢出** | 查询条件命中的数据量很大时，一次性加载全部记录可能造成内存溢出 | 当前不考虑，后续进行方案优化（如流式处理、分批查询等） |

### 8.3 逻辑不确定点

| 项目 | 说明 | 建议 |
|------|------|------|
| **q_context 取值** | `item.metadata.content_info.user_info` 可能为对象，需序列化为提示词可用的字符串 | 若为 dict，可 JSON 序列化或按业务约定格式化；若不存在则不填充 |
| **history_messages 格式** | 提示词期望的格式（如 `[{"type":"human","content":"..."}]`）与 DB 存的可能不一致 | 若 DB 存的是 LangChain BaseMessage 序列化形式，需反序列化；否则按现有格式直接传入 |
| **rewritten_data_service_agent 预加载** | 流程是否在 main 启动时预加载 | 检查 `config/flows/flow_loader.yaml` 或等价配置，若未预加载，首次调用时 `get_flow` 会加载，可接受 |

### 8.4 需要补充的配置

| 项目 | 说明 |
|------|------|
| flow_loader preload | 若希望首请求更快，可将 `rewritten_data_service_agent` 加入 preload 列表 |
| 超时配置 | 单次 API 请求超时时间（如 5 分钟），需与网关/反向代理一致 |
